## Resumen ejecutivo

Al armar un presupuesto de aguas/alimentos, los **paquetes padre** del nomenclador deben mostrar sus **determinaciones hijas** como desglose informativo (pH, turbidez, etc.), visible en create/edit/show y en el PDF (impresión + email). La jerarquía se **congela en snapshot** al guardar para que presupuestos enviados no cambien si se edita el nomenclador.

---

## Decisiones de scope (acordadas con el usuario)

| Tema | Decisión |
|------|----------|
| Formato hijos | Informativos, indentados, sin cantidad/precio |
| Profundidad | Todos los niveles (padre → sub-padre → hoja) |
| Persistencia | Snapshot JSON en `quote_items` al store/update |
| Presupuestos viejos | Fallback dinámico desde `test_id` si snapshot null |
| Precio | Solo en fila padre; hijos no afectan subtotal |
| Edición de hijos | No — no se pueden quitar hijos individualmente del presupuesto |
| Designer | No |

## Edge cases

- Ítem manual o servicio (`test_id` null): sin hijos, sin cambios.
- Duplicar presupuesto: copiar `children_snapshot` en replicate.
- Incrementar cantidad de padre ya agregado: no regenerar hijos (mantener snapshot del ítem).
- Subtotal/total: calcular solo sobre ítems padre (quantity × unit_price).
- Contador "Ítems" en show: puede seguir siendo count de padres.

---

## Advertencias

- No crear filas hijas en `quote_items` — usar JSON en el ítem padre.
- El builder de snapshot debe respetar orden del pivot `test_parents.order` y soportar sub-padres recursivos.
- Validar hidden inputs de `children_snapshot` en store/update para evitar payloads malformados.
