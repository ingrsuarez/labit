## TL;DR

Pantalla de monitoreo en labit para visualizar la ingesta de resultados desde LISCOM
(`POST /api/v1/results/batch` de v1.51.0). Vista unificada operativa + técnica con permisos
para mostrar/ocultar elementos según rol. Sin Designer — reusa el layout admin existente.

**Por qué importa:** después de v1.52.0 (cliente outbound LISCOM) los resultados entran
automáticamente. El bioquímico necesita ver qué llegó, qué se rechazó por estar ya validado
(situación importante operativamente) y el admin necesita ver qué sedes están desconectadas.

**No es bloqueante para producción** — la cadena LISCOM↔labit funciona desde v1.52.0 sin esta
versión. Pero es necesaria para que el laboratorio confíe en la automatización y detecte
problemas temprano.

## Tensiones identificadas (revisar antes de implementar)

### 1. **¿Cuál es el permission real para "ver protocolos"?**

El prompt usa `view-protocols` como placeholder. labit tiene módulos de **clínico**,
**muestras** y **veterinario**, cada uno probablemente con su propio permission
(`view-clinical-protocols`, `view-sample-protocols`, etc).

**Resolución sugerida (Dev):** revisar `database/seeders/RolesAndPermissionsSeeder.php` y elegir:
- Si existe un permission "ver cualquier tipo de protocolo" → usarlo.
- Si no, hacer el `Route::middleware('can:view-clinical-protocols|view-sample-protocols|view-vet-protocols')` o similar (sintaxis Spatie permite `'permission|otraPermission'`).
- Como último recurso, restringir solo a `auth` (cualquier usuario logueado) con el argumento de que el dashboard es informativo.

**Anotar la decisión en el CHANGELOG.**

### 2. **Nombres reales de rutas para `getProtocolUrl`**

El servicio asume:
- `route('admissions.show', ['number' => $protocolNumber])` → clínico
- `route('samples.show', ['number' => $protocolNumber])` → muestras
- `route('vet-admissions.show', ['number' => $protocolNumber])` → veterinario

**Verificar** en `routes/web.php` los nombres reales y el parámetro real (¿es `number`,
`protocol_number`, `id`, `slug`?). Ajustar el método `getProtocolUrl` con los nombres
y bindings correctos.

### 3. **Performance de counters con MySQL JSON**

`items_summary` en `ResultIngestion` es un JSON. Si el dashboard intenta agregarlos en vivo
(`SUM(JSON_EXTRACT(...))`), va a ser lento.

**Mitigación ya prevista:** usar los contadores agregados de `ResultBatch` (campos
`items_ingested`, `items_overwritten`, `items_rejected`, `items_duplicate`) que se
materializan en v1.51.0 al procesar el batch. Solo decodificar `items_summary` en el
detalle de **una** ingestion específica.

**Verificar en código:** `ApiMonitorService::getCounters()` debe leer de
`result_batches.items_*`, no calcular sumas sobre el JSON.

### 4. **Auto-refresh con muchos users → carga DB**

Si 5 bioquímicos abren el dashboard simultáneamente con polling 10s, son 30 queries por
minuto. Con índices del PASO 1 quedan en pocos ms cada una → tolerable.

Si en producción se ve carga, próximo paso: cachear los counters en Redis cada 30s y servir
desde cache. **No hace falta hoy.**

### 5. **`raw_request` puede ser truncado**

En v1.51.0, si el `raw_request` excede 16 KB se trunca con marker `__truncated__`. La vista
de detalle del batch (sección admin colapsable) debe detectar ese marker y avisar:

> "⚠ Payload truncado. Tamaño original: X bytes. Solo se muestran los primeros N items."

### 6. **Borrado en cascada**

`ResultIngestion` tiene FK a `ResultBatch` con `cascadeOnDelete()`. El comando `api:cleanup`
borra ingestions primero, luego batches. **Orden importa** para evitar warning si SQLite/MySQL
estricto. Ya está documentado en el comando.

### 7. **Sidebar badge de "rechazos del día"**

El partial del sidebar hace una query inline `ResultIngestion::where(...)->count()`.
Esta query corre en CADA pageview de cualquier pantalla. **Cachear con `Cache::remember(..., 60)`**
si se siente lenta:

```blade
@php
    $rejected = Cache::remember('api_monitor_rejected_today', 60,
        fn() => \App\Models\ResultIngestion::where('rejection_reason', 'ALREADY_VALIDATED')
                  ->whereDate('created_at', today())->count());
@endphp
```

### 8. **Permission `api-clients.manage` puede no existir aún**

Esa permission la crea v1.46.0. Si por algún motivo se mergea v1.53.0 antes que v1.46.0
(no debería pasar, pero...), las directivas `@can('api-clients.manage')` van a fallar
silenciosamente devolviendo `false` → la sección técnica nunca se ve.

**No es un bug**, pero verificar que v1.46.0 está mergeada antes del deploy de v1.53.0.
