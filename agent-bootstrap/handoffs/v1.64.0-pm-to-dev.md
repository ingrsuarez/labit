**Sesión PM:** 2026-04-26
**Versión:** v1.64.0 — DDJJ por impuesto + imputación de anticipos sufridos
**Prompt de tarea:** `agent-bootstrap/prompts/pendientes/v1.64.0-declaraciones-impuestos-imputacion-anticipos.md`

---

## Contexto del pedido

v1.63.0 dejó las percepciones cargándose como anticipos en cuentas de activo (crédito
fiscal). v1.63.1 cubrió las NC. Ambas acumulan saldos en `1.1.06`, `1.1.08` y subcuentas
que el contador asocie. Pero **nada los descarga** — los saldos solo crecen.

Esta versión cierra el círculo:

1. **Catálogo de impuestos** (`Tax`): agrupa varias percepciones bajo un mismo impuesto
   (ej: el impuesto "IVA" agrupa "Percepción IVA RG 2408", "Percepción IVA Aduana", etc.).
2. **DDJJ** (`TaxReturn`): el contador carga la declaración mensual del impuesto con el
   monto declarado del organismo.
3. **Imputación de anticipos** (`TaxReturnApplication`): el contador marca qué anticipos
   del período se imputan contra esa DDJJ. El sistema calcula saldo a pagar / saldo a favor.
4. **Asiento contable**: al confirmar la DDJJ, se acreditan las cuentas de anticipo
   imputadas y se debita la cuenta "X a pagar" (pasivo). Si quedó saldo a pagar > 0, queda
   como deuda con el organismo; si saldo < 0, queda saldo a favor en la misma cuenta.

**No incluye el pago real** del saldo a pagar — eso se hace después con un asiento
manual del Libro Diario o con un módulo de tesorería futuro. Esta decisión reduce
significativamente el scope sin bloquear funcionalidad.

## Decisiones tomadas en la sesión PM

| Decisión | Valor | Razón |
|---|---|---|
| Modelado | DDJJ formal por período + imputaciones explícitas | Es lo correcto contablemente; "imputación simple" sería medio paso |
| Catálogo `Tax` | Modelo nuevo separado de `PurchasePerception` | Un impuesto puede agrupar varias percepciones (IVA con RG 2408 + Aduana). Conceptualmente correcto |
| Asociación percepción ↔ impuesto | Columna `tax_id` (nullable FK) en `purchase_perceptions` | Permite que el wizard liste automáticamente los anticipos del impuesto |
| Pago del saldo | **NO incluido** | Reduce scope; se hace con asiento manual del Libro Diario |
| Periodicidad | Mensual / Trimestral / Anual (enum en `Tax.frequency`) | Cubre IVA mensual, IIBB mensual, Ganancias anual, etc. |
| Estados de DDJJ | `draft` / `confirmed` / `cancelled` | Permite cargar borrador antes de generar el asiento |
| Anulación | Sí — botón en la DDJJ confirmada genera asiento reverso | Útil para correcciones |
| Imputaciones cross-período | NO en esta versión | Si querés trasladar saldo a favor, se hace cargando la DDJJ del período siguiente con saldo a favor inicial (ajuste manual) |
| Designer primero | **Sí** | Hay 3 piezas de UI nuevas: CRUD Tax, wizard de DDJJ y listado/show. El wizard sobre todo merece diseño previo |
| Permisos | `taxes.manage` (admin + contador) y `tax-returns.manage` (admin + contador) | Mismas pautas que v1.63.0 |

## Tensiones y advertencias para el Dev

### 1. Validación: no imputar el mismo anticipo dos veces

Una `PurchaseInvoicePerception` (o `PurchaseCreditNotePerception` si v1.63.1 está
completada) NO puede aparecer en dos `TaxReturnApplication` confirmadas. Validar al
confirmar la DDJJ:

```sql
SELECT * FROM tax_return_applications tra
JOIN tax_returns tr ON tra.tax_return_id = tr.id
WHERE tra.purchase_invoice_perception_id = ?
  AND tr.status = 'confirmed'
  AND tr.id != ? -- excluir la actual
```

Si encuentra alguno → bloquear con mensaje claro: "El anticipo de FC X ya fue imputado
en la DDJJ de [tipo] [período]."

### 2. Filtrado automático de anticipos disponibles

En el wizard, cuando el usuario elige impuesto + período, el sistema lista automáticamente
las percepciones disponibles aplicando estos filtros:

- Percepciones cuyo `tax_id` = el impuesto elegido
- FC con `issue_date` dentro del período (mensual: month + year; trimestral: quarter; anual: year)
- Que NO estén ya imputadas en otra DDJJ confirmada

### 3. Cache de totales en `TaxReturn`

Para evitar agregaciones JSON cada vez que se muestre la DDJJ, mantener `applied_total`
y `balance` como columnas materializadas. Recalcular en `recalculateTotals()` cada vez
que se modifica una aplicación.

### 4. Asiento de anulación

El botón "Anular DDJJ" debe generar un asiento INVERSO al original (mismas líneas con
DB y CR intercambiados). Marcar la DDJJ como `cancelled` y dejar `journal_entry_id` con
el asiento original como referencia. Las aplicaciones quedan asociadas pero los anticipos
vuelven a estar disponibles para otra DDJJ.

### 5. Scope del cálculo de período

Cuando se elige impuesto mensual + año + mes, el filtro debe ser por `issue_date`
(fecha de la factura). NO usar `created_at` (puede ser cargada tarde) ni `due_date`.

Para impuestos anuales (Ganancias), el período es solo `year` y `month` queda en NULL.
Para trimestrales, el `month` representa el mes inicial del trimestre (1, 4, 7, 10).

### 6. UI del wizard

El wizard tiene 3-4 pasos lógicos. Puede implementarse como una sola página con
secciones colapsables (más simple, recomendado) o como wizard real con stepper.
Decisión final del Designer.

Una vez en producción, si el contador empieza a usarlo a fondo, las extensiones naturales
serían:

- **v1.64.1 — Pago del saldo a pagar de una DDJJ**: agregar paso "registrar pago" que

  archivos generados por los sistemas oficiales.

  de un período al siguiente.

Ninguna bloquea la funcionalidad de v1.64.0 — son extensiones puras.
