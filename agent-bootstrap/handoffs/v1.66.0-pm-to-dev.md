## Contexto del pedido

Hoy `/dashboard` (ruta `dashboard`, vista `resources/views/dashboard.blade.php`,
controller `app/Http/Controllers/DashboardController.php`) es **100% RRHH**:
empleados activos/inactivos, ausencias, nómina, cumpleaños, top puestos, accesos rápidos.

A pesar de que el sistema tiene módulos completos de Ventas (v2.1.0), Compras (v2.2.0),
Recibos de cobro (v1.39+), Órdenes de pago (v1.39.1) y Contabilidad (v3.0–v3.4),
**los admins no tienen ninguna pantalla de visibilidad financiera ejecutiva**. Hay que
ir módulo por módulo a mirar los números.

El cliente pidió un panel principal con 4 KPIs del mes en el dashboard:
**ventas, compras, ingresos, egresos**, cada uno con gráfico de barras.

## Tensiones y advertencias para el Dev

### 1. Performance: 4 datasets × 12 meses por empresa activa

Cada widget hace 1-2 queries con `groupBy(YEAR, MONTH)`. Hay 4 widgets. Total:
6-8 queries en el `index`. No es alarmante, pero hay que asegurar índices:

- `sales_invoices(company_id, issue_date)` — agregar si no existe
- `credit_notes(company_id, issue_date)` — agregar si no existe (relación al SalesInvoice
  vía `sales_invoice_id`; si no tiene `company_id` directo, usar JOIN)
- `purchase_invoices(company_id, issue_date)` — verificar
- `purchase_credit_notes(company_id?, issue_date)` — verificar (si no tiene `company_id`,
  derivar vía `purchase_invoice_id` con JOIN)
- `collection_receipts(company_id?, date)` — verificar (puede no tener `company_id`
  directo; en ese caso filtrar vía `customer.company_id` con JOIN, o verificar si tiene)
- `payment_orders(company_id, date)` — ya existe `company_id` (v2.2.0)

Si una migración necesita índices nuevos, agregarlos en `database/migrations/2026_..._add_dashboard_financial_indexes.php`.

### 2. CollectionReceipt y multi-empresa

`CollectionReceipt` (v1.39.0) **no tiene `company_id` en el fillable visible**. Verificar:

```bash
php artisan tinker --execute="dd(Schema::getColumnListing('collection_receipts'));"
```

Si NO tiene `company_id`:
- Filtrar vía `customer.company_id` (relación belongsTo `customer`)
- Documentar en el prompt el join correcto

Si SÍ tiene: usar `where('company_id', active_company_id())` directo.

Mismo análisis para `credit_notes` y `purchase_credit_notes`.

### 3. División por cero en variación %

Si el mes anterior tuvo total = 0 y el actual tiene > 0, la variación es **infinita**.
Estrategia:

```php
if ($prevMonth == 0) {
    $variation = $currentMonth > 0 ? null : 0;  // null = "sin base de comparación"
}
```

UI: si `$variation === null`, mostrar "—" o "sin datos previos" en lugar de "+∞%".

### 4. Mes en curso es parcial

Comparar "el mes en curso (al día de hoy)" vs "el mes pasado completo" puede ser
engañoso si estamos al día 5. **Decisión:** mostrar el dato bruto sin advertencia.
La barra del mes actual ya queda visualmente más corta y el usuario lo entiende.
Si después surge feedback negativo, agregar tooltip "mes parcial al día N".

### 5. Mudanza de RRHH: NO romper rutas existentes

Hay vistas y links que apuntan a `route('dashboard')` esperando contenido de RRHH
(ej: redirects desde `vacation.approval`, `leave.*`, `cumpleaños`, `top puestos`,
accesos rápidos). Después de mudar el contenido a `/rrhh`:

- **Mantener** la ruta `dashboard` apuntando a `DashboardController@index`. NO renombrar.
- **Crear** nueva ruta `rrhh.index` apuntando a `RrhhController@index`.
- **Buscar y actualizar** los links internos que esperaban contenido de RRHH en `/dashboard`:
  ```bash
  grep -r "route('dashboard')" resources/views app/
  ```
  Caso por caso decidir si seguir apuntando a `dashboard` (ahora financiero) o cambiar a `rrhh.index`.
  Probable que la mayoría siga al financiero (es el "home" del admin), pero los breadcrumbs
  que digan "Volver al panel de RRHH" deberían ir a `rrhh.index`.

### 6. Reusar lógica de DashboardController

Para no duplicar código, **literal mover** todo el bloque actual de RRHH (líneas ~39 a 240
de `DashboardController.php`) a `RrhhController@index`. NO refactorizar en este paso —
es una mudanza copy-paste. La vista nueva `rrhh/index.blade.php` también es copy-paste 1:1
de `dashboard.blade.php`, solo cambiando el `<x-admin-layout title="...">` y el header.

Después `DashboardController@index` queda con:
- Los redirects por rol (sin cambios)
- Los nuevos cálculos financieros
- `return view('dashboard', compact(...los 4 widgets...))`

### 7. Tests del controller actual

Hay tests que prueban `/dashboard` (`tests/Feature/...`) esperando datos de RRHH
(empleados, etc.). Después del cambio:

- Renombrar/mover esos tests a `tests/Feature/Rrhh/RrhhDashboardTest.php` y apuntar a `/rrhh`.
- Crear tests nuevos `tests/Feature/Dashboard/FinancialDashboardTest.php` con los 4 widgets.

```bash
grep -r "get('/dashboard')\|route('dashboard')" tests/Feature
```

## Criterio de done (resumen)

1. Ruta `/rrhh` con `RrhhController@index` accesible para `admin` y `contador`. Resto: 403 o redirect según permiso.
2. Vista `resources/views/rrhh/index.blade.php` muestra el contenido completo del dashboard actual de RRHH (mudanza 1:1, header cambia a "Panel de Recursos Humanos").
3. Sidebar admin tiene entrada nueva "Recursos Humanos" → `route('rrhh.index')`, con posición/ícono según diseño.
4. `DashboardController@index` mantiene los redirects por rol existentes.
5. `DashboardController@index` calcula los 4 datasets financieros filtrados por `active_company_id()`:
   - Ventas netas (FV − NC) por mes — 12 meses + total mes actual + total mes anterior + variación %
   - Compras netas (FC − NC proveedor) por mes — idem
   - Ingresos (recibos de cobro) por mes — idem
   - Egresos (órdenes de pago) por mes — idem
6. Vista `resources/views/dashboard.blade.php` reemplazada por el dashboard financiero según diseño.
7. Variación % maneja correctamente `mes_anterior == 0` (no muestra "+∞%", muestra "—" o "sin datos previos").
8. Tests Feature:
   - Admin/Contador acceden a `/dashboard` y ven los 4 widgets con datos correctos
   - Admin/Contador acceden a `/rrhh` y ven el panel de RRHH
   - Otros roles redirigidos según política existente
   - Cambio de empresa activa cambia los números del dashboard financiero
   - Variación % con mes anterior 0 no rompe
9. Tests existentes que apuntaban a `/dashboard` esperando datos RRHH se mueven a `tests/Feature/Rrhh/...` y siguen verde.
10. CHANGELOG.md actualizado.
11. STATUS.md actualizado.
12. Tag `v1.66.0` creado.

## Próxima versión sugerida (post-v1.66.0)

Si el cliente engancha y pide más, las extensiones naturales serían:

- **v1.66.1 — Drilldown desde los widgets**: clic en una barra del gráfico → listado del mes (`/sales-invoices?month=...`, etc.).
- **v1.66.2 — Más KPIs en el dashboard financiero**: saldo de tesorería actual (cuentas bancarias + caja), deuda de clientes (deudores), saldo a pagar a proveedores. No requiere módulos nuevos, todo está disponible.
- **v1.66.3 — Filtro de período personalizado**: en lugar de fijo a 12 meses, selector "últimos 6 / 12 / 24 meses" o rango.
- **v1.66.4 — Comparación interanual**: en cada widget, agregar serie del mismo mes año anterior para ver crecimiento year-over-year.
- **v1.66.5 — Export del dashboard a PDF**: para reuniones de directorio.

Ninguna bloquea la funcionalidad de v1.66.0 — son extensiones puras.
