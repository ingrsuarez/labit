## Resumen ejecutivo

Endpoint REST que recibe los resultados que LISCOM aprobó humanamente (v1.50.0) y los persiste contra `admission_tests` / `sample_determinations` / `vet_admission_tests`. La pieza central es la **regla de no overwrite si ya validado por bioquímico**: si la determinación destino tiene `is_validated=true`, el ítem se rechaza con `ALREADY_VALIDATED` y NO se toca el valor.

Sin UI nueva (la de monitoreo es v1.53.0). Solo endpoint + 2 modelos de auditoría + tests.

## Tensiones para el Dev

### 1. Campo del valor varía por modelo (verificar)

Los 3 modelos de determinación deberían tener `result` como campo del valor, pero hay que verificarlo en cada `$fillable`. Si alguno usa `value` u otro nombre, ajustar `persistResult` en el service.

### 2. Trait `Auditable` (v2.5.0) ya cubre overwrites?

Si `AdmissionTest` / `SampleDetermination` / `VetAdmissionTest` ya usan trait `Auditable`, el `save()` genera audit log automático. No hace falta sumar audit explícito en v1.51.0. **Verificar al implementar.** Si no lo tienen, decidir entre:
(a) Sumarlo en esta versión (chico, alineado con buenas prácticas)
(b) Postergarlo a una versión separada
Per default, optar por (a) si se puede hacer en <30min — la trazabilidad de quién/qué/cuándo cambió un resultado por API es valiosa.

### 3. `lab_branch_id` en los protocolos

`Admission` y `Sample` deberían tener `lab_branch_id` (de v1.30.0). `VetAdmission` también (v1.30.0+). Verificar y, si alguno no lo tiene, eliminar la rama `PROTOCOL_OUT_OF_BRANCH` para ese tipo o documentar.

### 4. Volumen del payload

El límite de 500 ítems por batch es defensivo. En la práctica LISCOM va a mandar 10-50 (lo aprobado en una ventana de polling). Si en QA ves volúmenes mayores constantes, revisar el batching del lado LISCOM (v1.52.0).

### 5. Performance de `findDetermination`

Cada item hace 1 SELECT por (protocolo + test_id). Para batches de 50 mensajes con 5 OBX cada uno = 250 queries. Si esto es lento (>1s), considerar:
- Eager loading de las determinaciones del protocolo en `loadProtocol`
- O usar `whereIn('test_id', $testIds)` por mensaje

**No optimizar prematuramente.** Medir primero.

### 6. Notificación al bioquímico

Cuando la API ingresa un resultado, el bioquímico debería ver un badge "tenés N resultados nuevos pendientes de validar del equipo X". Esto es **fuera de scope de v1.51.0** — se planifica como v1.51.1 o se incluye en v1.53.0 (monitoreo). Documentar como tensión y avisar.

### 7. Endpoint individual

NO se implementa POST `/api/v1/results` (singular). Si en el futuro un caso lo amerita (ej: equipo industrial que solo soporta requests sincrónicos uno a uno), replanificar.

## Criterio de done

La versión está completa cuando:

1. Migraciones aplicadas y verificadas en `db.sqlite3` (o MySQL local).
2. POST con curl/Postman devuelve 200 con response detallada por ítem.
3. Caso `ALREADY_VALIDATED` rechaza correctamente y devuelve `validated_by_name` + `validated_at`.
4. Caso overwrite (no validado) sobrescribe y registra `previous_value` en la response.
5. Idempotencia batch: reenviar mismo `batch_id` devuelve `duplicate=true`.
6. Idempotencia mensaje: reenviar mismo `control_id` en otro batch devuelve `status=duplicate`.
7. Sin auth → 401. Con auth inválida → 401. Con key inactiva → 401.
8. 14 tests Feature pasando.
9. Doc API en `docs/api/v1/results.md` completo y consistente con la response real.
10. CHANGELOG / STATUS / ROADMAP / BLUEPRINT actualizados.
11. Tag `v1.51.0` en master.

## Próximas versiones en la cadena

**v1.52.0 (LISCOM/Django)** — Cliente HTTP en LISCOM que toma los `ResultMessage` con `status=approved`, los empaqueta en batches y los postea a este endpoint. Cola persistente con retries exponenciales. Maneja las response statuses (`ingested`, `overwritten`, `duplicate`, `rejected`+reasons) y actualiza `ResultMessage.status` a `sent` o marca `Result.api_status='rejected_already_validated'` para no reintentar.

Antes de v1.52.0 conviene:
- Tener v1.51.0 en producción/staging para que el dev de LISCOM pueda integrar contra un endpoint real.
- Decidir frecuencia de la cola (¿cada 1 min? ¿cada 5 min? ¿al aprobar?).

**v1.53.0 (labit)** — UI admin de monitoreo de la API:
- Listado de últimos batches por API client con conteos.
- Detalle de un batch con todos los items y reasons.
- Filtros por estado, fecha, tipo de error.
- Bonus: badge "tenés N resultados nuevos para validar" en el módulo de protocolos.

Volver al PM antes de v1.52.0 para definir las decisiones operativas (frecuencia de cola, política de retries, modo offline).
