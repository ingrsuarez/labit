## Contexto del pedido completo (no solo esta versión)

El usuario pidió **integrar labit con LISCOM**, un servidor de interfases HL7 que vive en
`c:\wamp64\www\interfases` (Django + Channels + sqlite + motor HL7 MLLP). LISCOM corre en
LAN en cada sede y se comunica con los equipos de laboratorio por TCP/IP usando HL7 v2.x.

El flujo completo (que vamos a construir en 8 versiones encadenadas v1.46.0–v1.53.0):

```
labit (cloud) → admisión + protocolo + barcodes
                         ↓
                  etiquetas físicas
                         ↓
              equipo escanea barcode
                         ↓
              consulta a liscom (LAN)
                         ↓
        liscom hace pull desde labit API (cache local)
                         ↓
        liscom responde HL7 al equipo (qué determinaciones hacer)
                         ↓
              equipo procesa y manda HL7 con resultados
                         ↓
              liscom guarda en DB local (estado=pending_review)
                         ↓
              operador en liscom revisa y aprueba
                         ↓
        liscom envía a labit por API (cola con retries)
                         ↓
        labit guarda valores (sin auto-validar, queda pendiente como hoy)
```

**v1.46.0 (esta versión)** es solo el cimiento: la API key, el middleware, el CRUD admin
y el endpoint `/api/v1/ping`. **NO incluye endpoints de negocio.** Esos vienen en v1.47.0+.

## Decisiones tomadas en la sesión PM

| Decisión | Valor | Razón |
|---|---|---|
| Tipo de auth | API key en header `X-API-Key` | Simple, fácil de rotar, sin overhead. Sanctum sigue para usuarios humanos. |
| Granularidad de keys | Una key por sede de laboratorio | Topología decidida: una instancia liscom por sede. |
| Formato de key | Prefijo `labit_` + 40 chars random | Detección de leaks por prefijo (estilo Stripe/GitHub). |
| Persistencia | Solo hash SHA-256, key plana visible una vez | Mejor práctica de seguridad. |
| Permisos | Un solo permiso `api-clients.manage` (no granular) | Solo Administradores la usan, granularidad innecesaria. |
| Modelo de sync (futuro) | Pull-on-scan + sync periódica + revisión humana | Tolera caídas de internet naturalmente. |
| Auto-validación de resultados | NO — ingesta simple, validación humana después | Como hoy. |

## Tensiones y advertencias para el Dev

### 1. Tracking en background asume queue worker corriendo

El middleware usa `dispatch(...)->afterResponse()` para incrementar `requests_count`
y `last_used_at`. Si el queue driver es `sync` (default en dev local), se ejecuta inline
después de enviar la response — funciona pero suma latencia. Si es `database` o `redis`,
hay que tener `php artisan queue:work` corriendo en producción. **Documentar esto en el
README de operaciones cuando lleguemos a v1.53.0.**

### 2. `lab_branch_id` es inmutable post-creación

Por diseño, una vez creada la API key, no se puede cambiar la sede asociada. Si alguien
mueve una instancia liscom de una sede a otra, hay que crear una nueva key. Esto evita
inconsistencias en logs/auditoría y simplifica el filtrado automático en v1.47.0+.

### 3. La key plana NO se persiste en NINGÚN log

Asegurarse de que el logging del canal `api` NO incluya la key plana ni el hash. Solo
debe loguear `client_id` y `client_name`.

### 4. Patrón de "key visible una sola vez"

Es un patrón de UX importante (Stripe/GitHub/AWS lo usan así). El Dev debe asegurarse de:
- Pasar la key plana por session flash (no por query string)
- El modal/banner debe tener un checkbox "ya guardé la key" que el usuario debe marcar para cerrar
- Después del primer view, no hay forma de recuperarla (solo regenerar)

## Preguntas frecuentes anticipadas

**Q: ¿Por qué no Sanctum personal access tokens?**
R: Sanctum está pensado para tokens de usuarios humanos (mobile apps, SPAs). Para
integraciones máquina-a-máquina con una key persistente y rotable por admin, una
implementación dedicada con prefix + hash es más explícita y auditable.

**Q: ¿Por qué una key por sede y no una global con scopes?**
R: La topología real es una instancia liscom por sede. Una key por sede simplifica el
filtrado de protocolos (v1.47.0 filtra por `lab_branch_id` automáticamente sin que el
cliente API tenga que pasarlo) y limita el blast radius si una key se compromete.

**Q: ¿Rate limiting?**
R: NO en v1.46.0. Si LISCOM hace polling agresivo (no debería, hace sync periódica cada
30 min + pull-on-demand al scan), se gestiona en v1.53.0 o como hotfix. Laravel ya tiene
`ThrottleRequests` listo para usar.

**Q: ¿CORS?**
R: La API solo se consume desde otro backend (LISCOM Django), no desde browsers. CORS
no aplica salvo que después se exponga a un frontend público.

## Criterio de done

La versión está completa cuando:

1. Migración corre y `api_clients` existe.
2. `GET /api/v1/ping` con header válido retorna 200; sin header retorna 401.
3. Admin puede crear, ver listado, regenerar y eliminar API keys desde `/admin/api-clients`.
4. La key plana se ve UNA vez al crear/regenerar y nunca más.
5. Tests Feature (`ApiClient*Test`) todos verde.
6. BLUEPRINT.md actualizado con DD-005.
7. Tag `v1.46.0` creado, mergeado a develop según Gitflow del proyecto.
