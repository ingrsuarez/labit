## Resumen ejecutivo

Esta versión implementa la primera pieza del lado LISCOM:

- **Cliente HTTP** a la API v1 de labit (con retries, manejo de errores tipado)
- **4 modelos nuevos** en sqlite: `LabitConnection`, `LocalProtocol`, `LocalDetermination`, `SyncLog`
- **Servicio de sync** con 4 modos: today / incremental / range / on_demand
- **Comando management** `sync_protocols` para CLI y cron
- **UI básica** con 4 pantallas: configuración, listado, detalle, historial

NO incluye: mapeo de códigos de equipo (v1.49.0), recepción HL7 de resultados (v1.50.0),
cola de envío (v1.52.0).

## Decisiones tomadas en la sesión PM

| Decisión | Valor | Razón |
|---|---|---|
| Sync periódica | Cron / systemd timer (NO Celery embebido) | Simple, suficiente para una sede, no agrega complejidad |
| Storage | sqlite (NO postgres) | Una instancia LISCOM por sede, sqlite alcanza |
| API key | En sqlite plana (NO encriptada) | sqlite local de la sede, riesgo bajo. Encriptación se evalúa después si hace falta |
| Singleton conexión | SÍ — solo una `LabitConnection` activa | Topología decidida: una key por sede |
| `raw_data` JSON | SÍ — guarda response completa | Futureproof: si labit suma campos, no hay que migrar local |
| Reemplazo de determinaciones | Delete + create (no diff) | Simple para v1.48.0; labit es fuente de verdad |
| Sync on-scan | SÍ — fallback automático cuando barcode no está en cache | Cubre el caso "muestra vieja no cacheada" |
| Sync histórica | SÍ — `--range` o desde UI | Operador puede traer pasado bajo demanda |
| Designer | SÍ — diseñar las 4 pantallas antes de implementar | UI consistente con resto de LISCOM |
| WebSocket en UI | NO — polling cada 5s alcanza | Evitar over-engineering en pantallas internas |

## Tensiones para el Dev

### 1. Bootstrap del sistema de agentes

Si el Dev arranca y NO encuentra `.agents/` en interfases, debe avisar y esperar a que
el usuario lo copie. NO improvisar implementando sin contexto.

### 2. Layout de templates

LISCOM ya tiene `templates/` con un layout base. Las nuevas pantallas (config, protocols,
sync logs) deben extender ese layout, no crear uno nuevo. Si el layout actual es feo o
inconsistente, NO refactorizar acá — proponer en una versión separada.

### 3. Manejo de Patient en LocalProtocol

Decidí guardar `patient_data` como JSON en `LocalProtocol` (en vez de un modelo
`LocalPatient` separado). Esto simplifica el sync (un solo update) pero limita las
queries por paciente. Es una decisión consciente:
- Para v1.48.0–v1.49.0 (consulta por barcode), no necesitamos buscar por paciente
- Si en futuro se necesita (ej: "todos los protocolos de fulano"), se evalúa migrar
- El JSON es flexible si labit cambia los campos

### 4. Performance del sync paginado

El servicio hace upsert dentro de un `transaction.atomic()` POR cada protocolo.
Para sedes con 500+ protocolos/día, esto puede ser lento. Si el sync supera 30s,
el Dev puede:
- Hacer batch de N protocolos por transacción (ej: 50)
- Usar `bulk_create` con `ignore_conflicts` para los nuevos
- Mantener el upsert atómico para los que ya existen

NO optimizar prematuramente. Primero medir.

### 5. Tests con responses + pytest-django

El proyecto LISCOM probablemente NO tiene tests configurados todavía (revisar). El Dev
debe configurar `pytest.ini` o `pyproject.toml` con `DJANGO_SETTINGS_MODULE`. Si la
configuración actual usa `django.test.TestCase`, mantener consistencia con eso en vez
de pytest fixtures.

### 6. Conexión a labit local para testear

El Dev necesita un labit corriendo local (`http://localhost:8000` o equivalente) con:
- Migraciones de v1.46.0 y v1.47.0 aplicadas
- Al menos un usuario admin
- Al menos una `ApiClient` activa con `lab_branch_id` válido
- Al menos algunos protocolos de los 3 tipos para sincronizar

Si labit no está disponible, el Dev puede usar las fixtures de los tests para validar
las funciones, pero el QA browser test requiere ambiente real.

## Criterio de done

La versión está completa cuando:

1. Los 4 modelos están migrados y aparecen en el admin Django
2. `/config/labit/` permite configurar la conexión
3. "Test connection" responde verde si labit está disponible
4. "Sync now" trae protocolos del día
5. CLI: `sync_protocols --today / --barcode / --range` funcionan
6. `/protocols/` lista los sincronizados con filtros funcionales
7. `/protocols/{number}/` muestra detalle con `raw_data` desplegable
8. `/sync-logs/` muestra el historial
9. Tests con `responses` mock pasan (`pytest tests/`)
10. README de LISCOM actualizado + `docs/integration-labit.md` creado
11. Tag `v1.48.0` creado en el git de interfases (NO labit)
