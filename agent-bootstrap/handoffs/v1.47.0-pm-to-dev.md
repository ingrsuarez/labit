## Resumen ejecutivo

Esta versión expone los **primeros endpoints de negocio** sobre la fundación de v1.46.0.
Son endpoints **read-only** (GET) sobre protocolos de los 3 modelos que existen en labit:
clínico (Admission), muestras (Sample) y veterinario (VetAdmission).

La estructura de respuesta es **unificada** entre los 3 tipos para que LISCOM no tenga
que tener 3 caminos paralelos en su código.

## Tensiones para el Dev

### 1. Estructura de VetAdmission no fue revisada en la sesión PM

El prompt asume nombres de relaciones (`animal`, `customer`/`owner`, `species`, `breed`)
basados en el patrón típico, pero **el modelo real puede diferir**. El Dev debe revisar
`app/Models/VetAdmission.php` y modelos relacionados ANTES de implementar el branch
`vet` del `ProtocolResource::buildPatientData()` y del `ProtocolLookupService::eagerLoadFor()`.

Si la estructura de vet es muy distinta (ej: usa otro nombre de tabla pivot para
determinaciones), el Dev debe adaptar el código sin pedir validación al PM —
la decisión técnica es: **adaptarse a lo que hay, mantener la estructura de respuesta
unificada para liscom**.

### 2. `external_code` en `tests` no existe hoy

El campo `test_code` en la response devuelve `null` en esta versión. Esto es a propósito.
NO crear migración para agregarlo en v1.47.0. El mapeo de códigos se discute en v1.49.0.

### 3. Performance del listado unificado

3 queries + merge en PHP es la estrategia. Si en producción con datos reales
(ej: 500+ protocolos en un día en una sede grande), el endpoint pasa de 200ms p95,
abrir hotfix v1.47.1 con vista SQL materializada `protocols_unified`.

**Acción concreta del Dev:** después de implementar, hacer un seed con ~1000 protocolos
mezclados y medir tiempo de respuesta de `GET /api/v1/protocols`. Si > 200ms en local,
considerar refactor antes de mergear.

### 4. Eager loading obligatorio

Sin eager loading, el endpoint genera N+1 queries y se cae. El método `eagerLoadFor()`
está en el prompt — verificar con Telescope/Debugbar que el listado de 100 protocolos
NO ejecute más de 5-6 queries totales.

### 5. Orden de mezcla de los 3 tipos

Cuando se mergean los 3 tipos en una sola colección, se ordena por `updated_at desc`.
Esto significa que si labit acaba de actualizar 3 vet admissions, esas aparecen
primero aunque haya 50 clinical más viejos. **Es el comportamiento deseado** —
liscom necesita ver lo último que cambió primero para sync incremental.

## Sobre PII y consideraciones legales

El default `patient_data_level=minimal` es una decisión de diseño que reduce el
riesgo de leak de datos personales sensibles. Activar `standard` (con DNI) requiere:

- Decisión administrativa (responsabilidad del laboratorio)
- Justificación operativa (¿el equipo HL7 necesita el DNI? La mayoría no)
- Eventualmente: registro de quién activó esa configuración (la auditoría de v2.5.0
  que está en `api_clients` ya lo cubre)

El Dev debe:
- En el form de edición de API key, agregar **helper text** que explique las
  implicancias de cambiar a `standard`
- NO permitir cambiar `patient_data_level` desde la API misma — solo desde el admin
