# v1.67.5 — Fix: búsqueda y selección de determinaciones hijas en protocolos

**Tipo:** patch / fix
**Prioridad:** alta — bloquea la carga de determinaciones específicas en los tres módulos
**Dependencias:** ninguna

---

## Problema

En los formularios de admisión y en la vista de show de protocolos, las determinaciones
**hijas** no aparecen para ser seleccionadas. El usuario no puede agregar, por ejemplo,
"Recuento de Glóbulos Blancos" (hija de "Hemograma Completo") sin agregar el padre completo.

El problema existe confirmado en **Muestras (aguas/alimentos)** y **Veterinario**,
y potencialmente en **Lab Clínico** (revisar y aplicar si corresponde).

Las relaciones de jerarquía entre tests usan dos mecanismos:
- Columna `tests.parent` (FK legacy, relación `parentTest / children`)
- Tabla pivot `test_parents` (relación many-to-many `parentTests / childTests`)

## PASO 1 — Muestras: extender la query de tests para incluir hijos

### 1.1 — SampleController::create() y edit()

En ambos métodos, la query actual es:

```php
$tests = Test::whereJsonContains('categories', 'aguas_alimentos')
    ->orderBy('name')
    ->get();
```

Reemplazarla para que incluya también los hijos (vía pivot `test_parents`) de los tests
de aguas_alimentos, aunque los hijos no tengan esa categoría en su propio campo:

```php
$parentIds = Test::whereJsonContains('categories', 'aguas_alimentos')->pluck('id');

$tests = Test::where(function ($q) use ($parentIds) {
    $q->whereJsonContains('categories', 'aguas_alimentos')
      ->orWhereHas('parentTests', fn ($p) => $p->whereIn('parent_test_id', $parentIds));
})
->orderBy('name')
->get();
```

Aplicar el mismo cambio en `edit()` (mismo cuerpo de query).

### 1.2 — SampleController::show()

Agregar `$availableTests` al método `show()` con la misma query extendida:

```php
$parentIds = Test::whereJsonContains('categories', 'aguas_alimentos')->pluck('id');

$availableTests = Test::where(function ($q) use ($parentIds) {
    $q->whereJsonContains('categories', 'aguas_alimentos')
      ->orWhereHas('parentTests', fn ($p) => $p->whereIn('parent_test_id', $parentIds));
})
->orderBy('name')
->get();
```

Pasarla a la vista: `compact('sample', 'labelMaterials', ..., 'availableTests')`.

### 1.3 — resources/views/sample/show.blade.php

En la vista de show, el select de "Agregar Determinación" hace una query inline en Blade:

```php
@foreach(\App\Models\Test::whereJsonContains('categories', 'aguas_alimentos')->orderBy('name')->get() as $test)
```

Reemplazarla para que use la variable `$availableTests` pasada desde el controlador:

```php
@foreach($availableTests as $test)
```

## PASO 2 — Lab Clínico: eliminar el filtro que excluye hijos legacy

### 2.1 — LabAdmissionController::searchTests()

La query actual filtra solo tests sin padre legacy (`whereNull('parent')`):

```php
$tests = Test::whereNull('parent')
    ->where(function ($query) use ($search) {
        $query->where('code', 'like', "%{$search}%")
            ->orWhere('name', 'like', "%{$search}%");
    })
    ->limit(20)
    ->get(['id', 'code', 'name', 'nbu', 'price']);
```

Eliminar la restricción `whereNull('parent')` para que los hijos legacy también aparezcan:

```php
$tests = Test::where(function ($query) use ($search) {
        $query->where('code', 'like', "%{$search}%")
            ->orWhere('name', 'like', "%{$search}%");
    })
    ->limit(20)
    ->get(['id', 'code', 'name', 'nbu', 'price']);
```

El resto del método (lógica de nomenclador / obra social) no se modifica.

## PASO 3 — Veterinario: incluir hijos de tests veterinarios en la búsqueda

### 3.1 — VetAdmissionController::searchTests()

La query actual excluye completamente los hijos via pivot con `whereDoesntHave('parentTests')`:

```php
$tests = Test::where(function ($q) use ($query) {
    $q->where('code', 'like', "%{$query}%")
        ->orWhere('name', 'like', "%{$query}%");
})
    ->whereJsonContains('categories', 'veterinario')
    ->whereDoesntHave('parentTests')
    ->limit(20)
    ->get(['id', 'code', 'name', 'nbu']);
```

Reemplazar para incluir también los hijos de tests veterinarios (que quizás no tienen
la categoría `'veterinario'` en su propio campo `categories`):

```php
$vetParentIds = Test::whereJsonContains('categories', 'veterinario')->pluck('id');

$tests = Test::where(function ($q) use ($query) {
    $q->where('code', 'like', "%{$query}%")
        ->orWhere('name', 'like', "%{$query}%");
})
->where(function ($q) use ($vetParentIds) {
    $q->whereJsonContains('categories', 'veterinario')
      ->orWhereHas('parentTests', fn ($p) => $p->whereIn('parent_test_id', $vetParentIds));
})
->limit(20)
->get(['id', 'code', 'name', 'nbu']);
```

El resto del método (cálculo de precio NBU) no se modifica.

## PASO 4 — UX: indicar visualmente si un test es hijo en los resultados de búsqueda

En los tres módulos, cuando un test hijo aparece en los resultados de búsqueda, es útil
que el usuario sepa a qué padre pertenece. Agregar una línea de descripción en los
resultados de búsqueda.

### 4.1 — Muestras create.blade.php y edit.blade.php (Alpine.js)

En el array `allTests` que se serializa a JSON, incluir también el nombre del padre:

En el controlador (create y edit), mapear los tests para incluir info del padre:

```php
$tests = $tests->map(function ($test) {
    $test->parent_name = $test->parentTests->first()?->name
        ?? ($test->parentTest?->name);
    return $test;
});
```

En la vista, en el item de sugerencia del dropdown Alpine, mostrar el padre si existe:

```html
<span>
    <span class="font-medium text-teal-600" x-text="test.code"></span>
    <span class="text-gray-600" x-text="' - ' + test.name"></span>
    <template x-if="test.parent_name">
        <span class="text-xs text-gray-400 ml-1" x-text="'(hijo de ' + test.parent_name + ')'"></span>
    </template>
</span>
```

### 4.2 — Veterinario (JSON response de searchTests)

En la respuesta JSON del endpoint, incluir `parent_name`:

```php
return response()->json($tests->map(function (Test $test) use ($rate) {
    $nbu = (float) ($test->nbu ?? 0);
    return [
        'id'          => $test->id,
        'code'        => $test->code,
        'name'        => $test->name,
        'price'       => round($nbu * $rate, 2),
        'parent_name' => $test->parentTests->first()?->name
                         ?? $test->parentTest?->name,
    ];
}));
```

Para que `parentTests` esté disponible, agregar eager loading en la query de PASO 3:

```php
->with(['parentTests', 'parentTest'])
->get(['id', 'code', 'name', 'nbu']);
```

En la vista Blade de VetAdmission (el dropdown de búsqueda), mostrar el padre
si existe junto al resultado — seguir el mismo patrón del dropdown existente.

### 4.3 — Lab Clínico

Mismo patrón: en la respuesta JSON de `searchTests`, agregar `parent_name` si el test
tiene padre. Revisar la vista de admisión del lab clínico para mostrar ese dato.

## PASO 5 — Verificación manual

1. **Muestras create**: buscar "Globulos" o el nombre de una determinación hija → debe
   aparecer en el dropdown con la indicación del padre. Agregarla → debe quedar en la
   tabla de determinaciones seleccionadas. Guardar el protocolo → la determinación
   debe aparecer en el show.

2. **Muestras show (agregar post-protocolo)**: en el select de "Agregar Determinación",
   deben aparecer las determinaciones hijas disponibles (las que no están ya en el
   protocolo).

3. **Veterinario**: en la admisión vet, buscar una determinación hija → debe aparecer
   en la búsqueda AJAX con indicación del padre. Agregarla al protocolo.

4. **Lab Clínico**: buscar una determinación hija (ej: un sub-análisis de hemograma) →
   debe aparecer en la búsqueda AJAX.

5. Correr los tests existentes para verificar que no rompimos nada:

```bash

php artisan test --filter=VetAdmission
php artisan test --filter=LabAdmission
```
