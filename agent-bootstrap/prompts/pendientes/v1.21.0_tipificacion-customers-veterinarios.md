## Descripción

Agrega un campo `type` (JSON array) al modelo Customer para clasificarlos por tipo de negocio
(obra_social, aguas, veterinario, clinico, particular, laborales). Remueve las constraints unique
de `taxId` (CUIT) y `email` para permitir duplicados legítimos (ej: obras sociales facturadas a
un mismo ente recaudador).

Crea la tabla `veterinarians` con CRUD de profesionales veterinarios asociados a un Customer
de tipo "veterinario". Estos veterinarios podrán recibir resultados y, a futuro, loguearse al sistema.

{

    protected $fillable = [
        'name', 'phone', 'email', 'matricula',
        'customer_id', 'user_id', 'is_active',
    ];

    protected $casts = [
        'is_active' => 'boolean',

    {

    public function user()

## PASO 7 — VeterinarianController (CRUD)

Crear `app/Http/Controllers/VeterinarianController.php`.

El CRUD de veterinarios se gestiona **dentro del contexto de un customer**. Las rutas serían anidadas:

```php
Route::resource('customer.veterinarians', VeterinarianController::class)->except(['create', 'edit', 'show']);
```

Esto genera rutas como:
- `GET /customer/{customer}/veterinarians` — listado (JSON para AJAX)
- `POST /customer/{customer}/veterinarians` — crear
- `PUT /customer/{customer}/veterinarians/{veterinarian}` — actualizar
- `DELETE /customer/{customer}/veterinarians/{veterinarian}` — eliminar

Métodos:
- `index(Customer $customer)`: retorna JSON con los veterinarios del customer
- `store(Request $request, Customer $customer)`: validar name (required), phone (required), email, matricula. Crear.
- `update(Request $request, Customer $customer, Veterinarian $veterinarian)`: validar y actualizar.
- `destroy(Customer $customer, Veterinarian $veterinarian)`: eliminar.

**Verificación:** `php artisan route:list | grep veterinarian` muestra las rutas anidadas.

## Notas / Consideraciones

- El campo `type` es JSON array, no un string. Permite que un customer sea de múltiples tipos simultáneamente.
- La validación de `taxId` en `store()` y `update()` actualmente tiene `unique:customers,taxId`. Esto **debe removerse** porque el CUIT se puede repetir (caso ente recaudador).
- Los customers existentes se migran como `["aguas"]` por defecto. Si el usuario necesita reclasificar, lo hace desde la UI.
- La tabla `veterinarians` tiene `user_id` nullable preparado para el futuro login, pero no se implementa en esta versión.
- La sección de veterinarios en `edit.blade.php` usa Alpine.js + fetch para CRUD inline sin recargar la página.
- El modelo `Insurance` (obras sociales) es una entidad separada de `Customer` y no se toca en esta versión.
