# v1.1.0 — Normalización de line endings

**VERSIÓN:** v1.1.0  
**SLUG:** line-endings  
**DEPENDENCIAS:** v1.0.0  
**ESTIMACIÓN:** 30min  
**PRIORIDAD:** alta

---

## Descripción

El repositorio tiene 26+ archivos que aparecen como "modificados" en git status cuando no lo están — son diferencias de CRLF (Windows) vs LF (Unix) en los line endings. Esto ensucia el working tree, dificulta los diffs reales y genera confusión.

Esta versión configura `.gitattributes` para normalizar line endings, re-normaliza todos los archivos del repositorio y crea `.editorconfig` para mantener consistencia entre editores.

## PASO 2 — Crear o actualizar .gitattributes

Crear `.gitattributes` en la raíz del proyecto con reglas para normalizar line endings a LF en el repositorio:

```
# Normalización de line endings
* text=auto eol=lf

# PHP
*.php text eol=lf
*.blade.php text eol=lf

# Frontend
*.js text eol=lf
*.css text eol=lf
*.scss text eol=lf
*.vue text eol=lf

# Markup / Data
*.html text eol=lf
*.md text eol=lf
*.json text eol=lf
*.xml text eol=lf
*.yaml text eol=lf
*.yml text eol=lf

# Config
*.env text eol=lf
*.ini text eol=lf
.editorconfig text eol=lf
.gitattributes text eol=lf
.gitignore text eol=lf

# Shell
*.sh text eol=lf
*.bash text eol=lf

# Binarios (no normalizar)
*.png binary
*.jpg binary
*.jpeg binary
*.gif binary
*.ico binary
*.pdf binary
*.zip binary
*.woff binary
*.woff2 binary
*.ttf binary
*.eot binary
```

**Verificación:** `.gitattributes` existe y tiene reglas para todos los tipos de archivo del proyecto.

---

## PASO 3 — Crear .editorconfig

Crear `.editorconfig` en la raíz del proyecto:

```
root = true

[*]
charset = utf-8
end_of_line = lf
indent_style = space
indent_size = 4
insert_final_newline = true
trim_trailing_whitespace = true

[*.md]
trim_trailing_whitespace = false

[*.{yml,yaml}]
indent_size = 2

[*.{js,json,css,scss}]
indent_size = 2

[*.blade.php]
indent_size = 4
```

**Verificación:** `.editorconfig` existe con configuración coherente con el proyecto (PHP = 4 espacios, JS/CSS = 2 espacios, LF en todos).

---

## PASO 4 — Re-normalizar todos los archivos del repositorio

Este paso fuerza a git a re-procesar todos los archivos con las nuevas reglas de `.gitattributes`:

```bash
# Agregar los archivos de configuración primero
git add .gitattributes .editorconfig

# Re-normalizar todo el repositorio
git rm --cached -r .
git reset HEAD
git add .
```

Verificar el estado:

```bash
git status
git diff --stat --cached
```

Los archivos que antes aparecían como "modificados" por CRLF/LF ahora deben estar normalizados y aparecer como cambios reales en el staging.

**Verificación:** `git status` muestra los archivos re-normalizados como cambios staged. No debe haber archivos "modificados" fantasma después del commit.

## Notas / Consideraciones

- Después de este commit, cada desarrollador que haga pull va a ver todos los archivos como "modificados" una única vez — es normal, es la re-normalización
- Si hay otros clones del repositorio, deben hacer `git pull` y luego `git checkout -- .` para sincronizar los line endings
- La regla `* text=auto eol=lf` normaliza a LF en el repo; en Windows git puede hacer checkout con CRLF si `core.autocrlf` está en `true`
