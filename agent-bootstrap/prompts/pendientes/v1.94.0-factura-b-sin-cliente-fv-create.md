## Descripción

Hoy `sales_invoices.customer_id` es **obligatorio** (FK `constrained()` en migración base) y `SalesInvoiceController@store` / `@update` validan `customer_id` como `required`. `AfipService::createVoucher()` asume siempre `$invoice->customer` para `DocTipo`, `DocNro` y `CondicionIVAReceptorId`.

Se necesita emitir **Factura B** (y solo en alcance **create** manual en `/sales-invoices/create`) cuando el receptor **no** debe persistirse como **Cliente** del maestro: los datos del receptor viven **solo en la factura** (snapshot), con validación alineada a AFIP/WSFEv1.

**Fuera de alcance explícito:** facturar desde protocolo con el mismo patrón; recibos de cobro contra estas FC sin `customer_id` (no tocar flujos de cobranza salvo evitar errores obvios); Notas de crédito asociadas.

## PASO 1 — Modelo de datos: `customer_id` nullable + snapshot del receptor

1. **Migración** nueva que:
   - Haga `customer_id` **nullable** en `sales_invoices` (eliminar FK si hace falta y recrear nullable `foreignId()->nullable()->constrained()->nullOnDelete()` o equivalente según convención del proyecto).
   - Agregue columnas para receptor ocasional, por ejemplo (ajustar nombres a convención Laravel del repo):
     - `receiver_name` string (razón social o nombre visible en PDF)
     - `receiver_tax_condition` string nullable (valores compatibles con `AfipService::getDocTipo` / `getCondicionIvaReceptor`, ej. `consumidor final`, `monotributista`, `exento` — reutilizar strings ya usados en `Customer`)
     - `receiver_document_type` smallInteger nullable (código AFIP **DocTipo**: 80 = CUIT, 96 = DNI, 99 = CF sin identificar, etc.) **o** derivar siempre de `receiver_tax_condition` + documento como hoy; documentar la decisión en código mínimo.
     - `receiver_document_number` string nullable (solo dígitos o con guiones; normalizar al enviar a AFIP como en `Customer`)

2. Actualizar `SalesInvoice::$fillable` y `$casts` si aplica.

**Verificación:** `php artisan migrate` en entorno de desarrollo sin romper FC existentes (todas con `customer_id` set).

## PASO 2 — Reglas de negocio y validación (`store` / `update`)

1. Si `voucher_type === 'A'`: mantener **`customer_id` obligatorio** (sin cambios de UX para A).

2. Si `voucher_type === 'B'` (y, si el producto lo exige igual para `C`, definir regla explícita; por defecto **solo B** según pedido PM):
   - **Opción A:** `customer_id` presente → flujo actual.
   - **Opción B:** “Sin cliente” → `customer_id` null y obligatorios los campos snapshot del receptor (nombre + condición IVA + documento según reglas AFIP para ese DocTipo).

3. Validación condicional en `store` y `update` (Request dedicado opcional si reduce complejidad).

4. Al crear `SalesInvoice::create([...])`, persistir `customer_id` null y snapshot cuando corresponda.

**Verificación:** pruebas manuales: B con cliente sigue OK; B sin cliente con datos mínimos pasa validación; A sin cliente falla.
