## Descripción

Hoy el recibo de cobro **ya permite** imputar menos que el saldo de la factura: cada ítem guarda un monto parcial y al confirmar la factura pasa a `parcialmente_cobrada` con saldo pendiente.

El usuario percibe un bloqueo porque la pantalla exige que **la suma de “Monto a cobrar”** coincida con **la suma de medios de pago**. Si al seleccionar la factura el sistema prellena el saldo completo pero el usuario ingresa un cobro menor en medios, aparece el error genérico sin dejar claro que debe **reducir el monto a cobrar** en la tabla de facturas.

Esta versión mejora la **comprensión del flujo**, ofrece una **acción explícita** cuando hay una sola factura seleccionada para alinear montos con los medios, y agrega **validación en servidor** (monto por factura ≤ saldo vigente, pertenencia empresa/cliente).

## PASO 1 — Validación servidor en `validateCollectionReceiptRequest`

Después de las reglas `exists` de facturas, para cada elemento de `invoices`:

1. Cargar `SalesInvoice` con `company_id` = empresa activa y `customer_id` = `customer_id` del request (o equivalente según el flujo actual).
2. Calcular saldo imputable en este momento: usar `balance` actual del modelo (es el saldo pendiente antes de aplicar **este** recibo).
3. Si `amount` > saldo (+ tolerancia 0,01 si aplica), rechazar con mensaje claro por índice, p. ej. `invoices.{i}.amount`.
4. Si la factura no pertenece al cliente o a la empresa, 403/validación según convención del proyecto.

**Verificación:** intentar POST manual o test con monto mayor al saldo y debe fallar validación; monto parcial válido debe pasar.

## Notas / Consideraciones

- **Varias facturas:** no implementar reparto automático desde el total de medios sin regla de negocio acordada; el botón “igualar” solo con una factura seleccionada evita decisiones incorrectas.
- **Concurrencia:** dos usuarios cobrando la misma factura; la validación por saldo al guardar reduce pero no elimina condiciones de carrera; no es scope de esta versión salvo que el proyecto ya tenga patrón (locks/pessimistic).
- El comportamiento actual de `assertPaymentsMatchTotal` **se mantiene**: sigue siendo obligatorio que suma(medios) = suma(montos a cobrar por factura).
