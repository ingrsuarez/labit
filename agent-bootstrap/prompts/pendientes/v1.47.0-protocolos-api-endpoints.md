# v1.47.0 — Endpoints GET de protocolos (clínico, muestras, vet) para LISCOM

**VERSIÓN:** v1.47.0
**SLUG:** protocolos-api-endpoints
**DEPENDENCIAS:** v1.46.0 (auth con API key + middleware + `lab_branch_id` en `api_clients`)
**ESTIMACIÓN:** 1 día
**PRIORIDAD:** alta — habilita el primer flujo real de la cadena LISCOM (lectura de protocolos)

---

## Descripción

Exponer los endpoints GET de la API v1 para que LISCOM pueda:

1. **Sincronizar automáticamente los protocolos del día** (sync diaria + actualización al recibir scans).
2. **Resolver un barcode** cuando un equipo escanea (lookup directo).
3. **Consultar protocolos pasados bajo demanda** cuando el operador en LISCOM necesita
   procesar una muestra vieja que no está cacheada.

Soporta los **tres tipos de protocolo** que existen en labit:

| Modelo | Tipo en API | Prefijo barcode |
|---|---|---|
| `Admission` | `clinical` | `C-` |
| `Sample` | `sample` | `A-` |
| `VetAdmission` | `vet` | (revisar en código) |

La estructura de respuesta es **unificada** entre los 3 tipos. Los datos del paciente
**no incluyen DNI por defecto** (decisión PM: minimizar PII enviada por la red).
La key de API puede configurarse a nivel `standard` para incluir DNI si se justifica.

Filtrado automático invisible: la API key trae `lab_branch_id` y `company_id`, y todas
las queries filtran por esos valores. Una key NUNCA puede ver protocolos de otra sede.

```php
namespace App\Enums;

enum ProtocolType: string
{
    case Clinical = 'clinical';
    case Sample = 'sample';
    case Vet = 'vet';

    public function modelClass(): string
    {
        return match($this) {
            self::Clinical => \App\Models\Admission::class,
            self::Sample => \App\Models\Sample::class,
            self::Vet => \App\Models\VetAdmission::class,
        };
    }

    public static function fromBarcodePrefix(string $code): ?self
    {
        // Detecta el tipo por prefijo del protocol_number
        if (str_starts_with($code, 'A-')) return self::Sample;
        if (str_starts_with($code, 'C-')) return self::Clinical;
        // VET: revisar prefijo real en VetAdmission::generateProtocolNumber()
        if (str_starts_with($code, 'V-')) return self::Vet;
        return null;
    }
}
```

## PASO 4 — Service de lookup unificado

Crear `app/Services/Api/ProtocolLookupService.php`:

```php
class ProtocolLookupService
{
    public function findByBarcode(string $code, ApiClient $client): ?Model
    {
        $type = ProtocolType::fromBarcodePrefix($code);

        if ($type) {
            // Lookup directo (caso normal)
            $modelClass = $type->modelClass();
            return $modelClass::where('protocol_number', $code)
                ->where('lab_branch_id', $client->lab_branch_id)
                ->where('company_id', $client->company_id)
                ->first();
        }

        // Fallback: probar los 3 modelos (caso prefijo no estándar)
        foreach (ProtocolType::cases() as $type) {
            $found = $type->modelClass()::where('protocol_number', $code)
                ->where('lab_branch_id', $client->lab_branch_id)
                ->where('company_id', $client->company_id)
                ->first();
            if ($found) return $found;
        }

        return null;
    }

    public function query(array $filters, ApiClient $client): Collection
    {
        // Estrategia A elegida (recomendada por PM): 3 queries paralelas + merge en PHP
        // Si performance se vuelve un problema (>200ms p95), migrar a vista SQL en hotfix
        $results = collect();

        $types = $this->resolveTypes($filters['type'] ?? null);

        foreach ($types as $type) {
            $modelClass = $type->modelClass();
            $query = $modelClass::query()
                ->where('lab_branch_id', $client->lab_branch_id)
                ->where('company_id', $client->company_id);

            $this->applyFilters($query, $filters, $type);

            $results = $results->concat(
                $query->with($this->eagerLoadFor($type))->get()
            );
        }

        return $results->sortByDesc('updated_at')->values();
    }

    private function applyFilters($query, array $filters, ProtocolType $type): void
    {
        // date_from / date_to (default: hoy)
        $dateColumn = $type === ProtocolType::Sample ? 'entry_date' : 'date';
        $dateFrom = $filters['date_from'] ?? today()->toDateString();
        $dateTo = $filters['date_to'] ?? today()->toDateString();
        $query->whereBetween($dateColumn, [$dateFrom, $dateTo]);

        // status CSV
        $statuses = $filters['status'] ?? 'pending,in_progress';
        if ($statuses !== '*') {
            $query->whereIn('status', explode(',', $statuses));
        }

        // updated_since
        if (!empty($filters['updated_since'])) {
            $query->where('updated_at', '>', $filters['updated_since']);
        }

        // protocol_number exacto
        if (!empty($filters['protocol_number'])) {
            $query->where('protocol_number', $filters['protocol_number']);
        }
    }

    private function resolveTypes(?string $typeFilter): array
    {
        if (!$typeFilter) return ProtocolType::cases();

        return collect(explode(',', $typeFilter))
            ->map(fn($t) => ProtocolType::tryFrom(trim($t)))
            ->filter()
            ->all();
    }

    private function eagerLoadFor(ProtocolType $type): array
    {
        return match($type) {
            ProtocolType::Clinical => ['patient', 'labBranch', 'admissionTests.test.material'],
            ProtocolType::Sample => ['customer', 'labBranch', 'determinations.test.material'],
            ProtocolType::Vet => ['animal.species', 'animal.breed', 'customer', 'labBranch', 'tests.test.material'],
        };
    }
}
```

> ⚠️ Adaptá los nombres exactos de relaciones (especialmente para vet) según los modelos.

```php
public function __construct(private ProtocolLookupService $service) {}

public function index(Request $request)
{
    $filters = $request->only([
        'date_from', 'date_to', 'status', 'type',
        'updated_since', 'protocol_number'
    ]);

    $perPage = min((int) $request->input('per_page', 100), 500);

    $all = $this->service->query($filters, $request->get('api_client'));

    // Paginación manual sobre la collection mergeada
    $page = (int) $request->input('page', 1);
    $items = $all->forPage($page, $perPage);

    return ProtocolResource::collection($items)
        ->additional([
            'meta' => [
                'current_page' => $page,
                'per_page' => $perPage,
                'total' => $all->count(),
                'last_page' => max(1, (int) ceil($all->count() / $perPage)),
            ],
        ]);
}

public function showByBarcode(Request $request, string $code)
{
    $found = $this->service->findByBarcode($code, $request->get('api_client'));

    if (!$found) {
        return response()->json(['error' => 'Protocol not found'], 404);
    }

    return new ProtocolResource($found);
}

public function show(Request $request, string $type, int $id)
{
    $protocolType = ProtocolType::tryFrom($type);
    if (!$protocolType) {
        return response()->json(['error' => 'Invalid type'], 400);
    }

    $client = $request->get('api_client');
    $modelClass = $protocolType->modelClass();
    $found = $modelClass::where('id', $id)
        ->where('lab_branch_id', $client->lab_branch_id)
        ->where('company_id', $client->company_id)
        ->first();

    if (!$found) {
        return response()->json(['error' => 'Protocol not found'], 404);
    }

    return new ProtocolResource($found);
}
```

## Notas / Consideraciones

- **Performance:** Estrategia simple (3 queries + merge PHP) elegida por el PM. Si en
  pruebas con datos reales el endpoint pasa de 200ms p95, abrir hotfix v1.47.1 con
  vista SQL materializada.
- **Tensión barcode:** Hoy `barcode == protocol_number` y NO incluye material. El campo
  `barcode` en la respuesta refleja eso. La discusión sobre cambiar el formato del
  barcode se hace en v1.49.0. NO hagas cambios al barcode en esta versión.
- **`test_code` vacío:** El campo `external_code` (código corto del test para HL7) no existe
  hoy en el modelo `Test`. La respuesta lo devuelve `null` por ahora. Se completa con el
  mapeo de códigos por equipo en v1.49.0.
- **VetAdmission:** revisar el modelo real antes de asumir nombres de relaciones (puede
  ser `customer` o `owner`, `animal` puede tener otro nombre, etc.). Adaptar el
  `ProtocolResource::buildPatientData()` y el `eagerLoadFor()` a la realidad.
- **Sample sin paciente humano:** las muestras de aguas/alimentos no tienen `Patient`.
  El `patient` en la response devuelve los datos del cliente (Customer) y los campos
  específicos de sample (location, batch, product_name).
- **Eager loading obligatorio:** sin él, el endpoint genera N+1 queries y se cae con
  100+ protocolos. Verificar en logs o con Telescope/Debugbar.
- **No agregar Sanctum, no agregar rate limiting** — siguen los lineamientos de v1.46.0.
- **Validación de input:** los filtros deberían validarse con un FormRequest en versión
  futura. Por ahora la validación es laxa (los inválidos se ignoran).
