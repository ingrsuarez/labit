## Descripción

Adapta el módulo de compras para operar por empresa. Cada orden de compra, remito, factura de compra y orden de pago pertenece a una empresa específica. Todos los listados, formularios y reportes filtran por la empresa activa en sesión.

Los proveedores (`suppliers`), insumos (`supplies`) y categorías de insumos (`supply_categories`) siguen siendo globales — no se les agrega `company_id`. Los movimientos de stock también se mantienen globales por ahora (el inventario físico es compartido).

## PASO 1 — Migraciones: agregar `company_id`

Crear UNA migración que agregue `company_id` a todas las tablas del módulo de compras:

```bash
php artisan make:migration add_company_id_to_purchase_tables
```

Tablas a modificar:
- `purchase_orders`
- `delivery_notes`
- `purchase_invoices`
- `payment_orders`

Si existe `purchase_quotation_requests` → agregarle `company_id` también.

Para cada tabla:

```php
$table->foreignId('company_id')->nullable()->after('id')->constrained('companies');
```

```bash
php artisan migrate
```

**Verificación:** Las tablas tienen la columna `company_id`. No hay errores de migración.

## PASO 2 — Actualizar modelos

Para cada modelo (`PurchaseOrder`, `DeliveryNote`, `PurchaseInvoice`, `PaymentOrder`, y `PurchaseQuotationRequest` si existe):

1. Agregar `'company_id'` al array `$fillable`.
2. Agregar relación:

```php
public function company(): BelongsTo
{
    return $this->belongsTo(Company::class);
}
```

3. En el modelo `Company`, agregar las relaciones inversas:

```php
public function purchaseOrders(): HasMany { return $this->hasMany(PurchaseOrder::class); }
public function deliveryNotes(): HasMany { return $this->hasMany(DeliveryNote::class); }
public function purchaseInvoices(): HasMany { return $this->hasMany(PurchaseInvoice::class); }
public function paymentOrders(): HasMany { return $this->hasMany(PaymentOrder::class); }
```

**Verificación:** Cada modelo tiene la relación `company()` y `company_id` es fillable.

## PASO 3 — Actualizar controladores (filtrar por empresa activa)

### Patrón general (mismo que v2.1.0):

**En `index`:** filtrar por empresa activa:

```php
$query->where('company_id', active_company_id());
```

**En `store`:** asignar empresa activa:

```php
$data['company_id'] = active_company_id();
```

**En `show`/`edit`/`update`/`destroy`:** verificar pertenencia:

```php
abort_if($record->company_id !== active_company_id(), 403);
```

### Controladores a modificar:

1. **`PurchaseOrderController`** — index, store, show, edit, update, destroy
2. **`DeliveryNoteController`** — index, store, show, edit, update, destroy
3. **`PurchaseInvoiceController`** — index, store, show, edit, update, destroy
4. **`PaymentOrderController`** — index, store, show, edit, update, destroy

### Relaciones entre documentos de compra:

El flujo completo es: Cotización → OC → Remito → Factura → Orden de pago. Al crear un documento derivado (ej: remito desde OC), verificar que el documento origen pertenece a la empresa activa. Los dropdowns de selección de OC/remito/factura deben filtrar por empresa activa.

**Verificación:** Los listados solo muestran registros de la empresa activa. El flujo de compra completo funciona dentro de una empresa.

## Notas / Consideraciones

- Proveedores, insumos y categorías son GLOBALES — no llevan `company_id`.
- Movimientos de stock son globales por ahora (inventario físico compartido). Si se necesita stock por empresa, se planifica como versión separada.
- Los ítems de cada documento (PurchaseOrderItem, DeliveryNoteItem, PurchaseInvoiceItem, PaymentOrderItem) no necesitan `company_id` — se acceden a través de su documento padre.
- El `company_id` es `nullable` para robustez, pero el controlador siempre lo setea.
- v2.1.0 y v2.2.0 son independientes entre sí (ambas dependen solo de v2.0.0), por lo que pueden ejecutarse en paralelo si hay múltiples agentes.
