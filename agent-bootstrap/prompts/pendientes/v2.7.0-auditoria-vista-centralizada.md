# v2.7.0 — Vista centralizada de auditoría

**VERSIÓN:** v2.7.0

**DEPENDENCIAS:** v2.5.0

## Descripción

Agrega una sección "Auditoría" en el panel administrativo con una tabla centralizada y filtrable de todos los logs del sistema. Permite al admin buscar "qué hizo X usuario ayer" o "quién tocó este módulo" sin ir registro por registro. Usa la infraestructura de `audit_logs` creada en v2.5.0.

## PASO 2 — Controlador `AuditController`

Crear `app/Http/Controllers/AuditController.php` con un método `index`:

```php
class AuditController extends Controller
{
    public function index(Request $request)
    {
        $query = AuditLog::with('user')->latest();

        // Filtro por usuario
        if ($request->filled('user_id')) {
            $query->where('user_id', $request->user_id);
        }

        // Filtro por acción
        if ($request->filled('action')) {
            $query->where('action', $request->action);
        }

        // Filtro por módulo (auditable_type)
        if ($request->filled('module')) {
            $query->where('auditable_type', $request->module);
        }

        // Filtro por rango de fechas
        if ($request->filled('date_from')) {
            $query->whereDate('created_at', '>=', $request->date_from);
        }
        if ($request->filled('date_to')) {
            $query->whereDate('created_at', '<=', $request->date_to);
        }

        // Búsqueda por texto en descripción
        if ($request->filled('search')) {
            $query->where('description', 'like', '%' . $request->search . '%');
        }

        $logs = $query->paginate(50);

        // Datos para los filtros
        $users = \App\Models\User::orderBy('name')->get(['id', 'name']);
        $actions = AuditLog::distinct()->pluck('action')->sort();
        $modules = AuditLog::distinct()
            ->whereNotNull('auditable_type')
            ->pluck('auditable_type')
            ->map(fn($type) => class_basename($type))
            ->unique()
            ->sort();

        return view('audit.index', compact('logs', 'users', 'actions', 'modules'));
    }
}
```

**Verificación:** El controlador no da errores de sintaxis. La query con filtros funciona correctamente.

## PASO 4 — Vista `audit/index.blade.php`

Crear `resources/views/audit/index.blade.php` usando `<x-admin-layout>` (o el layout admin que use el proyecto).

### 4.1 — Header

Título "Auditoría del Sistema" con subtítulo "Registro de actividad de todos los módulos".

### 4.2 — Barra de filtros

Una barra horizontal con los filtros, estilo consistente con los listados existentes del sistema:

- **Buscar** — input de texto para buscar en descripción (con icono de lupa)
- **Usuario** — select/dropdown con todos los usuarios
- **Acción** — select con las acciones existentes (created, updated, deleted, login, logout, login_failed, validated, unvalidated, results_loaded, pdf_generated, email_sent)
- **Módulo** — select con los tipos de modelo auditados (Patient, Admission, Sample, etc.)
- **Desde** — date input
- **Hasta** — date input
- **Botón "Filtrar"** — submit del formulario GET
- **Botón "Limpiar"** — link a `/audit` sin parámetros

Los filtros se envían como query params (GET) para que sean bookmarkeables y funcionen con paginación.

### 4.3 — Tabla de resultados

Tabla con las siguientes columnas:

| Columna | Contenido | Notas |
|---------|-----------|-------|
| Fecha/Hora | `created_at` formateado `dd/mm/yyyy HH:mm` | Ordenado desc por defecto |
| Usuario | `user_name` (del snapshot) | Si `user_id` existe, link al usuario |
| Acción | Badge con color según tipo | Verde: created/login. Azul: updated/results_loaded. Rojo: deleted/login_failed. Teal: validated. Amarillo: unvalidated. Gris: logout/pdf_generated/email_sent |
| Descripción | `description` | Texto completo |
| Módulo | Nombre legible del `auditable_type` | "Paciente", "Admisión", "Muestra". Si tiene `auditable_id`, link al registro |
| IP | `ip_address` | Texto gris pequeño |

### 4.4 — Mapeo de nombres legibles

Los `auditable_type` son nombres de clase (ej: `App\Models\Patient`). Mapearlos a nombres legibles:

```php
$moduleNames = [
    'App\Models\Patient'   => 'Paciente',
    'App\Models\Admission' => 'Admisión',
    'App\Models\Sample'    => 'Muestra',
    // Agregar más cuando se extiendan los módulos auditados
];
```

Esto puede ir en el modelo `AuditLog` como accessor o en un helper.

### 4.5 — Mapeo de acciones a badges

```
created        → badge verde    "Creación"
updated        → badge azul     "Edición"
deleted        → badge rojo     "Eliminación"
login          → badge verde    "Login"
logout         → badge gris     "Logout"
login_failed   → badge rojo     "Login fallido"
validated      → badge teal     "Validación"
unvalidated    → badge amarillo "Desvalidación"
results_loaded → badge azul     "Carga de resultados"
pdf_generated  → badge gris     "PDF generado"
email_sent     → badge gris     "Email enviado"
```

### 4.6 — Paginación

Paginación estándar de Laravel al final de la tabla (50 por página). Mostrar contador: "Mostrando X-Y de Z registros".

### 4.7 — Estado vacío

Si no hay logs (o los filtros no devuelven resultados), mostrar mensaje: "No se encontraron registros de actividad."

**Verificación:** La vista carga correctamente. Los filtros funcionan individualmente y combinados. La paginación respeta los filtros. Los badges de acción muestran los colores correctos.

## PASO 6 — Links desde la tabla a los registros

Cuando un log tiene `auditable_type` y `auditable_id`, la columna Módulo debe ser un link clickeable que lleva al show del registro.

Agregar un método al modelo `AuditLog` para resolver la URL:

```php
public function getAuditableUrlAttribute(): ?string
{
    if (!$this->auditable_type || !$this->auditable_id) {
        return null;
    }

    return match ($this->auditable_type) {
        'App\Models\Patient'   => route('patients.show', $this->auditable_id),
        'App\Models\Admission' => route('lab.admissions.show', $this->auditable_id),
        'App\Models\Sample'    => route('samples.show', $this->auditable_id),
        default                => null,
    };
}
```

> Investigar los nombres reales de las rutas. Adaptar según lo que existe en `routes/web.php`.
> Si el registro fue eliminado (no existe en la DB), el link no se muestra — usar `try/catch` o verificar existencia.

**Verificación:** Hacer click en "Paciente #5" en la tabla de auditoría lleva al show del paciente.

## Notas / Consideraciones

- Esta versión depende de v2.5.0 — necesita la tabla `audit_logs` con datos.
- Los logs de auth (login/logout/failed) no tienen `auditable_type`/`auditable_id` — aparecen en la tabla pero sin link a registro.
- Cuando se agregue v2.6.0 (auditoría financiera/admin), los nuevos módulos aparecen automáticamente en los filtros sin cambiar esta vista.
- El mapeo de `auditable_type` → nombre legible y de `auditable_type` → URL debe mantenerse actualizado cuando se agregan nuevos módulos auditados. Considerar centralizar en un config o en el modelo AuditLog.
- La paginación a 50 registros es un buen balance. Si el volumen crece mucho, considerar exportación a Excel en una versión futura.
- No se incluye gráficos ni dashboard de auditoría (ej: "logins por día") — eso sería una versión futura si se necesita.
