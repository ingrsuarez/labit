## Descripción

Restringe cada layout (Admin, Lab, Portal) según el rol del usuario. Hoy cualquier rol que pase `check.access` ve todas las secciones del admin (Personal, Ausencias, Liquidaciones, Configuración) sin restricción. Esta versión:

1. Crea permisos de sección para las áreas desprotegidas del admin.
2. Protege el sidebar y las rutas con `@can` y middleware.
3. Redirige al usuario al layout correcto según su rol al loguearse.
4. Controla los links de navegación entre layouts (Admin ↔ Lab ↔ Portal).

**Regla de negocio:**
- `admin` → ve todo el layout admin
- `contador` → solo Liquidaciones, Ausencias, Compras, Ventas
- `compras` → solo Compras (puede o no ser empleado)
- `ventas` → solo Ventas (puede o no ser empleado)
- `recepcion-lab`, `tecnico-lab`, `bioquimico` → solo Lab + su Portal personal
- `empleado` (solo) → solo Portal

## PASO 4 — Actualizar `CheckSystemAccess` para redirección por rol

Archivo: `app/Http/Middleware/CheckSystemAccess.php`

### 4.1 — Definir grupos de roles

```php
protected array $portalOnlyRoles = ['empleado', 'employee'];

protected array $labOnlyRoles = ['recepcion-lab', 'tecnico-lab', 'bioquimico'];

protected array $adminCapableRoles = ['admin', 'contador', 'compras', 'ventas'];
```

### 4.2 — Lógica de redirección

La prioridad es: **admin-capable > lab > portal > pending**.

```
1. Si el usuario tiene AL MENOS UN rol admin-capable → pasa al admin layout ($next)
2. Si el usuario tiene AL MENOS UN rol lab-only (y ningún admin-capable) → redirect a lab
3. Si el usuario SOLO tiene roles de portal → redirect a portal (si tiene employee) o access.pending
4. Si no tiene roles pero tiene permisos directos → pasa al admin
5. Si tiene employee pero sin roles ni permisos → redirect a portal
6. Sin nada → access.pending
```

> CLAVE: Un usuario con `empleado` + `recepcion-lab` tiene un rol lab → va al lab (regla 2).
> Un usuario con `empleado` + `compras` tiene un rol admin-capable → va al admin (regla 1).

Redirigir a la ruta principal del lab. Investigar cuál es — probablemente `lab.index` o similar. Buscar en `routes/web.php` la primera ruta del grupo lab.

**Verificación:** Login con `recepcion-lab` → aterriza en lab. Login con `compras` → aterriza en admin (solo ve sección Compras). Login con `empleado` solo → aterriza en portal.

## PASO 6 — Dashboard: redirección inteligente

Archivo: `app/Http/Controllers/DashboardController.php` (o la ruta `/dashboard` en `web.php`)

El dashboard actual muestra stats de RRHH. Para roles que no tienen acceso a RRHH (ej: `compras`, `ventas`), redirigir a su sección principal:

```php
$user = auth()->user();

if ($user->hasRole('admin')) {
    // Dashboard completo (comportamiento actual)
}

if ($user->hasAnyRole(['recepcion-lab', 'tecnico-lab', 'bioquimico'])) {
    return redirect()->route('lab.index'); // o la ruta principal del lab
}

if ($user->hasRole('compras') && !$user->hasAnyRole(['admin', 'contador'])) {
    return redirect()->route('purchase-orders.index'); // o la ruta principal de compras
}

if ($user->hasRole('ventas') && !$user->hasAnyRole(['admin', 'contador'])) {
    return redirect()->route('sales-invoices.index'); // o la ruta principal de ventas
}

// Contador y admin ven el dashboard completo
```

> NOTA: Este paso puede simplificarse si CheckSystemAccess ya redirige correctamente en PASO 4. Evaluar si es redundante.

**Verificación:** Login con `compras` → no ve dashboard RRHH, va directo a compras. Login con `admin` → ve dashboard RRHH normalmente.

## Notas / Consideraciones

- Los roles `compras` y `ventas` pueden existir sin ser empleados (terceros). En ese caso no tienen acceso a Portal ni a Lab.
- Si un usuario tiene `compras` + `ventas`, ve ambas secciones en el admin.
- Si un usuario tiene `empleado` + `compras`, tiene un rol admin-capable → va al admin, ve solo Compras, y puede acceder a Mi Portal si tiene employee vinculado.
- El organigrama (`view.chart`) mantiene su permiso propio — no cambia.
- No se crean permisos granulares dentro de cada sección (crear/editar/eliminar empleado, etc.) — eso es una versión futura.
- El `RolesAndPermissionsSeeder` debe ser idempotente: si se ejecuta de nuevo, no duplica permisos ni rompe asignaciones existentes.
- Hay un posible bug en CheckSystemAccess línea 51: `$next($next)` en vez de `$next($request)`. Corregir si existe.
