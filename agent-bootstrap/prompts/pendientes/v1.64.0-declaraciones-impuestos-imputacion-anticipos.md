**VERSIÓN:** v1.64.0

**DEPENDENCIAS:** v1.63.0 (catálogo de percepciones + pivote de FC). v1.63.1 opcional (suma NC al cálculo). v3.4.0 (Libro Diario/Mayor)

## Descripción

Crear el módulo de **declaraciones de impuestos** que descarga los anticipos acumulados
en las cuentas de crédito fiscal (cargados vía v1.63.0/v1.63.1).

Componentes:

1. **Catálogo `Tax`** — modelo de impuesto con cuenta a pagar.
2. **Asociación percepción ↔ impuesto** — columna `tax_id` en `purchase_perceptions`.
3. **`TaxReturn`** — DDJJ por impuesto y período con monto declarado.
4. **`TaxReturnApplication`** — anticipos imputados a una DDJJ.
5. **Asiento contable** — CR cuentas de anticipo, DB cuenta a pagar, balance queda en
   la cuenta a pagar como saldo a pagar (CR) o a favor (DB).
6. **UI**: CRUD de `Tax`, wizard de DDJJ, listado/show de DDJJ, ajuste a la pantalla
   de saldos de percepciones.

**No incluye el pago** real del saldo a pagar — se hace después con asiento manual del
Libro Diario.

## PASO 7 — Servicio `TaxReturnService` (cálculo y asiento)

Crear `app/Services/TaxReturnService.php` con métodos:

### `availableAdvances(Tax $tax, int $year, ?int $month): Collection`

Devuelve las `PurchaseInvoicePerception` (y `PurchaseCreditNotePerception` si v1.63.1)
elegibles para imputar en una DDJJ del impuesto y período dados.

Filtros:
- Percepciones cuyo `purchase_perception_id` está en `tax->perceptions->pluck('id')`
- Pertenecen a una FC con `issue_date` dentro del período (mensual/trimestral/anual)
- NO están ya imputadas en una DDJJ confirmada (subquery anti-duplicate)

```php
public function availableAdvances(Tax $tax, int $year, ?int $month): Collection
{
    [$dateFrom, $dateTo] = $this->periodRange($tax->frequency, $year, $month);

    $perceptionIds = $tax->perceptions()->pluck('id');

    $alreadyImputed = TaxReturnApplication::query()
        ->whereHas('taxReturn', fn ($q) => $q->where('status', 'confirmed'))
        ->whereNotNull('purchase_invoice_perception_id')
        ->pluck('purchase_invoice_perception_id');

    return PurchaseInvoicePerception::query()
        ->with('purchaseInvoice.supplier')
        ->whereIn('purchase_perception_id', $perceptionIds)
        ->whereHas('purchaseInvoice', fn ($q) => $q
            ->where('company_id', active_company_id())
            ->whereBetween('issue_date', [$dateFrom, $dateTo])
        )
        ->whereNotIn('id', $alreadyImputed)
        ->orderBy('id')
        ->get();
}

private function periodRange(string $frequency, int $year, ?int $month): array
{
    return match ($frequency) {
        'monthly' => [
            Carbon::create($year, $month, 1)->startOfMonth(),
            Carbon::create($year, $month, 1)->endOfMonth(),
        ],
        'quarterly' => [
            Carbon::create($year, $month, 1)->startOfMonth(),
            Carbon::create($year, $month, 1)->addMonths(2)->endOfMonth(),
        ],
        'annual' => [
            Carbon::create($year, 1, 1)->startOfYear(),
            Carbon::create($year, 12, 31)->endOfYear(),
        ],
    };
}
```

### `confirm(TaxReturn $return): JournalEntry`

Genera el asiento al confirmar. Validaciones:
- Estado actual = `draft`
- `applications` no vacías o `declared_amount > 0`
- Ningún anticipo en duplicado con DDJJ ya confirmada

Asiento:

```
Por cada perception aplicada (agrupada por accounting_account_id snapshot):
    CR cuenta_anticipo                  sum(amount_applied)

DB tax.liability_account_id              declared_amount
```

Si declared_amount < applied_total → la diferencia queda como DB en `liability_account`
(saldo a favor en el pasivo).

Si declared_amount > applied_total → la diferencia queda como CR en `liability_account`
(saldo a pagar).

Implementación:

```php
public function confirm(TaxReturn $return): JournalEntry
{
    if (! $return->isDraft()) {
        throw new \DomainException('Solo se pueden confirmar DDJJ en estado borrador.');
    }

    $return->loadMissing(['tax.liabilityAccount', 'applications.purchaseInvoicePerception.accountingAccount']);

    $lines = [];

    // CR cada cuenta de anticipo (agrupada)
    $byAccount = $return->applications
        ->groupBy(fn ($app) => $app->purchaseInvoicePerception?->accounting_account_id
                              ?? $app->purchaseCreditNotePerception?->accounting_account_id);

    foreach ($byAccount as $accountId => $apps) {
        $total = round((float) $apps->sum('amount_applied'), 2);
        if ($total <= 0) continue;
        $account = AccountingAccount::find($accountId);
        $lines[] = [
            'account_code' => $account->code,
            'debit' => 0,
            'credit' => $total,
            'description' => 'Imputación anticipos — '.$return->tax->name.' '.$return->period_label,
        ];
    }

    // DB liability_account
    $declared = round((float) $return->declared_amount, 2);
    if ($declared > 0) {
        $lines[] = [
            'account_code' => $return->tax->liabilityAccount->code,
            'debit' => $declared,
            'credit' => 0,
            'description' => 'DDJJ '.$return->tax->name.' '.$return->period_label,
        ];
    }

    // Si declared < applied → balance contra liability_account (saldo a favor)
    $applied = round((float) $return->applied_total, 2);
    $diff = round($applied - $declared, 2);
    if ($diff > 0) {
        // applied > declared: hay saldo a favor → DB liability extra para balancear
        $lines[] = [
            'account_code' => $return->tax->liabilityAccount->code,
            'debit' => $diff,
            'credit' => 0,
            'description' => 'Saldo a favor — '.$return->tax->name.' '.$return->period_label,
        ];
    }

    $entry = app(AccountingEntryService::class)->createEntry(
        (int) $return->company_id,
        Carbon::create($return->period_year, $return->period_month ?? 12, 1)->endOfMonth(),
        'DDJJ '.$return->tax->name.' '.$return->period_label,
        $return,
        $lines
    );

    $return->update([
        'journal_entry_id' => $entry->id,
        'status' => 'confirmed',
        'confirmed_by' => auth()->id(),
        'confirmed_at' => now(),
    ]);

    return $entry;
}
```

> **Aclaración del balance**: cuando `declared > applied`, el saldo a pagar queda
> automáticamente como CR en `liability_account` porque la línea DB del declared menos
> los CR de los anticipos no balancea — agregar línea CR al `liability_account` por
> la diferencia para balancear (es la deuda a pagar).

Reescribiendo correctamente:

```php
$declared = round((float) $return->declared_amount, 2);
$applied = round((float) $return->applied_total, 2);

if ($declared > 0) {
    $lines[] = [
        'account_code' => $return->tax->liabilityAccount->code,
        'debit' => $declared,
        'credit' => 0,
        'description' => '...'
    ];
}

// Balance forzado: si applied > declared, hay saldo a favor → DB liability
//                  si applied < declared, hay saldo a pagar → CR liability
$diff = round($declared - $applied, 2);
if ($diff > 0) {
    // saldo a pagar
    $lines[] = ['account_code' => $return->tax->liabilityAccount->code,
                'debit' => 0, 'credit' => $diff,
                'description' => 'Saldo a pagar — '.$return->tax->name.' '.$return->period_label];
} elseif ($diff < 0) {
    // saldo a favor: es exceso de anticipos sobre lo declarado
    // queda en la cuenta de pasivo como DB (saldo a favor)
    // ya está implícito por el balance — no hace falta línea extra
}
```

**Validar al final** que `sum(debit) == sum(credit)`.

### `cancel(TaxReturn $return): JournalEntry`

Genera asiento INVERSO (DB y CR intercambiados) y marca `cancelled`.

```php
public function cancel(TaxReturn $return): JournalEntry
{
    if (! $return->isConfirmed()) {
        throw new \DomainException('Solo se pueden anular DDJJ confirmadas.');
    }

    $original = $return->journalEntry;
    if (! $original) throw new \RuntimeException('La DDJJ no tiene asiento original.');

    $reverseLines = $original->lines->map(fn ($line) => [
        'account_code' => $line->account->code,
        'debit' => $line->credit, // intercambio
        'credit' => $line->debit,
        'description' => 'ANULACIÓN — '.$line->description,
    ])->toArray();

    $entry = app(AccountingEntryService::class)->createEntry(
        (int) $return->company_id,
        now(),
        'ANULACIÓN DDJJ '.$return->tax->name.' '.$return->period_label,
        $return,
        $reverseLines
    );

    $return->update([
        'cancellation_journal_entry_id' => $entry->id,
        'status' => 'cancelled',
        'cancelled_by' => auth()->id(),
        'cancelled_at' => now(),
    ]);

    return $entry;
}
```

- Crear DDJJ con 2 aplicaciones, confirmar → ver asiento balanceado
- Anular → asiento reverso, anticipos quedan disponibles para otra DDJJ

## PASO 12 — Documentación

`BLUEPRINT.md`: agregar DD-010:

```
### DD-010 — Declaraciones de impuestos e imputación de anticipos (v1.64.0)

Las DDJJ por impuesto se modelan como `TaxReturn` con aplicaciones explícitas
(`TaxReturnApplication`) que apuntan a las percepciones sufridas (`PurchaseInvoicePerception`,
`PurchaseCreditNotePerception`).

El catálogo `Tax` agrupa las percepciones bajo un mismo impuesto vía `tax_id` en
`purchase_perceptions`. Esto permite que el wizard de DDJJ liste automáticamente los
anticipos elegibles del período.

El asiento al confirmar acredita las cuentas de anticipo imputadas y debita la cuenta
de pasivo (`liability_account_id` del `Tax`). El saldo neto en `liability_account`
representa lo que falta pagar (CR) o el saldo a favor (DB).

El pago real del saldo a pagar NO está incluido en v1.64.0 — se hace con asiento
manual del Libro Diario (v3.4.0). Versión futura puede agregar paso de pago.

Estados: draft, confirmed, cancelled. Anular genera asiento reverso y libera los
anticipos para otra DDJJ.
```
