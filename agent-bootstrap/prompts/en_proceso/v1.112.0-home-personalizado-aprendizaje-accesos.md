## Descripción

Hoy el link **«Administración»** (lab header/sidebar, `navigation-menu`) apunta a
`manage.index` (`/manage/index`), que muestra la **lista de empleados** — vestigio de
antes de **v1.66.0**, cuando el dashboard principal era RRHH.

El objetivo es que `/dashboard` sea el **home personalizado** de cada usuario: una
pantalla con **cards de acceso rápido** (patrón `admin.partials.section-cards`) ordenadas
por **frecuencia de uso real**.

El sistema debe **registrar silenciosamente** cada visita a rutas relevantes en **todo
el sistema** (admin, laboratorio y portal del empleado) y usar ese historial para
ordenar el home. Usuarios nuevos ven un orden por defecto según rol/permisos hasta
acumular historial.

El **dashboard financiero** actual (KPIs de ventas/compras/ingresos/egresos de v1.66.0)
se **mueve** a `/dashboard/financiero` y queda accesible como card «Resumen financiero»
para admin/contador.

**Handoff PM:** `agent-bootstrap/handoffs/v1.112.0-pm-to-dev.md`

### 2.3 — Métodos públicos

```php
public static function shortcuts(): array;
public static function shortcut(string $key): ?array;
public static function shortcutKeyForRoute(?string $routeName): ?string;
public static function shortcutsForUser(User $user, int $limit = 12): array;
```

Lógica de `shortcutsForUser`:

1. Filtrar shortcuts que el usuario **puede** ver (permiso/rol).
2. Cargar stats del usuario (`user_navigation_stats`) en una sola query.
3. Ordenar: `hit_count DESC`, `last_accessed_at DESC`, `default_priority ASC`.
4. Devolver máximo `$limit` items con forma `{ name, description, route, icon }`.
5. Si el usuario no tiene historial, el orden viene solo de `default_priority`.

**Verificación:** unit test básico de filtrado y ordenamiento.

## PASO 3 — Servicio de tracking `NavigationAccessService`

Crear `app/Services/NavigationAccessService.php`:

```php
public function recordHit(int $userId, string $shortcutKey): void
{
    UserNavigationStat::updateOrCreate(
        ['user_id' => $userId, 'shortcut_key' => $shortcutKey],
        []
    )->increment('hit_count', 1, ['last_accessed_at' => now()]);
}
```

Usar `increment` atómico o equivalente seguro en concurrencia.

**Verificación:** test unitario — dos hits → `hit_count = 2`.

## PASO 4 — Middleware `TrackNavigationAccess`

```bash
php artisan make:middleware TrackNavigationAccess
```

Registrar en el grupo `web` **después** de `auth` (en `bootstrap/app.php` o
`Kernel.php`, según convención del proyecto).

Lógica en `handle`:

```php
if (! $request->user()) return $next($request);
if (! $request->isMethod('GET')) return $next($request);

$routeName = $request->route()?->getName();
if (! $routeName) return $next($request);

// No registrar la propia home (evita ruido)
if ($routeName === 'dashboard') return $next($request);

$key = NavigationCatalog::shortcutKeyForRoute($routeName);
if (! $key) return $next($request);

app(NavigationAccessService::class)->recordHit($request->user()->id, $key);

return $next($request);
```

> El middleware corre en **todo** el stack web autenticado, incluyendo rutas del
> portal (`portal.*`) y lab (`lab.*`), cumpliendo el scope acordado en PM.

**Verificación:** visitar `purchase-invoices.index` como admin → fila en
`user_navigation_stats`.

## Notas / Consideraciones

- **Portal-only:** usuarios con solo rol `empleado` siguen yendo a `portal.dashboard`
  vía `CheckSystemAccess`; su navegación en portal **sí se registra** para cuando
  tengan acceso mixto.
- **No incluir** en v1.112.0: reordenar cards dentro de hubs RRHH/Compras, favoritos
  manuales, reset de historial desde UI.
- **Performance:** el middleware hace un `updateOrCreate` por request GET mapeada —
  aceptable; no usar jobs async.
- **Livewire/API:** rutas Livewire internas sin nombre estable no se trackean (OK).
- Reutilizar íconos/descripciones de `PurchaseSectionController`, `SalesSectionController`,
  `LabSectionController`, `RrhhNavigation` — no inventar textos nuevos si ya existen.
