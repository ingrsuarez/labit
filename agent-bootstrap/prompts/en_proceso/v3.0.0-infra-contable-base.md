**VERSIÓN:** v3.0.0

**DEPENDENCIAS:** v2.0.0 (multi-empresa)
**ESTIMACIÓN:** medio día
**PRIORIDAD:** alta

---

## Descripción

Crear la base del módulo de contabilidad para Labit: plan de cuentas argentino estándar
precargado, modelos de asiento contable con partida doble, permisos para el rol contador,
y sección nueva "Contabilidad" en el sidebar admin.

Este módulo es compartido entre empresas (plan de cuentas único) pero los asientos contables
llevan `company_id` para poder ver por empresa o consolidado.

No se generan asientos automáticos en esta versión — solo se construye la infraestructura
y el CRUD del plan de cuentas.

## PASO 1 — Migraciones

Crear 3 migraciones:

### 1.1 — Tabla `accounting_accounts` (Plan de cuentas)

```bash
php artisan make:migration create_accounting_accounts_table
```

Columnas:
- `id` — bigIncrements
- `code` — string(20), unique, index — código contable jerárquico (ej: "1", "1.1", "1.1.01")
- `name` — string(255) — nombre de la cuenta
- `type` — enum: `activo`, `pasivo`, `patrimonio_neto`, `resultado_positivo`, `resultado_negativo`
- `parent_id` — foreignId nullable, self-referencing a `accounting_accounts`
- `level` — tinyInteger — nivel de profundidad (1=rubro, 2=subrubro, 3=cuenta, 4=subcuenta)
- `is_header` — boolean, default false — true = cuenta de agrupación (no recibe imputaciones)
- `is_active` — boolean, default true
- `timestamps`

NO lleva `company_id` — el plan de cuentas es compartido entre ambas empresas.

### 1.2 — Tabla `journal_entries` (Asientos contables)

```bash
php artisan make:migration create_journal_entries_table
```

Columnas:
- `id` — bigIncrements
- `company_id` — foreignId, constrained('companies') — NO nullable, cada asiento pertenece a una empresa
- `date` — date — fecha del asiento
- `number` — unsignedInteger — número secuencial por empresa y año
- `description` — string(500) — descripción del asiento
- `source_type` — string nullable — clase del documento origen (polimórfico, ej: `App\Models\SalesInvoice`)
- `source_id` — unsignedBigInteger nullable — ID del documento origen
- `is_automatic` — boolean, default false — true = generado por el sistema, false = manual
- `created_by` — foreignId nullable, constrained('users')
- `timestamps`

Índices: `[company_id, date]`, `[source_type, source_id]`, `[company_id, number]` unique

### 1.3 — Tabla `journal_entry_lines` (Líneas del asiento)

```bash
php artisan make:migration create_journal_entry_lines_table
```

Columnas:
- `id` — bigIncrements
- `journal_entry_id` — foreignId, constrained, cascadeOnDelete
- `accounting_account_id` — foreignId, constrained('accounting_accounts')
- `debit` — decimal(15,2), default 0
- `credit` — decimal(15,2), default 0
- `description` — string(255) nullable — detalle de la línea
- `timestamps`

**Verificación:** `php artisan migrate` sin errores. Las 3 tablas existen con las columnas correctas.

## PASO 2 — Modelos Eloquent

### 2.1 — `app/Models/AccountingAccount.php`

```php
class AccountingAccount extends Model
{
    protected $fillable = [
        'code', 'name', 'type', 'parent_id', 'level', 'is_header', 'is_active',
    ];

    protected $casts = [
        'is_header' => 'boolean',
        'is_active' => 'boolean',
        'level' => 'integer',
    ];

    // Relaciones
    public function parent() → belongsTo(self::class, 'parent_id')
    public function children() → hasMany(self::class, 'parent_id')->orderBy('code')
    public function journalEntryLines() → hasMany(JournalEntryLine::class)

    // Scopes
    public function scopeActive($q) → where('is_active', true)
    public function scopeImputable($q) → where('is_header', false)
    public function scopeByType($q, $type) → where('type', $type)

    // Helpers
    public function hasMovements(): bool → journalEntryLines()->exists()
    public function getBalanceAttribute(): calcular saldo sumando debit-credit de sus líneas
                                           (para cuentas activo/resultado-: debit - credit;
                                            para pasivo/pn/resultado+: credit - debit)
}
```

### 2.2 — `app/Models/JournalEntry.php`

```php
class JournalEntry extends Model
{
    protected $fillable = [
        'company_id', 'date', 'number', 'description',
        'source_type', 'source_id', 'is_automatic', 'created_by',
    ];

    protected $casts = [
        'date' => 'date',
        'is_automatic' => 'boolean',
    ];

    // Relaciones
    public function company() → belongsTo(Company::class)
    public function lines() → hasMany(JournalEntryLine::class)
    public function source() → morphTo()
    public function creator() → belongsTo(User::class, 'created_by')

    // Validación de partida doble
    public function isBalanced(): bool → abs(lines->sum('debit') - lines->sum('credit')) < 0.01

    // Auto-numeración
    public static function nextNumber(int $companyId, int $year): int
        → self::where('company_id', $companyId)->whereYear('date', $year)->max('number') + 1 ?? 1
}
```

### 2.3 — `app/Models/JournalEntryLine.php`

```php
class JournalEntryLine extends Model
{
    protected $fillable = [
        'journal_entry_id', 'accounting_account_id', 'debit', 'credit', 'description',
    ];

    protected $casts = [
        'debit' => 'decimal:2',
        'credit' => 'decimal:2',
    ];

    // Relaciones
    public function journalEntry() → belongsTo(JournalEntry::class)
    public function account() → belongsTo(AccountingAccount::class, 'accounting_account_id')
}
```

### 2.4 — Relaciones inversas

En `app/Models/Company.php` agregar:
```php
public function journalEntries() → hasMany(JournalEntry::class)
```

## PASO 5 — Controlador y rutas

### 5.1 — `app/Http/Controllers/AccountingAccountController.php`

Crear controlador resource con estos métodos:

**index():**
- `$this->authorize('contabilidad.accounts.index')`
- Cargar todas las cuentas ordenadas por `code`, eager load `children`
- Construir estructura de árbol para la vista (cuentas de nivel 1, con hijos recursivos)
- Pasar a la vista

**create():**
- `$this->authorize('contabilidad.accounts.create')`
- Cargar cuentas header activas para dropdown de padre
- Pasar tipos (activo, pasivo, patrimonio_neto, resultado_positivo, resultado_negativo)

**store(Request $request):**
- `$this->authorize('contabilidad.accounts.create')`
- Validar: code (required, unique en accounting_accounts), name (required, max:255), type (required, in:...), parent_id (nullable, exists), level (required, 1-4), is_header (boolean)
- Si se selecciona parent_id, heredar type del padre automáticamente
- Si se selecciona parent_id, level = parent.level + 1
- Crear y redirect con éxito

**edit($id):**
- `$this->authorize('contabilidad.accounts.edit')`
- Cargar cuenta, cuentas header para dropdown

**update(Request $request, $id):**
- `$this->authorize('contabilidad.accounts.edit')`
- Validar (code unique excepto self)
- No permitir cambiar type si la cuenta tiene movimientos
- Actualizar y redirect

**destroy($id):**
- `$this->authorize('contabilidad.accounts.delete')`
- No permitir eliminar si `hasMovements()` o tiene hijos
- Soft delete (marcar `is_active = false`) en vez de delete real
- Redirect con éxito

## Notas / Consideraciones

- El plan de cuentas es COMPARTIDO entre empresas (sin company_id). Los asientos sí llevan company_id.
- Las cuentas no se eliminan físicamente — se desactivan (is_active = false)
- El seeder es idempotente: usa firstOrCreate por code
- La codificación jerárquica usa puntos: "1" > "1.1" > "1.1.01"
- Los tipos de cuenta determinan la naturaleza del saldo: Activo y Resultado- son deudoras (debit - credit), Pasivo/PN/Resultado+ son acreedoras (credit - debit)
- v3.1.0 va a agregar la generación automática de asientos desde facturas/cobros/pagos
- El contador podrá agregar cuentas al plan si necesita más detalle
