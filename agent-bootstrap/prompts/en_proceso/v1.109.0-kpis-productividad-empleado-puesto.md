## Descripción

Implementar un panel de **productividad diaria por empleado** en RRHH, con métricas según el **rol operativo** (`recepcion-lab`, `tecnico-lab`, `bioquimico`) y el **puesto** del organigrama (`Job` ↔ `Employee` ↔ `User`). Los KPIs son **del día calendario**, filtrables por **sede** (`lab_branch_id`), e incluyen los tres módulos de laboratorio: **clínico**, **veterinario** y **aguas/alimentos**.

**Denominador acordado:** protocolos **creados** en la sede ese día (suma Admission + VetAdmission + Sample).

En la misma versión se cierra la **trazabilidad** que hoy subcuenta a técnicos y recepcionistas:

- `result_entered_by` / `result_entered_at` en clínico y vet
- Auditoría en guardado de planilla de trabajo
- `payment_recorded` para cobros particulares
- **`result_delivered` idempotente** — KPI «Resultados entregados»: **1 por protocolo por empleado por día** (reimprimir 3 veces no suma 3)

Seguir el documento de diseño para layout, columnas, permisos y copy de UI.

## PASO 2 — Helper de auditoría idempotente

Crear `app/Support/LogsProtocolDelivery.php` (o método en trait) con:

```php
public static function logResultDeliveredOncePerDay(Model $protocol, ?int $userId = null): void
```

Lógica:

1. Usuario = `$userId ?? auth()->id()`; si null, return.
2. Verificar que el protocolo tiene al menos una determinación validada (reutilizar helper existente de `sendEmail` en cada controller — ej. `hasValidatedResults()`).

Registrar en `AuditLog::ACTION_LABELS`:

## PASO 4 — `EmployeeProductivityService`

Crear `app/Services/EmployeeProductivityService.php`.

Constructor o método principal:

```php
public function report(Carbon $date, ?int $labBranchId = null, ?int $jobId = null, ?int $employeeId = null): array
```

Retornar estructura:

```php
[
    'branch_summary' => [
        'protocols_created' => int,
        'patients_created' => int, // informativo
        'results_delivered' => int, // sum recepcion
        'validations' => int, // sum bio
    ],
    'rows' => [
        [
            'employee_id' => int,
            'employee_name' => string,
            'job_name' => string,
            'inferred_branch_name' => string,
            'roles' => ['recepcion-lab', ...],
            'metrics' => [ /* claves por rol — ver diseño */ ],
        ],
    ],
]
```

### Reglas de cálculo (día = `$date` startOfDay/endOfDay)

**Protocolos del día (denominador sede):** COUNT protocolos con `created_at` en el día y `lab_branch_id` = sede (o todas las sedes si filtro null).

**Por usuario — recepcionista:**

| Métrica | Query |
|---------|--------|
| `protocols_created` | DISTINCT `audit_logs` `created` donde auditable es Admission/Sample/VetAdmission |
| `patients_created` | `Patient` + audit `created` |
| `protocols_updated` | action `updated` |
| `payments_recorded` | action `payment_recorded` |
| `results_delivered` | action `result_delivered` (ya único) |
| `delivery_rate` | `results_delivered / protocols_created` × 100 (1 decimal; 0 si denominador 0) |

Filtrar audits por sede: join auditable → `lab_branch_id`.

**Por usuario — técnico:**

| Métrica | Query |
|---------|--------|
| `results_entered` | COUNT tests con `result_entered_at` en día y `result_entered_by` = user; vet idem; muestras `analyzed_by` |
| `protocols_with_results` | DISTINCT protocolos con ≥1 resultado ingresado ese día por user |
| `results_loaded_events` | COUNT audit `results_loaded` |
| `tests_removed` | COUNT audit `test_removed` |
| `load_rate` | `protocols_with_results / protocols_created` |

**Por usuario — bioquímico:**

| Métrica | Query |
|---------|--------|
| `tests_validated` | COUNT `validated_at` en día + `validated_by` |
| `protocols_validated` | DISTINCT protocolo con al menos una validación del día por user |
| `unvalidations` | COUNT audit `unvalidated` |
| `emails_sent` | COUNT audit `email_sent` |
| `validation_rate` | `protocols_validated / protocols_created` |
| `results_loaded_events` | COUNT audit `results_loaded` (opcional) |

**Empleados en reporte:** solo `Employee` con `user_id` no null y con **alguna actividad** en el día (o todos los activos con user — preferir solo con actividad para tabla más limpia).

**Sede inferida por fila:** sede con más eventos del usuario ese día; fallback `users.default_lab_branch_id` si existe.

**Excluir:** usuarios de ingesta API (sin employee o user sistema).

**Verificación:** Tinker o test con datos seed — números coherentes con audits manuales.
