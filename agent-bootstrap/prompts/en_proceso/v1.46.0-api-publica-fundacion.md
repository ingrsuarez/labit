# v1.46.0 — Fundación de API pública para integraciones (auth con API key)

**VERSIÓN:** v1.46.0
**SLUG:** api-publica-fundacion
**DEPENDENCIAS:** ninguna (labit ya tiene Sanctum, pero no se usa para esta integración)
**ESTIMACIÓN:** 1 día
**PRIORIDAD:** alta — destraba la integración con LISCOM (v1.47.0–v1.53.0)

---

## Descripción

Crear la infraestructura para que sistemas externos (empezando por LISCOM, el servidor de
interfases HL7 que vive en `c:\wamp64\www\interfases`) puedan consumir la API de labit
autenticándose con una **API key por instancia/sede**.

Esta versión NO expone aún endpoints de negocio (protocolos, resultados). Solo construye:

- Modelo y migración de `ApiClient` (con `lab_branch_id` y `company_id` requeridos).
- Middleware `auth.api_key` que valida el header `X-API-Key`, identifica al cliente,
  registra uso y bloquea si está inactivo.
- Endpoint `GET /api/v1/ping` para verificar conectividad/auth.
- CRUD admin en `/admin/api-clients` para que un Administrador genere, regenere e invalide keys.
- Permiso `api-clients.manage`, auditoría con trait `Auditable` (v2.5.0), logging estructurado.

La key plana se muestra **una sola vez al crearse** (estilo Stripe/GitHub), y solo se persiste
su hash SHA-256. El prefijo es `labit_` para identificar visualmente el origen y facilitar
detección de leaks.

**Topología decidida:** una API key por sede de laboratorio (no global). Esto se aprovechará
en v1.47.0 para filtrar protocolos automáticamente sin que el cliente API tenga que pasar
el `lab_branch_id` en cada request.

## PASO 1 — Migración y modelo `ApiClient`

Crear migración `create_api_clients_table`:

```bash
php artisan make:migration create_api_clients_table --create=api_clients
```

Esquema:

```php
Schema::create('api_clients', function (Blueprint $table) {
    $table->id();
    $table->string('name');                          // "LISCOM Sede Centro"
    $table->string('api_key_hash', 64)->unique();    // sha256 de la key plana
    $table->string('key_preview', 12);               // primeros 8 chars (ej: "labit_AbC1...") para mostrar en listado
    $table->foreignId('lab_branch_id')->constrained();
    $table->foreignId('company_id')->constrained();
    $table->boolean('active')->default(true);
    $table->timestamp('last_used_at')->nullable();
    $table->unsignedBigInteger('requests_count')->default(0);
    $table->text('notes')->nullable();
    $table->foreignId('created_by')->nullable()->constrained('users');
    $table->timestamps();

    $table->index(['active', 'lab_branch_id']);
});
```

Crear modelo `app/Models/ApiClient.php`:
- `$fillable`: `name`, `lab_branch_id`, `company_id`, `active`, `notes`, `created_by`, `key_preview`, `api_key_hash`
- `$hidden`: `api_key_hash`
- `$casts`: `active => bool`, `last_used_at => datetime`
- Relaciones: `labBranch()`, `company()`, `createdBy()`
- Trait `Auditable` (de v2.5.0, ver cómo lo usan otros modelos)
- Método estático `generateKey(): string` → retorna `'labit_' . Str::random(40)`
- Método estático `hashKey(string $plain): string` → `hash('sha256', $plain)`

## PASO 2 — Middleware `auth.api_key`

Crear `app/Http/Middleware/AuthenticateApiKey.php`:

```php
public function handle(Request $request, Closure $next)
{
    $key = $request->header('X-API-Key');

    if (!$key) {
        return response()->json(['error' => 'API key required'], 401);
    }

    $hash = ApiClient::hashKey($key);
    $client = ApiClient::where('api_key_hash', $hash)->first();

    if (!$client || !$client->active) {
        return response()->json(['error' => 'Invalid or inactive API key'], 401);
    }

    // Tracking en background para no penalizar latencia
    dispatch(function () use ($client) {
        $client->increment('requests_count');
        $client->update(['last_used_at' => now()]);
    })->afterResponse();

    $request->merge(['api_client' => $client]);

    return $next($request);
}
```

Registrar el middleware en `bootstrap/app.php` con alias `auth.api_key`.

**Verificación:**
- Sin header X-API-Key → 401
- Header con key inválida → 401
- Header con key de cliente desactivado → 401
- Header válido → request avanza

## PASO 3 — Endpoint `GET /api/v1/ping`

En `routes/api.php`:

```php
Route::prefix('v1')->middleware('auth.api_key')->group(function () {
    Route::get('ping', function (Request $request) {
        $client = $request->get('api_client');
        return response()->json([
            'status' => 'ok',
            'client' => $client->name,
            'branch' => $client->labBranch->name,
            'company' => $client->company->name,
            'time' => now()->toIso8601String(),
        ]);
    });
});
```

**Verificación:**
- `curl -H "X-API-Key: labit_xxx" http://labit.local/api/v1/ping` → 200 con JSON correcto

        Route::post('api-clients/{apiClient}/regenerate', [ApiClientController::class, 'regenerate'])

    });

## Notas / Consideraciones

- **NO usar Sanctum para esta integración.** Sanctum sigue disponible para auth de usuarios humanos (jetstream). La API pública usa el middleware nuevo y no comparte el flujo.
- **El `lab_branch_id` no se puede cambiar después de crear el cliente.** Si una sede cambia de instancia liscom, hay que crear una nueva API key (no reasignar la vieja). Esto evita inconsistencias en logs/auditoría.
- **Tracking en background con `dispatch(...)->afterResponse()`** para no agregar latencia al request. Asume que el queue worker está corriendo. Si el proyecto usa el queue driver `sync`, se ejecuta inline pero sigue siendo después de enviar la respuesta.
- **NO incluir rate limiting en esta versión.** Si LISCOM hace polling agresivo, se gestiona en v1.53.0 (monitoreo) o como hotfix.
- **El prefijo `labit_` es importante para detección de leaks** en logs/git/screenshots. Mantener ese formato.
- La vista de listado debe ordenar por `last_used_at desc` para que las keys activas estén arriba (con nullable al final).
- El permiso es solo `api-clients.manage` (no `view`/`create`/`update`/`delete` separados) — granularidad innecesaria para una feature usada solo por Administradores.
