## Descripción

Al cambiar el valor NBU de una obra social, empresa laboral o veterinaria, las admisiones existentes **no deben modificarse automáticamente**. Opcionalmente, el usuario puede marcar “Actualizar admisiones existentes”, elegir una **fecha desde** y confirmar tras una **vista previa**.

**Diseño UI obligatorio:** leer `docs/designs/DISEÑO_v1.105.0-nbu-inmutabilidad-retroactivo.md` antes de implementar.

## PASO 1 — Servicio `NbuRetroactivePricingService`

Crear `app/Services/NbuRetroactivePricingService.php` con:

### `previewClinical(Insurance $insurance, float $newNbuValue, string $fromDate): array`

- Admisiones: `Admission::where('insurance', $insurance->id)->where('date', '>=', $fromDate)->uninvoiced()`.
- Contar admisiones elegibles, filas `admission_tests` con `price > 0` (o todas las padre facturables), admisiones facturadas excluidas en el rango (para el mensaje).
- Opcional: `estimated_total_delta` sumando diferencia old vs new por fila.

### `applyClinical(Insurance $insurance, float $newNbuValue, string $fromDate): array`

- Por cada admisión elegible (`uninvoiced`), por cada `admission_test` padre (price > 0 o criterio alineado a facturación):
  - Recalcular con lógica equivalente a `AdmissionInsuranceTestPricing::resolve()` pero pasando `$newNbuValue` como multiplicador (extraer helper o parámetro opcional en el servicio de pricing).
  - Actualizar `price`, `nbu_units`, `copago` si aplica.
- `$admission->calculateTotals()` por admisión tocada.
- Retornar `{ admissions_updated, rows_updated, excluded_invoiced }`.

### `previewVet(Customer $customer, float $newRate, string $fromDate): array` / `applyVet(...)`

- Misma idea sobre `VetAdmission` + `vet_admission_tests`.
- Precio: `VetAdmissionController::veterinaryPriceFromNbu($newRate, $nbu)`.
- Actualizar `total_price` de cada vet admission.

**Regla:** Nunca tocar protocolos con `isInvoiced() === true`.

  "admissions_count": 42,
