# v1.35.1 — Remito: lote/vencimiento solo si el insumo controla lote

**VERSIÓN:** v1.35.1  
**SLUG:** remito-aceptar-lote-solo-tracks-lot  
**DEPENDENCIAS:** v1.32.4 (remitos editables / stock), v1.27.0+ (campo `tracks_lot` en insumos)  
**ESTIMACIÓN:** 1–2 h  
**PRIORIDAD:** alta

---

## Problema (reporte de usuario)

Al **aceptar un remito** (pantalla `delivery-notes/show`, formulario “Aceptar Remito”),
el navegador exige completar **Nº de lote** y **fecha de vencimiento** para ítems cuyos
insumos fueron dados de alta **sin** control de lote/vencimiento. El comportamiento esperado:
**solo** los insumos con `tracks_lot = true` deben mostrar esos campos y validarlos.

---

## Contexto técnico (estado actual del repo)

- `resources/views/delivery-notes/show.blade.php`: el bloque “Datos de Lote” solo debería
  renderizar inputs por ítem si `$item->supply && $item->supply->tracks_lot`.
- `DeliveryNoteController::accept()`: reglas `required` solo si `$item->supply->tracks_lot`.
- Si en pantalla aparecen lote/vencimiento con `required` para **todos** los ítems, en la
  práctica eso implica que **`tracks_lot` está en true en BD** para esos insumos, **salvo**
  que exista un bug de relación/caché o valor atípico.

El desarrollador debe **confirmar la causa** (datos vs código) y aplicar las mejoras siguientes.

---

## PASO 0 — Diagnóstico rápido

1. Para 2–3 insumos afectados (ej. agujas, apósitos), en BD o tinker:
   `Supply::where('name', 'like', '%...%')->value('tracks_lot')`.
2. Si `tracks_lot` es `1` / `true`: el sistema se comporta “bien” respecto al dato; el usuario
   debe **desmarcar** “¿Controla lote/vencimiento?” en `supplies/{id}/edit` **o** se corrige
   masivamente con script/SQL acordado.
3. Si `tracks_lot` es `0` / `false` pero igual se muestran campos obligatorios: buscar bug
   (relación `supply` nula y otro código path, caché de vista, duplicado de formulario, etc.).

---

## PASO 1 — Endurecer condiciones (Blade + PHP)

Usar comprobación explícita y null-safe:

- Blade: `@if($item->supply && $item->supply->tracks_lot === true)`  
  (o equivalente que respete el cast booleano de Eloquent).
- `accept()`: solo agregar reglas si  
  `$item->supply && $item->supply->tracks_lot === true`.
- Mismo criterio al leer `lot_number` / `expiration_date` del request.

**Ítems sin insumo** (`supply_id` null o insumo eliminado): no exigir lote; evaluar si se debe
**bloquear** la aceptación del remito con mensaje claro (evitar `$item->supply->tracks_lot` sin nullsafe).

---

## PASO 2 — HTML5 `required` vs validación Laravel

- Mantener `required` en inputs **solo** cuando `tracks_lot === true` (como hoy el Blade).
- Opcional: quitar `required` del HTML y dejar **solo** validación Laravel para evitar tooltips
  nativos duplicados; documentar decisión en comentario breve.

---

## PASO 3 — UX: transparencia en la vista show

En la tabla “Ítems recibidos”, agregar una columna o badge pequeño por fila:

- “Controla lote” si `tracks_lot`
- “Sin lote” en caso contrario

Así el usuario entiende **antes** de aceptar qué ítems van a pedir datos de lote.

---

## PASO 4 — Creación de insumos (modal desde remito + CRUD)

Verificar que al crear insumo desde el remito (AJAX) y desde `supplies/create`:

- `tracks_lot` quede en `false` si el checkbox **no** está marcado.
- No enviar valores fantasma (p. ej. `hidden` con valor `1` junto al checkbox).

Si el modal del remito ya envía bien `tracks_lot: false`, documentar que la corrección de datos
históricos es vía edición de insumo.

---

## PASO 5 — Tests

- Remito pendiente con dos ítems: insumo A `tracks_lot false`, insumo B `tracks_lot true`.
  - Vista show: solo B tiene inputs de lote con `required` (o solo B en bloque Datos de Lote).
  - `accept` sin llenar lote para A: **éxito**; sin llenar para B: **error de validación**.
- Ítem con `supply` null: comportamiento definido (error controlado o skip), sin excepción PHP.
