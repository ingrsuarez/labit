## Descripción

Adapta el módulo de ventas para operar por empresa. Cada factura de venta, presupuesto, nota de crédito, recibo de cobro y punto de venta pertenece a una empresa específica. Todos los listados, formularios y reportes filtran por la empresa activa en sesión.

Los clientes (`customers`) y servicios (`services`) siguen siendo globales — no se les agrega `company_id`.

La integración AFIP (`AfipService`) se adapta para leer la configuración (CUIT, certificados) desde el modelo `Company` de la empresa activa en lugar de desde `config/afip.php`.

## PASO 1 — Migraciones: agregar `company_id`

Crear UNA migración que agregue `company_id` a todas las tablas del módulo de ventas:

```bash
php artisan make:migration add_company_id_to_sales_tables
```

Tablas a modificar:
- `sales_invoices`
- `quotes`
- `collection_receipts`
- `credit_notes`
- `points_of_sale`

Para cada tabla:

```php
$table->foreignId('company_id')->nullable()->after('id')->constrained('companies');
```

> Se usa `nullable()` porque puede haber datos preexistentes sin empresa. Si la BD está vacía, igualmente dejarlo nullable para robustez.

```bash
php artisan migrate
```

**Verificación:** Las 5 tablas tienen la columna `company_id`. No hay errores de migración.

## PASO 2 — Actualizar modelos

Para cada modelo (`SalesInvoice`, `Quote`, `CollectionReceipt`, `CreditNote`, `PointOfSale`):

1. Agregar `'company_id'` al array `$fillable`.
2. Agregar relación:

```php
public function company(): BelongsTo
{
    return $this->belongsTo(Company::class);
}
```

3. En el modelo `Company`, agregar las relaciones inversas:

```php
public function salesInvoices(): HasMany { return $this->hasMany(SalesInvoice::class); }
public function quotes(): HasMany { return $this->hasMany(Quote::class); }
public function collectionReceipts(): HasMany { return $this->hasMany(CollectionReceipt::class); }
public function creditNotes(): HasMany { return $this->hasMany(CreditNote::class); }
public function pointsOfSale(): HasMany { return $this->hasMany(PointOfSale::class); }
```

**Verificación:** Cada modelo tiene la relación `company()` y `company_id` es fillable.

## PASO 3 — Actualizar controladores (filtrar por empresa activa)

### Patrón general a aplicar en cada controlador:

**En `index`:** filtrar la query por empresa activa:

```php
$query->where('company_id', active_company_id());
```

**En `create`/`store`:** asignar la empresa activa al crear:

```php
$data['company_id'] = active_company_id();
```

**En `show`/`edit`/`update`/`destroy`:** verificar que el registro pertenece a la empresa activa:

```php
abort_if($record->company_id !== active_company_id(), 403);
```

### Controladores a modificar:

1. **`SalesInvoiceController`** — index, store, show, edit, update, destroy
2. **`QuoteController`** — index, store, show, edit, update, destroy
3. **`CollectionReceiptController`** — index, store, show, edit, update, destroy
4. **`CreditNoteController`** (si existe) — index, store, show, edit, update, destroy
5. **`PointOfSaleController`** (si existe) — index, store, show, edit, update

### Puntos de venta:

Los puntos de venta son por empresa. Al crear una factura, el dropdown de punto de venta debe filtrar por empresa activa:

```php
$pointsOfSale = PointOfSale::where('company_id', active_company_id())->get();
```

**Verificación:** Los listados solo muestran registros de la empresa activa. Crear un registro lo asocia a la empresa activa.

## PASO 5 — Adaptar AfipService para multi-empresa

Modificar `app/Services/AfipService.php` para que lea la configuración de AFIP desde el modelo `Company` en lugar de `config/afip.php`:

1. El constructor o los métodos que usan CUIT, cert_path, key_path deben recibir o resolver la empresa activa.
2. Patrón sugerido: recibir `Company $company` como parámetro o resolverlo desde `active_company()`.
3. Mantener `config/afip.php` como **fallback** para datos que no estén en la empresa (backward compatible).

```php
$company = active_company();
$cuit = $company?->cuit ?? config('afip.cuit');
$certPath = $company?->afip_cert_path ?? config('afip.cert_path');
$keyPath = $company?->afip_key_path ?? config('afip.key_path');
```

4. Los datos del emisor en el PDF fiscal (`razon_social`, `domicilio`, `condicion_iva`, etc.) deben venir del modelo `Company`.

**Verificación:** AfipService usa la config de la empresa activa. El fallback a config/afip.php funciona.

## Notas / Consideraciones

- Clientes y servicios son GLOBALES — no llevan `company_id`. Un cliente puede tener facturas de ambas empresas.
- El `company_id` es `nullable` en las migraciones por robustez, pero el controlador siempre lo setea al crear.
- Si no hay empresa activa en sesión, los listados podrían estar vacíos — esto es correcto y esperado (el middleware de v2.0.0 se encarga de setear una por defecto).
- Los ítems de factura (`SalesInvoiceItem`, `QuoteItem`, `CollectionReceiptItem`, `CreditNoteItem`) no necesitan `company_id` — se acceden a través de su documento padre.
