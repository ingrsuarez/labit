## Descripción

Implementar en el **módulo Lab clínico** una pantalla de **sincronización con el FTP** de Santa Cruz O&G SRL: listar y descargar archivos `*.xml` nuevos, **previsualizar** paciente y prácticas a importar, resolver **mapeos** de códigos de prestación del cliente hacia `Test` de Labit, e **importar** las admisiones seleccionadas. Tras importación exitosa, **mover** cada archivo XML a la subcarpeta de procesados en el servidor FTP.

**Formato XML (referencia en repo):** `customer_files/Santa Cruz/*.xml` — raíz `<Solicitud>` con paciente (`TipoDocumento`, `NumeroDocumento`, `Apellidos`, `Nombres`, `Sexo`, `FechaNacimiento` `YYYYMMDD`, `Direccion/*`, `Telefono`, `Mail`, `CUIT`, etc.), `<OrdenPrestacion>` con `<AccessionNumber>`, `<Fecha>`, `<Hora>`, `<Practicas>/<Practica>` con `<Subnomenclador>`, `<Prestacion>`, `<Nombre>`.

**Reglas de negocio acordadas (sesión PM 2026-05-13):**

| Tema | Decisión |
|---|---|
| Obra social | **Siempre** la obra social Santa Cruz O&G SRL (resolver `insurance` por ID configurable en `.env`, ej. `SANTA_CRUZ_INSURANCE_ID`, o constante documentada si ya existe en seeders). |
| Número de afiliado | `affiliate_number = 'lcm' . AccessionNumber` (el `AccessionNumber` coincide con el segundo segmento del nombre de archivo `{DNI}_{AccessionNumber}_Examen.xml`). |
| Mapeo prácticas | **(C)** Primero por **código** `<Prestacion>` (normalizado: trim, posible normalización de puntos en códigos tipo `08.05.01`), luego por **nombre** `<Nombre>` (match razonable: exacto insensible a mayúsculas o trim; documentar si se usa `LIKE` limitado). Si no hay match → **sin mapeo**; no se puede importar esa fila hasta resolver. |
| Sin mapeo | **Bloqueo:** no importar el paciente/archivo si queda alguna práctica sin mapeo. En la UI: botón **“Crear mapeo”** → buscador de determinaciones Labit → guardar equivalencia **persistente** para futuros ingresos. |
| Paciente existente (mismo DNI) | **Reutilizar** y **completar solo campos vacíos** del paciente con datos del XML (no pisar valores ya cargados). |
| Paciente nuevo | Crear `Patient` con datos del XML. |
| Post-importación FTP | **Mover** el archivo a subcarpeta de procesados (configurable, ej. `SANTA_CRUZ_FTP_PROCESSED_SUBPATH=procesados` relativo al path base). Crear carpeta remota si no existe cuando sea posible con el driver usado. |
| Ubicación en la app | **(A)** Dentro del lab clínico: ruta tipo `lab/santa-cruz/sync` + entrada en **sidebar** del layout del laboratorio. |

**Fuera de alcance (v1.97.0):**

- Cron / sincronización automática programada (candidato **v1.97.1**).
- Edición manual de datos en la preview antes de importar (solo mapeos).
- Mapeo de `<Sucursal>` a `lab_branch_id` automático (usar sede por defecto del usuario o la misma política que `LabAdmissionController@store` ya use para `lab_branch_id`).
- Suite de tests contra FTP real (mockear cliente FTP en tests).

## PASO 4 — Controlador y rutas

- Rutas bajo middleware de lab y permiso dedicado, ej. `santacruz.import`:
  - `GET  lab/santa-cruz/sync` — página principal (lista última sincronización en sesión o query param `?refresh=1`).
  - `POST lab/santa-cruz/sync/scan` — conectar FTP, parsear todos los XML listados, devolver JSON o redirect con flashes para la tabla de preview (preferencia: **Livewire** solo si ya es estándar en el proyecto para este tipo de flujo; si no, Blade + form POST tradicional).

  - Opcional `GET lab/santa-cruz/mappings` — listado CRUD simple para mantenimiento.
