## Descripción

El sistema Labit actualmente opera como single-tenant. Esta versión introduce el modelo `Company` y toda la infraestructura necesaria para que el sistema soporte múltiples empresas (inicialmente 2: una empresa unipersonal y una sociedad, ambas bajo ley argentina, con CUITs diferentes).

**Reglas de negocio clave:**
- Los módulos de laboratorio NO se ven afectados (nunca tendrán filtro por empresa).
- Clientes y proveedores son **globales** — no llevan `company_id`.
- Cada documento contable (factura, OC, etc.) pertenecerá a una empresa específica (en versiones posteriores).
- Los usuarios contables pueden cambiar de empresa activa. Los de lab tienen una por defecto asignada por el admin.
- El logo es el mismo para todas las empresas (es de la organización, no de cada empresa).
- Las dos empresas comparten organigrama y estructura organizacional.

Esta versión solo crea la infraestructura. Los módulos de ventas, compras y RRHH se adaptan en v2.1.0, v2.2.0 y v2.3.0 respectivamente.

## PASO 4 — Middleware SetActiveCompany

Crear `app/Http/Middleware/SetActiveCompany.php`:

Lógica:
1. Si el usuario está autenticado y no hay `active_company_id` en sesión → setear su empresa default.
2. Compartir con todas las vistas: `$activeCompany` (modelo o null) y `$userCompanies` (colección de empresas del usuario).
3. Si el usuario no tiene empresas asignadas → no setear nada (no romper módulos como lab que no usan empresa).

```php
public function handle(Request $request, Closure $next)
{
    if (auth()->check()) {
        if (!session()->has('active_company_id')) {
            $default = auth()->user()->defaultCompany();
            if ($default) {
                session(['active_company_id' => $default->id]);
            }
        }

        view()->share('activeCompany', active_company());
        view()->share('userCompanies', auth()->user()->companies);
    }

    return $next($request);
}
```

Registrar en `app/Http/Kernel.php` dentro del array `$middlewareGroups['web']`, **después** de `StartSession` y `ShareErrorsFromSession`:

```php
\App\Http\Middleware\SetActiveCompany::class,
```

**Verificación:** Al loguearse, `session('active_company_id')` tiene un valor. Las vistas reciben `$activeCompany` y `$userCompanies`.

## PASO 5 — Ruta y controlador para cambiar empresa activa

Crear `app/Http/Controllers/CompanyController.php` con método `switchCompany`:

```php
public function switchCompany(Request $request)
{
    $request->validate(['company_id' => 'required|exists:companies,id']);

    $user = auth()->user();
    $hasAccess = $user->companies()->where('companies.id', $request->company_id)->exists();

    if (!$hasAccess) {
        abort(403);
    }

    session(['active_company_id' => $request->company_id]);

    return redirect()->back();
}
```

Ruta (dentro del grupo autenticado en `web.php`):

```php
Route::post('/switch-company', [CompanyController::class, 'switchCompany'])
    ->name('company.switch');
```

**Verificación:** POST a `/switch-company` con `company_id` válido cambia la sesión y redirige.

## Notas / Consideraciones

- NO se agrega `company_id` a ningún modelo existente en esta versión — eso se hace en v2.1.0 (ventas), v2.2.0 (compras) y v2.3.0 (RRHH).
- Los módulos de laboratorio NO se tocan y nunca tendrán filtro por empresa.
- Clientes (`customers`) y proveedores (`suppliers`) son GLOBALES — no llevan `company_id`.
- Servicios (`services`) e insumos (`supplies`) son globales.
- El logo es de la organización, no de cada empresa — no hay campo logo en la tabla.
- El selector de empresa solo aparece cuando el usuario tiene >1 empresa asignada.
- Si un usuario no tiene empresa asignada, el middleware no setea nada (para no romper lab).
- El campo `short_name` permite mostrar un nombre corto en el selector del header cuando la razón social es muy larga.
- Los campos AFIP en la tabla `companies` se usarán cuando se adapte el AfipService en v2.1.0.
