# v1.76.1 — API key global (sin sede) para LISCOM

**VERSIÓN:** v1.76.1  
**SLUG:** api-key-global-sin-sede  
**DEPENDENCIAS:**
  - Internas (labit): ninguna
  - Externas (LISCOM): ninguna (LISCOM no cambia; cuando labit responda 200 el sync on-demand funcionará solo)
**ESTIMACIÓN:** 30–45 min  
**PRIORIDAD:** alta (bloquea producción — mensajes HL7 quedan ORPHAN en LISCOM)

---

## Descripción

El laboratorio tiene una sola instancia LISCOM que recibe mensajes HL7 de equipos de **todas las
sedes**. Al hacer sync on-demand de un protocolo, LISCOM llama a `GET /api/v1/protocols/by-barcode/{code}`
usando la API key configurada.

El problema: `ProtocolLookupService` filtra **siempre** por `lab_branch_id` del `ApiClient`.
Si el protocolo escaneado pertenece a una sede distinta a la de la key, labit devuelve 404,
LISCOM no puede cachear el protocolo y el `ResultMessage` queda marcado como **ORPHAN** en la bandeja.

**Ejemplo real:** protocolo `C2605070011` (Primet, otra sede) llegó al equipo conectado al LISCOM
de "Central". Labit devolvió 404 → ORPHAN. El operador no puede aprobar ese resultado.

**Solución:** hacer `lab_branch_id` nullable en `ApiClient`. Cuando es `null`, el
`ProtocolLookupService` omite el filtro de sede y devuelve protocolos de cualquier sede.
Las keys existentes con sede asignada no cambian de comportamiento.

## PASO 3 — ProtocolLookupService: omitir filtro de sede si la key es global

En `app/Services/Api/ProtocolLookupService.php`, modificar `findByBarcode` y `query`:

### `findByBarcode`

```php
public function findByBarcode(string $code, ApiClient $client): ?Model
{
    $type = ProtocolType::fromBarcodePrefix($code);

    if ($type) {
        $modelClass = $type->modelClass();
        $query = $modelClass::query()->where('protocol_number', $code);
        if (! $client->isGlobal()) {
            $query->where('lab_branch_id', $client->lab_branch_id);
        }
        $found = $query->first();
        if ($found) {
            return $found;
        }
    }

    foreach (ProtocolType::cases() as $candidate) {
        if ($type && $candidate === $type) {
            continue;
        }
        $modelClass = $candidate->modelClass();
        $query = $modelClass::query()->where('protocol_number', $code);
        if (! $client->isGlobal()) {
            $query->where('lab_branch_id', $client->lab_branch_id);
        }
        $found = $query->first();
        if ($found) {
            return $found;
        }
    }

    return null;
}
```

### `query` (sync masivo)

En el método `query`, dentro del `foreach ($types as $type)`:

```php
$query = $modelClass::query()
    ->with($this->eagerLoadFor($type));

if (! $client->isGlobal()) {
    $query->where('lab_branch_id', $client->lab_branch_id);
}
```

Quitar la línea `->where('lab_branch_id', $client->lab_branch_id)` que estaba hardcodeada dentro.

**Verificación:** Una key con `lab_branch_id = null` debe devolver protocolos de todas las sedes. Una key con sede asignada debe seguir filtrando igual que antes.

## PASO 4 — Controlador ProtocolController: ajustar `show` por id

En `app/Http/Controllers/Api/V1/ProtocolController.php`, el método `show` (detalle por tipo + id)
también tiene el filtro hardcodeado:

```php
$found = $modelClass::query()
    ->where('id', $id)
    ->where('lab_branch_id', $client->lab_branch_id)
    ->first();
```

Reemplazar por:

```php
$query = $modelClass::query()->where('id', $id);
if (! $client->isGlobal()) {
    $query->where('lab_branch_id', $client->lab_branch_id);
}
$found = $query->first();
```

**Verificación:** el endpoint `GET /api/v1/protocols/{type}/{id}` responde 200 para protocolos
de cualquier sede cuando la key es global.

## Notas / Consideraciones

- **No tocar `ApiResultIngestionService`**: ese servicio compara `lab_branch_id` del protocolo contra el de la key al ingestar resultados POST. Esa lógica es independiente y correcta — no forma parte de este hotfix.
- **Retrocompatibilidad garantizada**: todas las API keys existentes tienen `lab_branch_id` asignado → `isGlobal()` devuelve `false` → comportamiento idéntico al actual.
- **En producción:** después de deployar, editar la `ApiClient` que usa LISCOM y vaciar su `lab_branch_id`. LISCOM no necesita reinicio; el próximo sync on-demand (o clic en "Reprocesar después de sync") va a funcionar.
- **Protocolo C2605070011 en LISCOM:** después del deploy, ejecutar en el servidor LISCOM:
  ```bash
  python manage.py shell -c "
  from integrations.sync_service import ProtocolSyncService
  result = ProtocolSyncService().sync_on_demand('C2605070011')
  print('Resultado:', result)
  "
  ```
  Si devuelve el protocolo, el `ResultMessage` id 148 se puede reprocesar desde la UI.
