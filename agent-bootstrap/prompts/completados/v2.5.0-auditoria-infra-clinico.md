## Descripción

Crea el sistema de auditoría del proyecto con un log por registro (historial de cambios visible en cada entidad) y lo aplica a autenticación, pacientes, admisiones/protocolos de lab clínico y muestras de aguas/alimentos. Nivel básico: registra quién hizo qué acción, cuándo y desde dónde, sin detalle campo-a-campo.

## PASO 3 — Trait `Auditable`

Crear `app/Traits/Auditable.php`:

```php
namespace App\Traits;

use App\Models\AuditLog;

trait Auditable
{
    public function auditLogs()
    {
        return $this->morphMany(AuditLog::class, 'auditable')->latest();
    }

    public function logAudit(string $action, string $description, ?int $userId = null): AuditLog
    {
        $user = $userId ? \App\Models\User::find($userId) : auth()->user();

        return $this->auditLogs()->create([
            'user_id'    => $user?->id,
            'user_name'  => $user?->name ?? 'Sistema',
            'action'     => $action,
            'description'=> $description,
            'ip_address' => request()?->ip(),
        ]);
    }
}
```

**Uso desde cualquier controlador:**
```php
$patient->logAudit('created', 'Creó el paciente ' . $patient->full_name);
```

**Verificación:** Un modelo con el trait puede llamar `logAudit()` y el registro aparece en `audit_logs`.

## PASO 5 — Auth logging (login / logout / failed)

### 5.1 — Listener para eventos de auth

Laravel dispara eventos `Login`, `Logout`, `Failed` automáticamente. Crear listeners o usar un subscriber.

Crear `app/Listeners/AuthAuditSubscriber.php`:

```php
namespace App\Listeners;

use App\Models\AuditLog;
use Illuminate\Auth\Events\Login;
use Illuminate\Auth\Events\Logout;
use Illuminate\Auth\Events\Failed;

class AuthAuditSubscriber
{
    public function handleLogin(Login $event): void
    {
        AuditLog::create([
            'user_id'     => $event->user->id,
            'user_name'   => $event->user->name,
            'action'      => 'login',
            'description' => 'Inició sesión',
            'ip_address'  => request()?->ip(),
        ]);
    }

    public function handleLogout(Logout $event): void
    {
        if (!$event->user) return;

        AuditLog::create([
            'user_id'     => $event->user->id,
            'user_name'   => $event->user->name,
            'action'      => 'logout',
            'description' => 'Cerró sesión',
            'ip_address'  => request()?->ip(),
        ]);
    }

    public function handleFailed(Failed $event): void
    {
        AuditLog::create([
            'user_id'     => null,
            'user_name'   => $event->credentials['email'] ?? 'desconocido',
            'action'      => 'login_failed',
            'description' => 'Intento de login fallido con email: ' . ($event->credentials['email'] ?? '?'),
            'ip_address'  => request()?->ip(),
        ]);
    }

    public function subscribe($events): array
    {
        return [
            Login::class  => 'handleLogin',
            Logout::class => 'handleLogout',
            Failed::class => 'handleFailed',
        ];
    }
}
```

### 5.2 — Registrar el subscriber

En `EventServiceProvider` (o `AppServiceProvider` si se usa el approach de Laravel 11):

```php
protected $subscribe = [
    \App\Listeners\AuthAuditSubscriber::class,
];
```

> Investigar cómo están registrados los eventos en este proyecto (EventServiceProvider vs providers). Adaptar según lo que exista.

**Verificación:** Login genera un registro en `audit_logs` con action=`login`. Logout genera `logout`. Un login con password incorrecto genera `login_failed`.

## Notas / Consideraciones

- Nivel básico: solo descripción textual de la acción, sin diff campo-a-campo. Si en el futuro se necesita detalle, se pueden agregar columnas `old_values` / `new_values` (JSON) sin romper lo existente.
- `user_name` se guarda como snapshot para que el log sea legible incluso si el usuario se elimina.
- Los logs de auth (login/logout/failed) NO tienen `auditable` (no están asociados a un registro específico). Se consultan filtrando por `user_id` y `action`.
- El componente `<x-audit-history>` es reutilizable — en v2.6.0 se agrega a facturas, compras, liquidaciones, etc. sin cambiar la infraestructura.
- No se incluye paginación en el historial por ahora. Si un registro acumula muchos logs, se puede agregar paginación lazy-load en una versión futura.
- Los nombres de métodos en los controladores (saveResults, validate, etc.) deben investigarse al momento de implementar — este prompt indica la intención, el dev debe adaptar al código real.
