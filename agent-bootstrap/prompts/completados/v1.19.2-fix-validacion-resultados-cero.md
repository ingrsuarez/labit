# v1.19.2 — Fix validación y PDF para resultados con valor cero

**VERSIÓN:** v1.19.2  
**SLUG:** fix-validacion-resultados-cero  
**DEPENDENCIAS:** ninguna  
**ESTIMACIÓN:** 1h  
**PRIORIDAD:** alta (bug en producción)

---

## Descripción

Cuando una determinación del laboratorio clínico tiene resultado `0` (cero) — por ejemplo,
Basófilos = 0 — el sistema lo trata como "sin resultado". Esto causa:

1. **No se puede validar** la determinación (ni individual ni con "Validar todos")
2. **No aparece el botón de validar** (muestra "-" en vez de "✓")
3. **El estado muestra "Pendiente"** en vez de "Con resultado"
4. **No aparece en el informe PDF** (se omite la fila o el valor)

La causa raíz es el uso de checks falsy de PHP (`!$value`, `if($value)`, `empty()`) sobre
el campo `result`, donde `0` y `"0"` son evaluados como `false` en PHP.

El fix debe cambiar todos estos checks por comparaciones estrictas contra `null` y `''`.

## PASO 1 — Fix `LabAdmissionController::validateTest()`

**Archivo:** `app/Http/Controllers/LabAdmissionController.php`

Buscar en el método `validateTest` la condición:
```php
if (! $admissionTest->result) {
    return redirect()->back()->with('error', 'No se puede validar una práctica sin resultado.');
}
```

**Reemplazar por:**
```php
if ($admissionTest->result === null || $admissionTest->result === '') {
    return redirect()->back()->with('error', 'No se puede validar una práctica sin resultado.');
}
```

**Por qué:** `! $admissionTest->result` evalúa `"0"` y `0` como true (PHP falsy). La comparación estricta contra `null` y `''` permite que `0` pase como resultado válido.

**Verificación:** Cargar resultado 0 en una determinación → el botón validar debe funcionar.

## PASO 2 — Fix `LabAdmissionController::validateAll()`

**Archivo:** `app/Http/Controllers/LabAdmissionController.php`

Buscar en el método `validateAll` la condición del foreach:
```php
if ($admissionTest->result && ! $admissionTest->is_validated) {
```

**Reemplazar por:**
```php
if ($admissionTest->result !== null && $admissionTest->result !== '' && ! $admissionTest->is_validated) {
```

**Por qué:** `$admissionTest->result && ...` excluye resultados `0`/`"0"` del bulk validate.

**Verificación:** Cargar resultado 0 en una det + "Validar todos" → la determinación con 0 debe quedar validada.

## PASO 5 — Revisar PDF de admisiones clínicas

**Archivo:** `resources/views/lab/admissions/pdf-mpdf.blade.php`

Revisar todos los usos de `$at->result` en el template. El patrón `{{ $at->result ?? '-' }}` (null coalescing) **es correcto** para `0` — no hace falta cambio.

**Sin embargo**, revisar si hay algún bloque condicional que use `@if($at->result)` o similar para decidir si renderizar la fila. Si existe, cambiar a `$at->result !== null && $at->result !== ''`.

También revisar el uso de `?:` (Elvis operator) en valores de referencia:
```php
$at->reference_value ?: ($at->test->other_reference ?? '')
```

Si `reference_value` pudiera ser `"0"`, el Elvis lo trataría como falsy. Cambiar a:
```php
($at->reference_value !== null && $at->reference_value !== '') ? $at->reference_value : ($at->test->other_reference ?? '')
```

**Verificación:** Generar PDF de un protocolo con basófilos = 0 → debe aparecer "0" en la columna resultado.

## Notas / Consideraciones

- La causa raíz es **semántica PHP**: `empty("0")` es `true`, `!"0"` es `true`, `"0" == false` es `true`.
  La solución canónica es **siempre** usar `=== null || === ''` para campos que pueden contener `0` como valor válido.
- El campo `result` en la BD es `varchar` (string), por lo que PHP lo recibe como `"0"` (string).
  Tanto `"0"` como `0` son falsy en PHP.
- No se modifica la lógica de guardado (`saveResults`) porque usa `??` (null coalescing) que es correcto para `0`.
- El módulo de muestras probablemente no tenga este bug en validación (usa `status` en vez de `result`),
  pero se revisa preventivamente el PDF y la lógica de cumplimiento.
