# v1.108.0 — Hotfix: protocolos clínicos duplicados (race condition)

**VERSIÓN:** v1.108.0  
**SLUG:** hotfix-protocolos-clinicos-duplicados  
**DEPENDENCIAS:** ninguna  
**ESTIMACIÓN:** 2–3 h  
**PRIORIDAD:** 🔴 urgente (incidente producción 2026-05-28 — duplicado `C2605280083` ya corregido manualmente en prod; este hotfix evita recurrencia)

---

## Descripción

Al crear admisiones clínicas casi en simultáneo, dos requests pueden leer el mismo último
`protocol_number` del día y persistir el mismo código (ej. `C2605280083`). Impacto: confusión
operativa, etiquetas/muestras cruzadas, equipos y LISCOM.

**Causa raíz:**

1. `GeneratesProtocolNumber::generatePrefixedProtocolNumber()` hace read-then-increment **sin
   transacción ni lock** (`app/Traits/GeneratesProtocolNumber.php`).
2. La tabla `admissions` tiene **índice** en `protocol_number` pero **no UNIQUE** (a diferencia
   de `samples` y `vet_admissions`, donde el segundo insert fallaría).

**Objetivo:** Garantizar unicidad en DB + generación segura bajo concurrencia. **No** renumerar
datos en producción (el duplicado ya fue resuelto manualmente).

## PASO 1 — Migración: UNIQUE en `admissions.protocol_number`

Crear migración `add_unique_protocol_number_to_admissions_table.php`:

1. Antes de agregar el unique, verificar que no queden duplicados en la DB local/staging:

```sql
SELECT protocol_number, COUNT(*) AS c
FROM admissions
WHERE protocol_number IS NOT NULL
GROUP BY protocol_number
HAVING c > 1;
```

Si hay duplicados → **abortar migración** con mensaje claro (no auto-renumerar; en prod ya está limpio).

2. Agregar constraint unique:

```php
$table->unique('protocol_number');
```

3. El índice simple previo (`index('protocol_number')`) queda cubierto por el unique — dropear
   el índice redundante si la migración lo permite sin conflicto de nombres.

**Nota:** `protocol_number` nullable — en MySQL múltiples NULL siguen permitidos; OK.

## PASO 2 — Generación atómica en el trait

Refactorizar `app/Traits/GeneratesProtocolNumber.php`:

1. Envolver la lectura del último registro en `DB::transaction()`.
2. Usar `lockForUpdate()` al consultar el último `protocol_number` del día:

```php
$lastRecord = static::where($column, 'like', $todayPrefix.'%')
    ->orderBy($column, 'desc')
    ->lockForUpdate()
    ->first();
```

3. Documentar en docblock que **debe invocarse dentro de la misma transacción del `create`**
   del protocolo para máxima seguridad; el UNIQUE + retry (PASO 3) cubre el edge case del
   primer protocolo del día (sin filas que bloquear).

4. Aplicar el mismo trait a los 3 módulos (C/A/V) — vet y muestras ya tienen UNIQUE pero
   se benefician del lock para evitar errores 1062 al usuario.

**No cambiar** el formato `[PREFIJO][AAMMDD][NNNN]`.

## PASO 3 — Retry en puntos de creación (clínico)

### 3.1 — `LabAdmissionController::store`

Envolver la creación de la admisión (al menos `Admission::create` + lógica inicial de tests
en la misma transacción) con retry ante violación de unique:

- Máximo **5 intentos**.
- Capturar `Illuminate\Database\QueryException` / `UniqueConstraintViolationException` solo
  cuando el mensaje/código indique duplicate key en `protocol_number`.
- En cada reintento, regenerar `protocol_number` (no reutilizar el colisionado).
- Si agotan intentos → relanzar excepción con log en canal `single` o `daily` indicando
  fallo de generación de protocolo.

Patrón sugerido (adaptar al código existente sin inflar el método):

```php
DB::transaction(function () use (...) {
    $admission = Admission::create([
        'protocol_number' => Admission::generateProtocolNumber(),
        // ...
    ]);
    // ... admission tests dentro de la misma transacción si es posible
});
```

### 3.2 — `SantaCruzImportService`

Mismo patrón de retry al importar admisiones (misma causa raíz).

### 3.3 — Vet y muestras (opcional, recomendado)

Si `VetAdmissionController::store` y `SampleController::store` generan protocolo fuera de
transacción, aplicar el mismo retry por paridad — bajo riesgo, evita 500 silenciosos en
concurrencia.
