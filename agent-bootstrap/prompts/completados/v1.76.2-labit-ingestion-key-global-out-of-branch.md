# v1.76.2 — Fix ingesta: key global omite validación PROTOCOL_OUT_OF_BRANCH

**VERSIÓN:** v1.76.2  
**SLUG:** ingestion-key-global-out-of-branch  
**DEPENDENCIAS:**
  - Internas (labit): v1.76.1 (API key global sin sede — debe estar deployada)
  - Externas (LISCOM): ninguna
**ESTIMACIÓN:** 15 min  
**PRIORIDAD:** alta (bloquea envío de resultados a labit desde LISCOM — dispatches en dead-letter)

---

## Descripción

En v1.76.1 se hizo `lab_branch_id` nullable en `ApiClient` para que una key global
pueda consultar protocolos de cualquier sede (GET). Pero el endpoint POST de ingesta de
resultados (`ApiResultIngestionService::processItem`) tiene su propio chequeo de sede:

```php
if (isset($protocol['model']->lab_branch_id)
    && $protocol['model']->lab_branch_id !== $client->lab_branch_id
) {
    return $this->rejectMessage(..., self::REASON_PROTOCOL_OUT_OF_BRANCH);
}
```

Cuando `$client->lab_branch_id === null` (key global), la comparación es
`5 !== null` → `true` → rechaza con `PROTOCOL_OUT_OF_BRANCH`. LISCOM recibe ese
rechazo y el dispatch queda en dead-letter.

**Fix:** si la key es global (`lab_branch_id === null`), omitir la validación de sede.
Una línea de cambio.

## PASO 1 — Fix en ApiResultIngestionService

En `app/Services/Api/ApiResultIngestionService.php`, método `processItem`,
reemplazar el bloque de validación de sede:

**Antes (líneas ~128–132):**

```php
if (isset($protocol['model']->lab_branch_id)
    && $protocol['model']->lab_branch_id !== $client->lab_branch_id
) {
    return $this->rejectMessage($client, $batch, $item, self::REASON_PROTOCOL_OUT_OF_BRANCH);
}
```

**Después:**

```php
if (! $client->isGlobal()
    && isset($protocol['model']->lab_branch_id)
    && $protocol['model']->lab_branch_id !== $client->lab_branch_id
) {
    return $this->rejectMessage($client, $batch, $item, self::REASON_PROTOCOL_OUT_OF_BRANCH);
}
```

El cambio es agregar `! $client->isGlobal() &&` al inicio de la condición.
Cuando la key es global, el bloque entero se saltea y el resultado se ingesta normalmente.

**Verificación:** el método queda legible y la condición tiene sentido semántico:
"rechazar por sede solo si la key NO es global Y el protocolo tiene sede Y no coincide".

  "Re-encolar", o por shell:

  ```bash
      status='dead_letter',

      rejection_reason='',
      locked_by='',
      send_locked_until=None,

  ```
