## Descripción

Crear un dashboard operativo como primera vista al entrar al laboratorio. Muestra KPIs,
gráficos de barras por estado de protocolos, por tipo de laboratorio, por sede, y una
alerta de protocolos atrasados. El objetivo es dar visibilidad para que ningún protocolo
quede pendiente de validar o enviar.

El dashboard reemplaza la vista hub actual de "Laboratorio Clínico" (`/lab/clinico`) que
hoy muestra solo tarjetas de navegación. Las tarjetas de acceso rápido se conservan debajo
de los gráficos. Los roles de laboratorio (`recepcion-lab`, `tecnico-lab`, `bioquimico`)
son redirigidos a esta vista desde `DashboardController`.

Patrón visual: barras con `<div>` + Tailwind + Alpine.js (mismo patrón que v1.66.0,
sin librerías de charts externas). Tooltips con Alpine.js al hover.

Período por defecto: últimos 30 días para KPIs de actividad. Los conteos de pendientes
son acumulados vivos (sin filtro de período).

## PASO 1 — Crear LabDashboardService

Crear `app/Services/LabDashboardService.php` con los siguientes métodos:

### Constructor
Recibe `?int $labBranchId` (sede a filtrar, null = todas).

### `kpis(int $days = 30): array`
Retorna 4 valores:
- `created` — protocolos creados en los últimos `$days` días (suma de Admission + VetAdmission + Sample)
- `pending_validation` — protocolos con determinaciones pendientes de validar (acumulado vivo, sin filtro de fecha). Para Admission: `status IN ('pending','in_progress','completed')`. Para VetAdmission: `status IN ('pending','in_progress','completed')`. Para Sample: `validation_status != 'validated'` AND `status != 'cancelled'`.
- `validated` — protocolos validados en los últimos `$days` días. Para Admission: `status = 'validated'` AND updated_at en rango. Para VetAdmission: `status = 'validated'` AND updated_at en rango. Para Sample: `validation_status = 'validated'` AND validated_at en rango.
- `sent` — protocolos enviados en los últimos `$days` días (los 3 modelos: `sent_at` en rango).

Cada query filtra por `lab_branch_id` si se proporcionó.

### `byStatus(): array`
Retorna conteos agrupados por estado para los 3 tipos sumados:
```php
[
    'pending'    => N,
    'in_progress' => N,
    'completed'  => N,
    'validated'  => N,
    'sent'       => N,
]
```
Los "enviados" se calculan aparte: son protocolos con `sent_at IS NOT NULL`. Los estados se obtienen del campo `status` (Admission, VetAdmission) y `calculated_status` derivado para Sample.

**Importante para Admission:** El campo `status` se sincroniza externamente (v1.69.0). Si `status` no está sincronizado para todos los registros, usar la misma lógica que `computeStatus()` para contar. Pero preferir queries directas contra `admissions.status` porque es más performante. Si `status` es null o vacío en registros legacy, tratarlos como `pending`.

Filtrar por `lab_branch_id`. No filtrar por fecha (esto es el estado vivo actual).

### `byLabType(): array`
Retorna conteos de protocolos pendientes de validar, desglosados:
```php
[
    'clinico'     => N,  // Admission donde status != 'validated' AND status != 'cancelled'
    'veterinario' => N,  // VetAdmission donde status != 'validated' AND status != 'cancelled'
    'aguas'       => N,  // Sample donde validation_status != 'validated' AND status != 'cancelled'
]
```
Filtrar por `lab_branch_id`.

### `byBranch(): array`
Retorna conteos de protocolos pendientes de validar por sede:
```php
[
    ['id' => 1, 'name' => 'Sede Central', 'pending' => N],
    ['id' => 2, 'name' => 'Sede Norte',   'pending' => N],
    ...
]
```
Consulta las 3 tablas, agrupa por `lab_branch_id`, JOIN con `lab_branches` para el nombre.
Solo sedes activas (`is_active = true`). NO filtrar por `$labBranchId` (este gráfico siempre muestra todas).

### `overdue(int $days = 3): array`
Retorna:
```php
[
    'count' => N,   // total de protocolos pendientes > $days días
    'oldest_date' => '2026-05-01',  // fecha del más antiguo
]
```
Protocolos con `status != 'validated' AND status != 'cancelled'` (o equivalente para Sample)
AND `created_at < now() - $days days`. Filtrar por `lab_branch_id`.

**Verificación:** El service puede instanciarse. Todos los métodos retornan arrays con la estructura correcta. No hay N+1 — todas son queries directas con COUNT/GROUP BY, sin cargar modelos.

## Notas / Consideraciones

- Los estados de Admission usan el campo `status` (sincronizado por v1.69.0). Si hay admisiones legacy sin `status` sincronizado, tratarlas como `pending`.
- Sample no tiene campo `status` directo para validación; usar `validation_status` para saber si está validado y el accessor `calculated_status` como referencia de la lógica.
- El gráfico "por sede" se oculta automáticamente si solo hay una sede activa.
- No usar Livewire ni auto-refresh en esta versión. Si el usuario quiere datos actualizados, recarga la página (F5). Se puede agregar auto-refresh con Livewire en una versión futura.
- Patrón de barras: usar `max()` del array para calcular altura relativa; si el máximo es 0, no renderizar barras (mostrar estado vacío con mensaje "Sin protocolos en el período").
- Para el gráfico de "Estado de protocolos", la categoría "Enviado" se cuenta independientemente del estado (un protocolo validado Y enviado aparece en ambas barras). Si se prefiere mutuamente excluyente (enviado saca de validado), ajustar la query.
