# v1.59.0 — Prellenar lote/vencimiento en FC al cargar desde remito

**VERSIÓN:** v1.59.0
**SLUG:** prellenar-lote-vencimiento-fc-desde-remito
**DEPENDENCIAS:** v1.56.0 (botón Crear FC desde remito)
**ESTIMACIÓN:** 20min
**PRIORIDAD:** alta

---

## Descripción

Al abrir "Nueva Factura de Compra" desde un remito (via `?delivery_note_id=X` o botón
"Crear Factura de Compra"), los ítems de la grilla aparecen con los campos Lote y
Vencimiento vacíos, aunque el remito sí tenía esos datos cargados.

**Root cause (dos partes, ambas en `create.blade.php`):**

1. **PHP (línea ~548):** el mapeo `$deliveryNoteItemsJson` no incluye `lot_number`
   ni `expiration_date` — los omite al construir el JSON inicial.
2. **JS (línea ~726):** el spread del map inicial los pisa con `''` explícitamente:
   ```js
   deliveryNoteItems.map(i => ({...i, lot_number: '', expiration_date: '', ...}))
   ```

**Nota:** cuando el usuario agrega un remito dinámicamente via el selector
("Agregar remito"), `onSelectDeliveryNoteToAdd()` ya funciona bien — llama al
endpoint `getItems` y el spread preserva los valores. No tocar esa ruta.

**Nota:** `edit.blade.php` carga ítems ya guardados en BD (con lot/exp intactos)
— no tiene el mismo problema. No modificar.

## PASO 1 — Agregar lot_number y expiration_date al mapeo PHP

Archivo: `resources/views/purchase-invoices/create.blade.php`

Localizar el bloque `@php` que construye `$deliveryNoteItemsJson` (línea ~548).

**ANTES:**
```php
$deliveryNoteItemsJson = $deliveryNote
    ? $deliveryNote->items->map(function ($i) {
        return [
            'description' => $i->supply ? $i->supply->name : ('Ítem remito - Cant: ' . $i->quantity_received),
            'supply_id' => $i->supply_id,
            'purchase_service_id' => '',
            'quantity' => floatval($i->quantity_received),
            'unit_price' => $i->supply ? floatval($i->supply->last_price ?? 0) : 0,
            'iva_rate' => '21',
        ];
    })->values()
    : collect();
```

**DESPUÉS:**
```php
$deliveryNoteItemsJson = $deliveryNote
    ? $deliveryNote->items->map(function ($i) {
        return [
            'description' => $i->supply ? $i->supply->name : ('Ítem remito - Cant: ' . $i->quantity_received),
            'supply_id' => $i->supply_id,
            'purchase_service_id' => '',
            'quantity' => floatval($i->quantity_received),
            'unit_price' => $i->supply ? floatval($i->supply->last_price ?? 0) : 0,
            'iva_rate' => '21',
            'lot_number' => $i->lot_number ?? '',
            'expiration_date' => $i->expiration_date?->format('Y-m-d') ?? '',
        ];
    })->values()
    : collect();
```

## PASO 2 — Quitar el override de lot_number y expiration_date en el JS

Archivo: `resources/views/purchase-invoices/create.blade.php`

Localizar la línea ~726 en la función `invoiceForm()`:

**ANTES:**
```js
items: deliveryNoteItems.length > 0 ? [...deliveryNoteItems.map(i => ({...i, lot_number: '', expiration_date: '', _supply_code: '', updates_stock: false}))] : [],
```

**DESPUÉS:**
```js
items: deliveryNoteItems.length > 0 ? [...deliveryNoteItems.map(i => ({...i, _supply_code: '', updates_stock: false}))] : [],
```

**Verificación PASO 2:**
- Abrir FC desde un remito con ítem que tiene lote "L-2026-001" y vencimiento "2026-12-31"
- Los campos Lote y Vencimiento de esa fila aparecen prellenados
- Ítem sin lote/vencimiento (supply que no trackea lote) → los inputs no se muestran
  (la lógica `getSupplyTracksLot(item.supply_id)` ya maneja esto — sin cambios)
