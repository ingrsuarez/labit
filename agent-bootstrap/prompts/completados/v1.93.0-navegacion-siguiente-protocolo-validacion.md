## Descripción

Varios bioquímicos validan y firman **simultáneamente**. Hoy, tras trabajar un protocolo, hay que **volver al listado**, **reaplicar filtros** y ubicar el próximo pendiente. Se requiere un flujo **“Siguiente”** (y **“Anterior”**, añadido en esta entrega) en **laboratorio clínico** (`Admission`), **veterinario** (`VetAdmission`) y **muestras** (`Sample`) que:

1. **Respeta los mismos filtros** que el listado desde el que el usuario entró (criterio **A** acordado en PM).
2. Ejecuta una consulta **viva** en cada paso (reaplica filtros + reglas de elegibilidad contra la BD) para convivir con validaciones concurrentes.
3. El candidato a “siguiente” / “anterior” debe ser un protocolo **no validado** y **no enviado**:
   - **Clínico:** `status !== validated` (misma noción que “aún no validado” del dominio) **y** `sent_at` es `null`. Alinear con `Admission` y listado `LabAdmissionController@index` (incl. filtro especial `status=enviado` si aplica solo a listados, no a elegibles de siguiente).
   - **Veterinario:** `status !== 'validated'` (y no cancelado si el listado lo excluye) **y** `sent_at` es `null`.
   - **Muestras:** `validation_status !== 'validated'` **y** `sent_at` es `null` (coherente con `Sample` y estado “enviado”).
4. **Orden de avance (“sentido del tiempo”):** entre los registros que cumplen (1)+(3), el siguiente es el de **menor `protocol_number` estrictamente mayor** al del protocolo actual, con **desempate** `id` ascendente si hiciera falta. **Anterior:** simétrico con **`protocol_number` estrictamente menor**, orden `desc` por número + `id`. Ejemplo esperado: desde `C2605120001` → `C2605120002` si `0002` sigue sin validar y sin enviar; si `0002` ya no califica, saltar a `0003`, etc. **Nota:** el índice clínico/vet ordena por `date desc` / `id desc`; eso **no** define el orden de “Siguiente”: el orden de navegación es explícitamente **ascendente por `protocol_number`** (y `id`) entre el subconjunto filtrado y elegible.
5. Si **no hay** candidato siguiente o anterior: respuesta clara (flash) + redirect al **listado con los mismos query params** de filtros donde aplica.

**Fuera de alcance explícito (sin cambio):**

- Cambiar reglas de negocio de validación o permisos (solo consumir las existentes).
- Documento **AGENTE_DESIGNER** separado: reutilizar patrones de cabecera/acciones del lab; botones discretos alineados al layout existente.

## PASO 1 — Estrategia de contexto de filtros (segura y reproducible)

1. Extraer la construcción del query base de cada `index` a **métodos privados o traits** reutilizables, p. ej. `applyAdmissionIndexFilters(Request $request, Builder $query): void`, equivalente para vet y sample, sin cambiar el comportamiento del listado actual (tests de regresión o smoke manual).
2. Para transportar “mismos filtros” desde el detalle hasta el resolver de “siguiente”:
   - Preferencia: **mismos query parameters** que ya usa cada `index`, propagados con `withQueryString()` en enlaces hacia `show` / validación, y devueltos en el botón Siguiente (GET redirect a la siguiente ficha).
   - Alternativa aceptable: **URL firmada** (`URL::hasValidSignature()` o `hash_hmac` de payload JSON con TTL corto) que incluya filtros + `current_id` + tipo de protocolo — documentar elección y motivo (tamaño de URL, seguridad).
3. Autorización: el usuario debe poder ver/editar el protocolo destino con las mismas reglas que hoy; no exponer IDs fuera de empresa/sede por un simple manipuleo de query (validar `company_id` / `lab_branch_id` como en el listado).

**Verificación:** manipular manualmente `id` en query a un protocolo de otra empresa → 403 o 404, no leak.

## PASO 2 — Endpoint o acción “resolver siguiente”

Por módulo (o una acción polimórfica interna bien acotada), implementar resolución:

1. Reconstruir `$query` = modelo correspondiente + `apply*IndexFilters` + restricciones de visibilidad (`company_id`, sede activa, etc., **idénticas** al index).
2. Añadir condiciones de elegibilidad **Siguiente**:
   - `sent_at` null.
   - No validado según puntos de Descripción (clínico/vet/muestras).
3. Filtrar `where('protocol_number', '>', $current->protocol_number)` **o** comparación robusta acordada al formato real de números (si hay guiones/prefijos variables, usar el mismo criterio que garantice el ejemplo C…0001 → C…0002; si hiciera falta ordenar por `date` + `id` en vez de string, **documentar** decisión con ejemplo de datos reales).
4. `orderBy('protocol_number')->orderBy('id')` → `first()`.
5. Si hay resultado: `redirect()->route(..., [$next] + $request->query())` manteniendo query string de filtros.
6. Si no hay: `back()` o redirect al index con flash “No hay siguiente protocolo pendiente con estos filtros” + query string preservado.

**Anterior:** misma base filtrada + elegibles; `where('protocol_number', '<', $current->protocol_number)` + `orderByDesc('protocol_number')->orderByDesc('id')` → `first()`.

**Muestras (filtros Alpine):** sincronizar filtros relevantes a **query string** en `sample/index` (lectura inicial desde `request()`, escritura Alpine) para que enlaces a `show`/`validate` incluyan los mismos criterios que el resolver en PHP.

**Verificación:** con dos pestañas, validar un protocolo en una; en la otra pulsar Siguiente y ver que el candidato refleja el estado actual.

## Notas / Consideraciones

- **Tensión resuelta en PM:** “Orden del listado” en pantalla ≠ orden de “Siguiente”; el producto eligió **avance temporal por numeración de protocolo** dentro del subconjunto filtrado.
- Si `protocol_number` no ordena lexicográficamente en todos los módulos, usar columnas subyacentes (`date`, `id`) documentadas en código.
- Entrada directa por URL sin query de filtros: comportamiento documentado (tratar como “sin filtros extra” = solo defaults del index + sede activa).
