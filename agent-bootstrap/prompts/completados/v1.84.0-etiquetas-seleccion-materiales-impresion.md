**DEPENDENCIAS:** ninguna

---

## Descripción

Hoy, cuando un protocolo tiene varios materiales (tubos), al reimprimir etiquetas el sistema envía **todas** las etiquetas. Es frecuente querer reimprimir solo una (ej. hemograma / un solo material).

**Objetivo:** en el modal de impresión Zebra, listar **una fila por etiqueta** que el backend ya considera imprimible, con **checkbox marcado por defecto**; el usuario **desmarca** las que no necesita; la impresión Zebra y el fallback por navegador deben respetar solo lo seleccionado.

**Alcance obligatorio — tres módulos:**

1. **Laboratorio clínico** — `LabAdmissionController`, `resources/views/lab/admissions/show.blade.php`, `resources/views/lab/admissions/label.blade.php`.
2. **Laboratorio veterinario** — `VetAdmissionController`, `resources/views/vet/admissions/show.blade.php`, `resources/views/vet/admissions/label.blade.php`.
3. **Muestras (aguas/alimentos)** — `SampleController`, `resources/views/sample/show.blade.php`, vista fallback `resources/views/sample/label.blade.php` (o equivalente multipágina si aplica).

**Sin Designer previo:** reutilizar estilos del modal existente (Tailwind, morado/teal acorde a cada vista).

## PASO 1 — API JSON unificada para muestras

En `SampleController::labelData`:

- Reutilizar la **misma regla de negocio** que ya determina qué tests aportan material (la misma lógica que alimenta `$labelMaterials` en `show` / `printLabel`: filtro por `material`, exclusión de hijos cuando el padre está en el protocolo, etc.).
- Construir un arreglo `labels` **uno por material distinto** (agrupar por `materialRelation->id` o por `material_abbreviation` de forma estable), análogo a `groupByMaterial` del lab clínico.
- Cada elemento debe incluir al menos los campos que `generateZPL` usa en la rama `data.labels`: `protocol_number`, `customer_name`, `material` (sigla), `entry_date`, y para la línea ZPL de contexto: `sample_type` (texto como hoy, ej. tipo de muestra en mayúsculas) y si aplica `branch_name` o equivalente — **alinear nombres de keys** con lo que ya envían clínico/vet para no duplicar ramas en JS.
- Incluir un identificador estable por fila para la UI, por ejemplo `material_key` (string: preferir id numérico del material como string, o `material_code` + sufijo si hubiera colisión teórica).
- Mantener compatibilidad hacia atrás solo si es trivial; si no, actualizar solo el consumidor (`zebra-label-print.js` + modales) en el mismo commit. Objetivo: **siempre** que haya impresión por material desde muestras, el JSON lleve `labels` cuando corresponda más de un tubo o un solo tubo (array de un elemento).

Actualizar tests Feature existentes de `labelData` de muestras si los hay; agregar test mínimo que aserte estructura `labels` y conteo por materiales distintos.

## PASO 2 — `zebra-label-print.js` y modal Alpine

### 2.1 — Cargar datos al abrir el modal

- Tras `openModal(labelDataUrl)`, además de detectar impresoras, hacer **fetch** a `labelDataUrl` (headers como hoy) y guardar en el estado del componente: `payload`, `labels`, `selectedKeys` (Set o array).
- Inicializar `selectedKeys` con **todas** las claves al cargar.
- Si el JSON no trae `labels` pero trae el formato legacy de una sola etiqueta, tratar como un solo ítem seleccionable (compatibilidad defensiva) o migrar completamente en PASO 1 para que no quede legacy.

### 2.2 — UI en el modal (tres vistas Blade)

En cada `show.blade.php` (lab, vet, sample), en la sección “Zebra disponible”, **antes** de impresora/copias o **después** de copias según UX limpia:

- Listado con checkbox por cada `label` (texto: sigla + nombre de material si viene `material_name`, o solo sigla).
- Enlaces o botones pequeños: **Seleccionar todas** / **Desmarcar todas** (opcional pero recomendado).
- Deshabilitar botón **Imprimir** si no queda ninguna seleccionada; mostrar mensaje breve.
- Vista previa: actualizar el texto “Se imprimirán N etiquetas” y la miniatura para reflejar **solo seleccionadas** (primer seleccionado o rotación simple — no hace falta carrusel).

### 2.3 — Generación ZPL

- Refactorizar para que `printSampleLabel` reciba opcionalmente el subconjunto de `labels` filtrado por `selectedKeys`, **o** filtrar dentro antes de `generateZPL`.
- `generateZPL` debe recibir solo las etiquetas a imprimir.
- **Copias:** mantener el comportamiento actual documentado: el campo `copies` aplica a cada bloque `^PQ` / repetición según la implementación vigente; verificar que con **varias** etiquetas seleccionadas no se duplique mal el reemplazo global de `^PQ1` (ajustar si hoy `replaceAll` rompe multipágina).

### 2.4 — Mensaje de éxito

- Si se imprimen varias: texto plural coherente (“Etiqueta(s) enviada(s)…” ya existe en clínico; unificar en sample si sigue en singular).

## PASO 3 — Fallback “Imprimir vía navegador”

**Clínico y vet:** el enlace actual abre la ruta HTML que renderiza **todas** las etiquetas.

- Pasar query string con materiales seleccionados, por ejemplo `?materials=EDTA,SUE` (siglas en el mismo orden/canónico que el backend), **o** un único parámetro codificado. Validar en controlador: filtrar `$labels` a intersección con los pedidos; si parámetro vacío o inválido, default = **todas** (comportamiento actual) o exigir al menos una — **preferido:** si el modal construye la URL siempre con la selección, no hace falta default ambiguo.
- El modal, al hacer clic en “Imprimir vía navegador”, debe abrir la URL **dinámica** con Alpine (`:href` o `window.open`) según selección.

**Muestras:**

- Si hoy `printLabel` solo genera una página con todos los materiales en texto, evolucionar a **multipágina** como `lab/admissions/label.blade.php` (una `.label` por material con su barcode `BarcodeFormatService::forLabel`), y soportar el mismo filtro por query.
- Reutilizar estilos de `lab/admissions/label.blade.php` donde tenga sentido para no divergir tamaños de etiqueta.

## PASO 5 — Verificación manual

- Protocolo con ≥2 materiales: todas preseleccionadas → imprime N ZPL / N páginas HTML.
- Desmarcar todas menos una → una sola etiqueta.
- Muestras: protocolo con un solo material → un checkbox, flujo igual.
- Sin Zebra: solo navegador con query correcto.
- Permisos: sin cambios de policy salvo que el query en HTML requiera mismo permiso `*.print` existente.

## Criterios de aceptación

1. En **lab clínico**, **vet** y **muestras**, el modal lista materiales seleccionables, todos marcados al abrir.
2. La impresión Zebra solo envía etiquetas de materiales **marcados**.
3. El fallback por navegador imprime solo las seleccionadas.
4. No regresión: un material sigue funcionando; permisos intactos.
5. Tests Feature mínimos en JSON de muestras + smoke manual documentado en el mensaje de commit si no hay Zebra en CI.
