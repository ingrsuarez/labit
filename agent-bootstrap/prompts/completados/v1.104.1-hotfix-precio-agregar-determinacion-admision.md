# v1.104.1 — Hotfix: precio correcto al agregar determinación a admisión existente

**VERSIÓN:** v1.104.1  
**SLUG:** hotfix-precio-agregar-determinacion-admision  
**DEPENDENCIAS:** ninguna  
**ESTIMACIÓN:** 1–2 h  
**PRIORIDAD:** alta (bug de facturación activo)

---

## Descripción

Al **crear** una admisión clínica, el precio de cada determinación se calcula correctamente según la obra social / cliente laboral (nomenclador × valor NBU). Al **agregar manualmente** una determinación desde el detalle del protocolo (`lab/admissions/show`), el precio se calcula mal: usa `test.price` genérico o `test.nbu × nbu_value` de la OS, **sin** consultar el nomenclador base (`nbu_units` del pivot `insurance_tests`).

El backend (`LabAdmissionController::addTest`) confía en el `price` enviado por el cliente sin recalcular server-side.

**Objetivo:** Paridad con el alta de admisión y con `DeterminationProfileApplicationService` (que ya usa `AdmissionInsuranceTestPricing::resolve()`).

## PASO 1 — Backend: recalcular precio en `addTest`

En `LabAdmissionController::addTest`:

1. Cargar `$admission->insuranceRelation` (o `Insurance::find($admission->insurance)`).
2. Obtener pricing con `AdmissionInsuranceTestPricing::resolve($insurance, $test)`.
3. Usar `$pricing['price']` y `$pricing['nbu_units']` al crear `AdmissionTest` — **ignorar** `$request->price` para el cálculo (puede mantenerse en validación por compatibilidad, pero el valor persistido es el del servicio).
4. Respetar `requires_authorization` / `copago` del pricing si el request no los envía explícitamente (o mapear `authorization_status` desde `$pricing['requires_authorization']` cuando venga `not_required` por defecto del modal).
5. Hijos: mantener `price => 0` como hoy.

**Verificación:** Agregar una práctica cuyo nomenclador base tenga `nbu_units` distinto de `test.nbu` debe persistir el precio del nomenclador, no el fallback genérico.

## PASO 2 — Frontend: modal agregar práctica en `show.blade.php`

Reemplazar búsqueda client-side sobre `$availableTests` por fetch a la API existente:

```
GET {{ route('lab.admissions.searchTests') }}?q={query}&insurance_id={{ $admission->insurance }}
```

Comportamiento (paridad con `create.blade.php`):

- Mínimo 2 caracteres para buscar.
- Mostrar `calculated_price` en cada resultado.
- Enviar en el form hidden `price` = `calculated_price` (el backend igual recalcula en PASO 1).
- Filtrar prácticas ya presentes en el protocolo (`existingTestIds` desde `$admission->admissionTests->pluck('test_id')`).

Eliminar lógica incorrecta:

```javascript
let price = test.price || 0;
if (price === 0 && test.nbu) {
    price = test.nbu * insuranceNbu;
}
```

Opcional: dejar de pasar `$availableTests` desde `show()` si ya no se usa (solo si grep confirma cero referencias).

## PASO 3 — Tests Feature

Crear `tests/Feature/LabAdmissionAddTestPricingTest.php` (o nombre equivalente):

**Caso A — nomenclador base:**

1. Nomenclador base con práctica X, `nbu_units = 5`.
2. Obra social con `nomenclator_id` apuntando al base, `nbu_value = 100`.
3. Admisión existente con esa OS.
4. `POST lab.admissions.addTest` con `test_id = X`.
5. Assert: `admission_tests.price = 500`, `nbu_units = 5`.

**Caso B — fallback laborales sin nomenclador:**

1. OS tipo `laborales`, `nbu_value = 100`, sin `nomenclator_id`.
2. Test con `nbu = 2.5`, `price = 0`.
3. Agregar a admisión existente.
4. Assert: `price = 250`.

**Caso C — no usar test.price genérico:**

1. Test con `price = 9999` (precio genérico alto).
2. OS con nomenclador que da precio distinto.
3. Assert precio ≠ 9999.

Permisos: usar usuario con `lab-admissions.edit` o el permiso que exija la ruta `addTest`.
