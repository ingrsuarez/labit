# v1.89.0 — Preservar estado del formulario de recibo de cobro ante errores

**Tipo:** bugfix UX — pérdida de datos en formulario
**Prioridad:** alta — impacto operativo directo (reproceso manual)
**Dependencias:** ninguna
**Estimación:** 1h

---

## Qué hace

Cuando el formulario de creación de recibo de cobro (`/collection-receipts/create`) falla
por validación o por un error de runtime, el formulario debe volver con **todo** el estado
que el usuario había cargado: facturas seleccionadas con sus montos, medios de pago
(tipo, monto, cuenta bancaria, datos de e-cheq) y retenciones.

Hoy solo se preservan 3 campos simples (cliente, fecha, notas). Todo lo demás (facturas,
pagos, retenciones) lo maneja Alpine.js que siempre arranca vacío al recargar, ignorando
el `old()` de Laravel.

---

## Contexto — Archivos a leer antes de empezar

```
app/Http/Controllers/CollectionReceiptController.php  ← store() líneas 75-123, validateCollectionReceiptRequest() línea 391+
resources/views/collection-receipts/create.blade.php  ← formulario Alpine.js, init() línea 316, hidden inputs líneas 243-270
```

---

## Causa raíz

1. **Alpine.js ignora `old()`**: el `init()` del componente Alpine (línea 316) inicializa
   `invoices: []`, `paymentLines: [emptyPaymentLine()]`, `withholdingLines: []` — siempre
   desde cero. Cuando Laravel redirige back con `withInput()`, los datos están en la sesión
   pero Alpine no los lee.

2. **`catch (\Throwable)` re-lanza la excepción** (línea 116-118): en lugar de redirigir
   con input, produce una página 500 donde el usuario pierde todo.

---

## PASO 1 — Controller: manejar `\Throwable` con redirect graceful

En `CollectionReceiptController::store()`, cambiar el catch de `\Throwable`:

**Antes (líneas 116-118):**
```php
} catch (\Throwable $e) {
    DB::rollBack();
    throw $e;
}
```

**Después:**
```php
} catch (\Throwable $e) {
    DB::rollBack();
    \Log::error('Error al crear recibo de cobro: '.$e->getMessage(), [
        'trace' => $e->getTraceAsString(),
        'customer_id' => $request->input('customer_id'),
    ]);
    return redirect()->back()->withInput()->with('error', 'Ocurrió un error al guardar el recibo. Por favor intentá nuevamente.');
}
```

Esto garantiza que incluso errores inesperados redirigen de vuelta con el input preservado.

## PASO 2 — Vista: pasar `old()` como JSON a Alpine.js

En `resources/views/collection-receipts/create.blade.php`, **antes** de la función
`collectionForm()` (antes de la línea 280 aprox, dentro del bloque `<script>` o
`x-data`), agregar variables con los datos previos:

```javascript
const oldInvoices = @json(old('invoices', []));
const oldPayments = @json(old('payments', []));
const oldWithholdings = @json(old('withholdings', []));
```

Ubicar estas líneas junto a la declaración de `invoicesByCustomer` y `bankAccounts`
que ya existen como variables Blade-to-JS.

## PASO 3 — Alpine: restaurar facturas seleccionadas desde `old()`

En el `init()` del componente Alpine, **después** de la línea que carga las facturas
del cliente:

```javascript
this.invoices = JSON.parse(JSON.stringify(invoicesByCustomer[this.customerId] || []));
```

Agregar la restauración de selecciones previas:

```javascript
if (oldInvoices.length > 0) {
    const oldMap = {};
    oldInvoices.forEach(oi => {
        oldMap[oi.sales_invoice_id] = parseFloat(oi.amount);
    });
    this.invoices.forEach(inv => {
        if (oldMap[inv.id] !== undefined) {
            inv.selected = true;
            inv.amount = oldMap[inv.id];
        }
    });
}
```

**Lógica:** recorre las facturas que vienen del servidor (con saldos actualizados) y
marca como seleccionadas las que el usuario había elegido, restaurando también el monto
que había ingresado.

## PASO 4 — Alpine: restaurar medios de pago desde `old()`

En la declaración del componente, cambiar la inicialización de `paymentLines`:

**Antes (línea 314):**
```javascript
paymentLines: [emptyPaymentLine()],
```

**Después:**
```javascript
paymentLines: oldPayments.length > 0
    ? oldPayments.map(p => ({
        line_type: p.line_type || 'efectivo',
        amount: parseFloat(p.amount) || 0,
        bank_account_id: p.bank_account_id || '',
        cheque_number: p.cheque_number || '',
        bank_name: p.bank_name || '',
        due_date: p.due_date || '',
    }))
    : [emptyPaymentLine()],
```

## PASO 7 — Verificación manual

Probar los siguientes escenarios:

1. **Validación falla (monto vacío):** cargar recibo con 2+ facturas y un medio de pago
   sin monto → submit → verificar que el formulario vuelve con todo seleccionado.

2. **Validación falla (totales no coinciden):** cargar recibo donde el total de pagos +
   retenciones no coincide con el total de facturas → submit → verificar que vuelve con
   las facturas marcadas, montos, pagos y retenciones intactos.

3. **Seleccionar cuenta bancaria para transferencia:** cargar transferencia con cuenta
   seleccionada → provocar error → verificar que la cuenta bancaria sigue seleccionada.

4. **E-cheq con datos:** cargar e-cheq con número, banco y vencimiento → provocar error →
   verificar que los datos del e-cheq se preservan.

5. **Retenciones cargadas:** cargar 2 retenciones con tipo/monto → provocar error →
   verificar que ambas retenciones vuelven con sus datos.
