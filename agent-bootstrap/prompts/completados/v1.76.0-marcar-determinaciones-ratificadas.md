# v1.76.0 — Marcar determinaciones como “ratificadas” (valores anormales controlados)

**Tipo:** minor — nueva capacidad funcional  
**Prioridad:** media-alta — trazabilidad clínico / QA en resultados fuera de rango  
**Dependencias:** ninguna

---

## Qué hace

En los protocolos donde se cargan y validan resultados (**Laboratorio Clínico**, **Veterinario** y **Muestras aguas/alimentos**) se agrega un indicador opcional por **determinación** (fila): **Ratificado**.

Significa que el resultado es **anormal** o requirió **revisión** y fue **explicitamente verificado por el equipo** antes de cerrar validación.

- Es **opcional**: por defecto desmarcado.  
- Queda **persistido en base de datos** con marca de tiempo y usuario (igual filosofía que `validated_by`/`validated_at`).  
- Visible en pantalla como checkbox + leyenda; **opcionalmente** un símbolo o nota discreta en el **PDF del informe** cuando esté marcado.

---

## Modelo de datos

Tres tablas pivote línea-determinación:

| Tabla | Modelo |
|-------|--------|
| `admission_tests` | `AdmissionTest` |
| `vet_admission_tests` | `VetAdmissionTest` |
| `sample_determinations` | `SampleDetermination` |

### Migración (una por tabla o combinada)

Por cada tabla agregar columnas nullable / con default:

```php
$table->boolean('is_ratified')->default(false)->after('is_validated'); // o al final antes de timestamps
$table->timestamp('ratified_at')->nullable();
$table->foreignId('ratified_by')->nullable()->constrained('users')->nullOnDelete();
```

**Casts en modelo:** `'is_ratified' => 'boolean'`, `'ratified_at' => 'datetime'`  
**`$fillable`:** incluir `is_ratified`, `ratified_at`, `ratified_by`.

### Reglas de negocio (recomendadas)

1. Solo filas que **llevan resultado** (`hasResult()` o equivalente) pueden marcarse como ratificadas (no tiene sentido en padres “título”).
2. Al **marcar** el checkbox desde la UI oficial: guardar `is_ratified = true`, `ratified_at = now()`, `ratified_by = auth()->id()`.
3. Al **desmarcar**: `false`, `null`, `null`.
4. **`unvalidate`/desvalidar** determinación (donde aplique): decidir política (**recomendado:** limpiar también ratificación como se limpian `validated_*`, para que vuelvan a cargar/controlar).
5. Roles: checkbox editable solo si el usuario puede validar (`lab-results.validate`, permisos equivalentes vet y muestras) **o**, si prefieren operativa mixta, permitir marca al guardar resultado con rol `lab-results.create` — **valor por defecto del prompt:** solo quien tiene permiso de **validar** puede marcar/desmarcar “Ratificado”; si el negocio quiere técnico + bioquímico, ajustar en implementación tras confirmación del usuario.

## Backend — rutas/controladores

### Laboratorio Clínico

- **`LabAdmissionController::saveResults`**  
  Por cada entrada en `results`, si el `AdmissionTest` corresponde a la admisión y **no está validado**, además de `result`/`unit`/`reference_value` aceptar `is_ratified` (presente/`'1'`/`true`).  
  Aplicar lógica de `ratified_at`/`ratified_by` según el valor booleano.

- **`LabAdmissionController::validateTest`** (y **`validateAll` si incluye payloads**): opcionalmente, si viene `is_ratified` en POST, persistir igual; si solo usan checkbox en bulk `saveResults`, no hace falta.

- **`unvalidateTest`**: si se adopta política limpiar ratificación → resetear tres campos.

Comprobar si existe **`saveResult`** individual (`POST` por una práctica) y alinear igual.

### Veterinario y Muestras

Repetir los mismos puntos donde hoy se persiste resultado masivo (`saveResults` / rutas paralelas).  
Ubicaciones típicas: `VetAdmissionController`, `SampleController` (*load-results / save results* mirar nombres reales).

Revisión de validación **`ALREADY_VALIDATED`** en API de ingesta externa (**fuera de scope**: no debe tocar ese flujo salvo orden explícita).

## Vistas Blade

Por cada tabla de cargar resultados (show o load-results):

1. Nueva columna de cabecera: **Ratif.** (o **Ratificado** con `title`).
2. Checkbox **solo filas hijo/detalle**, no agrupadores.
3. `disabled`/solo lectura si la fila **ya está validada** (consistente con campos resultado).
4. Tooltip accesible: *“Valor anormal/atípico — verificado bioquímico”*.

Asegurar que el campo viaja en el `name` del mismo array que usa `Guardar Resultados` (ej. `results[][is_ratified]`) usando `hidden`/`value="0"` + checkbox convención HTML si hace falta enviar unchecked.

Referencias de contexto probable:

```
resources/views/lab/admissions/show.blade.php
resources/views/vet/admissions/show.blade.php   (y/o load-results equivalente)
resources/views/sample/load-results.blade.php   (confirmar archivo real del listado muestras)
```

## PDF (opcional pero recomendado)

En plantillas **`pdf-mpdf`** de admisión clínico, vet y muestra:

- Si `is_ratified`: mostrar marca discreta junto al resultado (ej. `*` o texto pequeño “ratif.”)  
- Una sola línea de leyenda al pie si **existe alguna determinación ratificada**  
  ejemplo:  
  **`*`** *Resultados marcados fueron revisados ante valores atípicos.*

Si el usuario prefiere **no** mencionar ratificación en el informe público → incluir configuración rápida (env/feature flag) — **valor por defecto del prompt:** **sí** mostrar marca + leyenda (transparencia hacia médico solicitante).

## Verificación

1. Crear protocolo/fila resultado anormal → marcar Ratificado sin validar → guardar → debe persistir usuario/fecha si `true`.  
2. Validar práctica → ratificación visible read-only si no se limpia al validar.  
3. Desvalidar → según política (limpieza esperada si se adoptó punto 5).  
4. PDF muestra marca solo en filas ratificadas cuando corresponda.  
5. Módulos paralelos tres veces — no regresionar permisos.
