## Descripción

El módulo de laboratorio clínico existe y funciona, pero hoy solo el admin puede
acceder. Se necesitan 3 roles nuevos con permisos granulares para habilitar el flujo
operativo real del laboratorio:

1. **Recepcionista** (`recepcion-lab`) — crea admisiones, ve el estado de protocolos,
   imprime y envía informes **solo cuando están validados**.
2. **Técnico de laboratorio** (`tecnico-lab`) — carga y edita resultados en admisiones
   existentes, ve preview del informe antes de validar.
3. **Bioquímico** (`bioquimico`) — carga resultados + valida resultados + preview +
   imprime y envía informes validados + accede a reportes.

Flujo lineal: Recepcionista crea admisión → Técnico/Bioquímico carga resultados →
Bioquímico valida → Recepcionista/Bioquímico imprime y envía informe validado.

## PASO 2 — Proteger rutas con middleware

En `routes/web.php`, buscar las rutas del módulo de laboratorio y agregar middleware
de permisos donde falte:

1. La sección de laboratorio debe requerir `lab.section`:
   ```php
   Route::middleware(['permission:lab.section'])->group(function () {
       // rutas de lab
   });
   ```

2. Las rutas de creación de admisiones deben requerir `lab-admissions.create`.
3. Las rutas de resultados deben requerir `lab-results.create` o `lab-results.edit`.
4. La ruta de validación debe requerir `lab-results.validate`.
5. Las rutas de reportes (preview, print, send) deben tener sus permisos respectivos.

> ⚠️ Verificar las rutas reales del proyecto — pueden diferir de lo esperado.
> Leer `routes/web.php` y adaptarse a la estructura existente.

**Verificación:** `php artisan route:list | grep lab` muestra las rutas con middleware de permisos.

## PASO 3 — Proteger las vistas con @can

Recorrer las vistas del módulo de laboratorio y agregar directivas `@can`:

### Vistas de admisiones (listado)
- Botón "Nueva admisión": `@can('lab-admissions.create')`
- Link a detalle: `@can('lab-admissions.show')`

### Vistas de detalle de admisión (show)
- Sección/formulario de carga de resultados: `@can('lab-results.create')`
- Botón "Validar": `@can('lab-results.validate')`
- Botón "Preview": `@can('lab-reports.preview')`
- Botón "Imprimir": `@can('lab-reports.print')` — además verificar que el protocolo esté validado
- Botón "Enviar": `@can('lab-reports.send')` — además verificar que el protocolo esté validado

### Lógica condicional para imprimir/enviar
Los botones de imprimir y enviar deben tener **doble condición**:
```blade
@can('lab-reports.print')
    @if($admission->is_validated)
        {{-- mostrar botón imprimir --}}
    @endif
@endcan
```

Si la admisión no tiene un campo `is_validated` o equivalente, buscar cómo se
determina hoy que un protocolo está validado y usar esa condición.

### Sidebar
- La sección "Laboratorio" en el sidebar debe ser visible para usuarios con `lab.section`:
```blade
@can('lab.section')
    {{-- links del módulo de laboratorio --}}
@endcan
```

**Verificación:** cada botón/sección solo aparece para los roles que tienen el permiso correspondiente.

## PASO 4 — Proteger los controladores

En cada método de los controladores de laboratorio, agregar autorización:

```php
public function store(Request $request)
{
    $this->authorize('lab-admissions.create');
    // ...
}
```

O usar middleware en el constructor:
```php
public function __construct()
{
    $this->middleware('permission:lab-admissions.create')->only(['create', 'store']);
}
```

Adaptar al patrón que el proyecto ya use (verificar cómo lo hacen los controladores
de ventas/compras que ya tienen permisos implementados).

**Verificación:** acceder a una ruta protegida sin permiso devuelve 403.

## PASO 5 — Asignar roles a usuarios existentes

Crear un seeder o comando para asignar los roles a los usuarios actuales según su
puesto en el organigrama. Esto requiere:

1. Identificar los User IDs correspondientes a cada puesto (pueden buscarse por email
   o por el Employee vinculado).
2. Asignar los roles:
   ```php
   // Recepcionistas
   User::where('email', 'secretaria@ipac.com.ar')->first()?->assignRole('recepcion-lab');
   User::where('email', 'secretaria3@ipac.com.ar')->first()?->assignRole('recepcion-lab');
   
   // Técnicos
   User::where('email', 'rrhh@ipac.com.ar')->first()?->assignRole('tecnico-lab');
   
   // Bioquímicos y Directores Técnicos
   User::where('email', 'laboratorio.cipolletti@ipac.com.ar')->first()?->assignRole('bioquimico');
   // ... etc
   ```

> ⚠️ Los emails de la captura son orientativos — verificar los emails reales
> de los usuarios en la base de datos antes de asignar.

**Verificación:** cada usuario tiene el rol correcto. Verificar con tinker.
