# v1.111.1 — Botón extracciones: visible solo con pendientes y orden por protocolo

**VERSIÓN:** v1.111.1  
**SLUG:** boton-extracciones-condicional-sede  
**DEPENDENCIAS:** v1.110.0 (extracciones pendientes lab clínico)  
**ESTIMACIÓN:** 1–2 h  
**PRIORIDAD:** alta — UX operativa recepción / sala de extracción  
**DISEÑO:** no — ajuste de visibilidad sobre UI existente (v1.110.0)

---

## Descripción

Refinar el botón **Extracciones** del header del lab clínico (v1.110.0):

1. **Ocultar** el botón cuando no hay extracciones pendientes (`count === 0`).
2. **Mostrar** el botón solo cuando `count > 0` (con badge y estilos actuales).
3. **Filtrar por sede activa** cuando el usuario tiene una sede seleccionada en el header.
4. Con **"Todas las sedes"** activo: contador y modal deben incluir **todas** las pendientes de **todas** las sedes.
5. La lista del modal debe ordenarse por **`protocol_number` ascendente** (no por `created_at`).

**Decisiones PM acordadas:**

| Tema | Decisión |
|------|----------|
| Visibilidad botón | Solo si `count > 0`; oculto por defecto hasta el primer fetch |
| Sede concreta | Contador y lista filtrados por `active_lab_branch_id()` |
| Todas las sedes | Suma global; lista con todas las pendientes + columna sede si hay mezcla |
| Orden lista | `protocol_number` ASC |
| Mobile | Fuera de alcance (header lab oculto en `md:` hoy) |
| Designer | No |

**Fuera de alcance:** modal redesign, formulario admisión, KPIs, polling más frecuente, backfill histórico.

## PASO 1 — Orden por protocolo en el servicio

En `AdmissionSampleDrawService`:

1. Cambiar el orden en `listPending()` de `orderBy('created_at')` a **`orderBy('protocol_number')`** (ASC).
2. Aplicar el mismo criterio en `pendingQuery()` o centralizar el `orderBy` ahí para consistencia.
3. Mantener el filtro por sede existente en `pendingQuery(?int $labBranchId)`:
   - `$labBranchId` null → sin filtro de sede (todas).
   - `$labBranchId` set → `where('lab_branch_id', $labBranchId)`.

**Nota:** el filtrado post-query por `admissionRequiresSampleDraw()` se mantiene; verificar que el orden por protocolo se preserve tras el `filter()` (si hace falta, re-ordenar la colección con `sortBy('protocol_number')` después del filter).

**Verificación:** dos admisiones pendientes en distinta sede con protocolos `C-2026-000010` y `C-2026-000002` → la lista devuelve primero `...000002`.

## PASO 2 — UI header: botón condicional

En `resources/views/lab/partials/header.blade.php`:

1. Envolver el **botón** (no necesariamente todo el modal) con visibilidad condicional:
   - `x-show="count > 0"` + `x-cloak` para evitar flash al cargar.
   - Estado inicial `count: 0` (ya está así).
2. Mantener estilos actuales cuando visible: pulse + badge si `count > 0`.
3. Eliminar o simplificar el estado visual “botón apagado con count 0” (`bg-rose-400` sin pendientes) — **con count 0 el botón no se renderiza visible**.
4. Tras `register()` exitoso, `refreshCount()` ya existe; si `count` pasa a 0, el botón debe ocultarse automáticamente.
5. Opcional UX: si el modal queda abierto y `items.length === 0`, cerrar modal (`modalOpen = false`) — solo si es trivial en Alpine.

**Verificación manual:**

- Sin pendientes en sede activa → botón no visible.
- Crear admisión con material sin tomador → botón aparece con badge.
- Registrar extracción → botón desaparece si era la última pendiente.
- Cambiar sede en header (reload) → contador refleja la sede nueva.
