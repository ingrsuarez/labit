# PROMPT v1.48.5 — Formato extendido de barcode (`protocol_number^material_abbreviation`)

> **Stack:** Laravel 11 / PHP 8.2 / Picqer Barcode Generator.
> **Repo:** `c:\wamp64\www\labit` (este).
> **Agente ejecutor:** AGENTE_PROGRAMADOR.
> **Depende de:** ninguna versión previa (cambio quirúrgico en generación de etiquetas).
> **Necesario para:** v1.49.0 (liscom usa `^material_abbr` para filtrar determinaciones por tubo).

---

## Contexto

Hoy las etiquetas de protocolos generan **un código de barras CODE_128 que codifica solo
`protocol_number`** (ej: `C-2026-001234`). Como un protocolo tiene **una etiqueta por material**
pero todas con el mismo código, el equipo de laboratorio que escanea no puede saber por barcode
qué material está procesando.

Esto rompe el flujo de v1.49.0 (mapeo HL7 + respuesta a queries en liscom), donde el equipo
necesita escanear un tubo concreto y recibir solo las determinaciones de ese material.

**Decisión PM (2026-04-18):** Cambiar el contenido del barcode a:

```
{protocol_number}^{material_abbreviation}
```

Ejemplos:
- `C-2026-001234^EDTA` (clínico, tubo EDTA)
- `A-2026-005678^SUE` (muestras, suero)
- `V-2026-000012^ORI` (veterinaria, orina)

**Por qué `^` como separator:**
- CODE_128 lo soporta sin problemas (ASCII 0x5E).
- No choca con `|` (separator HL7 de campos).
- Es el separator estándar HL7 de componentes (los equipos suelen tratarlo bien).
- No aparece en los `protocol_number` actuales (que usan `-`).

**Defensivo:** si por alguna razón un test no tiene `material_abbreviation`, generar el barcode
con solo `{protocol_number}` (formato actual). No romper.

## Archivos a modificar

### 1. Crear `app/Services/BarcodeFormatService.php`

```php
<?php

namespace App\Services;

class BarcodeFormatService
{
    /**
     * Genera el contenido del barcode para una etiqueta de protocolo.
     *
     * Formato: {protocol_number}^{material_abbreviation}
     * Si material_abbreviation es null/vacío, devuelve solo protocol_number.
     *
     * Separator '^' elegido por compatibilidad CODE_128 + estándar HL7 de componentes.
     *
     * @param  string       $protocolNumber  Ej: "C-2026-001234"
     * @param  string|null  $materialAbbreviation  Ej: "EDTA", "SUE", "ORI"
     * @return string  Ej: "C-2026-001234^EDTA"
     */
    public static function forLabel(string $protocolNumber, ?string $materialAbbreviation): string
    {
        $material = trim((string) $materialAbbreviation);

        if ($material === '') {
            return $protocolNumber;
        }

        return "{$protocolNumber}^{$material}";
    }
}
```

## Test Feature

Crear `tests/Feature/BarcodeFormatTest.php`:

```php
<?php

namespace Tests\Feature;

use App\Services\BarcodeFormatService;
use Tests\TestCase;

class BarcodeFormatTest extends TestCase
{
    public function test_forLabel_returns_protocol_with_material(): void
    {
        $this->assertSame(
            'C-2026-001234^EDTA',
            BarcodeFormatService::forLabel('C-2026-001234', 'EDTA')
        );
    }

    public function test_forLabel_returns_protocol_only_when_material_is_null(): void
    {
        $this->assertSame(
            'C-2026-001234',
            BarcodeFormatService::forLabel('C-2026-001234', null)
        );
    }

    public function test_forLabel_returns_protocol_only_when_material_is_empty_string(): void
    {
        $this->assertSame(
            'A-2026-005678',
            BarcodeFormatService::forLabel('A-2026-005678', '')
        );
    }

    public function test_forLabel_trims_whitespace(): void
    {
        $this->assertSame(
            'V-2026-000012^ORI',
            BarcodeFormatService::forLabel('V-2026-000012', '  ORI  ')
        );

        $this->assertSame(
            'V-2026-000012',
            BarcodeFormatService::forLabel('V-2026-000012', '   ')
        );
    }

    public function test_supports_typical_material_abbreviations(): void
    {
        $cases = ['EDTA', 'SUE', 'ORI', 'CIT', 'HEP', 'FE', 'BV', 'HEM'];
        foreach ($cases as $abbr) {
            $this->assertSame(
                "C-2026-000001^{$abbr}",
                BarcodeFormatService::forLabel('C-2026-000001', $abbr)
            );
        }
    }
}
```

Ejecutar: `php artisan test --filter=BarcodeFormatTest`. Debe pasar 5/5.

## Tensiones (anotar en handoff, NO resolver acá)

1. **ZPL en JS de Zebra.** Si `public/js/zebra-label-print.js` arma el ZPL tomando
   `protocol_number` directamente del JSON de `labelData()` (en vez de un nuevo campo
   `barcode_content`), las impresoras Zebra seguirán imprimiendo el formato viejo. Verificar y
   reportar. Eventual hotfix: agregar `barcode_content` en `labelData()` y consumir desde JS.
2. **Etiquetas pre-impresas en circulación.** Las etiquetas ya impresas (formato viejo) seguirán
   funcionando porque liscom debe parsear barcode con/sin separator (lo hace v1.49.0). No requiere
   reimprimir nada.
3. **Etiquetas de muestras con un solo barcode.** Hoy `SampleController::printLabel` imprime una
   sola etiqueta concatenando materiales. El barcode toma "el primer material" — si el equipo lo
   escanea, recibirá determinaciones de ese material únicamente. Si en el futuro hay equipos HL7
   conectados a muestras, replanificar como v1.48.6 con etiquetas por material.
