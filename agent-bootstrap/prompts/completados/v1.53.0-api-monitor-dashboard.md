# v1.53.0 — Dashboard de monitoreo de la API (ingesta de resultados desde LISCOM)

**VERSIÓN:** v1.53.0
**SLUG:** api-monitor-dashboard
**REPO:** `c:\wamp64\www\labit` (Laravel)
**DEPENDENCIAS:** v1.46.0 (ApiClient + permission `api-clients.manage`) + v1.51.0 (ResultBatch + ResultIngestion)
**ESTIMACIÓN:** 1.5 días
**PRIORIDAD:** media — desbloquea visibilidad operativa pero no bloquea producción

---

## Descripción

Pantalla de monitoreo en labit para ver qué resultados están entrando por la API desde LISCOM
(v1.51.0). Reusa el layout admin existente (sidebar + contenido + Tailwind + Livewire 3).
Sin Designer dedicado.

**Foco:** ingesta de resultados (batches, mensajes, items, rechazos por ALREADY_VALIDATED).
Otros aspectos de la API (sync de protocolos, access logs generales, latencia) quedan fuera
de esta versión — entrarían en una hipotética v1.54.0+ si surge necesidad.

**Vista unificada con permisos para ocultar elementos técnicos:**
- Bioquímico (cualquier user con `view-protocols` o equivalente) ve la parte operativa:
  - Counters del día / semana
  - Lista de batches con drill-down a mensajes y items
  - Qué se ingestó / sobrescribió / rechazó (con razón humana)
  - Link directo al protocolo en labit para ir a revisarlo
- Admin/IT (con permission `api-clients.manage` de v1.46.0) además ve:
  - Datos técnicos del request (raw payload, IP, timestamp exacto)
  - Estado de las API keys (last_used_at, requests_count, sedes desconectadas)
  - Botones de gestión de keys (revocar, regenerar, ver detalle) — link al CRUD de v1.46.0

## PASO 2 — Servicio `ApiMonitorService` (lectura agregada)

Crear `app/Services/Api/ApiMonitorService.php`:

```php
<?php

namespace App\Services\Api;

use App\Models\ApiClient;
use App\Models\ResultBatch;
use App\Models\ResultIngestion;
use Carbon\Carbon;

class ApiMonitorService
{
    /**
     * Counters globales de los últimos N días.
     * Retorna array para los counter cards del dashboard.
     */
    public function getCounters(int $days = 1): array
    {
        $since = Carbon::now()->subDays($days)->startOfDay();

        $ingestionsQuery = ResultIngestion::where('created_at', '>=', $since);

        return [
            'batches_total' => ResultBatch::where('created_at', '>=', $since)->count(),
            'messages_total' => (clone $ingestionsQuery)->count(),
            'messages_ingested' => (clone $ingestionsQuery)->where('status', 'ingested')->count(),
            'messages_partial' => (clone $ingestionsQuery)->where('status', 'partial')->count(),
            'messages_rejected' => (clone $ingestionsQuery)->where('status', 'rejected')->count(),
            'messages_duplicate' => (clone $ingestionsQuery)->where('status', 'duplicate')->count(),
            'rejected_already_validated' => (clone $ingestionsQuery)
                ->where('status', 'rejected')
                ->where('rejection_reason', 'ALREADY_VALIDATED')
                ->count(),
            'rejected_protocol_not_found' => (clone $ingestionsQuery)
                ->where('status', 'rejected')
                ->where('rejection_reason', 'PROTOCOL_NOT_FOUND')
                ->count(),
            // Suma de items_summary (puede ser caro con MySQL JSON; preferir agregados materializados de v1.51.0)
            'items_ingested' => ResultBatch::where('created_at', '>=', $since)->sum('items_ingested'),
            'items_overwritten' => ResultBatch::where('created_at', '>=', $since)->sum('items_overwritten'),
            'items_rejected' => ResultBatch::where('created_at', '>=', $since)->sum('items_rejected'),
            'period_days' => $days,
            'period_start' => $since->toIso8601String(),
        ];
    }

    /**
     * Estado por sede: ApiClients con su última actividad y conteo de batches recientes.
     * Útil para detectar sedes desconectadas (si LISCOM no postea en X horas, algo pasa).
     */
    public function getClientsStatus(int $days = 7): array
    {
        $since = Carbon::now()->subDays($days)->startOfDay();
        $now = Carbon::now();

        return ApiClient::with('labBranch:id,name')
            ->where('active', true)
            ->orderBy('last_used_at', 'desc')
            ->get()
            ->map(function (ApiClient $client) use ($since, $now) {
                $batchesRecent = ResultBatch::where('api_client_id', $client->id)
                    ->where('created_at', '>=', $since)
                    ->count();

                $lastBatch = ResultBatch::where('api_client_id', $client->id)
                    ->orderBy('created_at', 'desc')
                    ->first();

                $minutesSinceLastUse = $client->last_used_at
                    ? $now->diffInMinutes($client->last_used_at)
                    : null;

                return [
                    'client' => $client,
                    'lab_branch_name' => $client->labBranch?->name ?? '-',
                    'batches_recent' => $batchesRecent,
                    'last_batch_at' => $lastBatch?->created_at,
                    'minutes_since_last_use' => $minutesSinceLastUse,
                    'health' => $this->classifyHealth($minutesSinceLastUse),
                ];
            })
            ->all();
    }

    /**
     * Clasificar salud de una sede según última actividad.
     * Umbrales razonables para laboratorio (operación 8-12h/día):
     */
    private function classifyHealth(?int $minutesSinceLastUse): string
    {
        if ($minutesSinceLastUse === null) {
            return 'never_used';   // nunca se conectó
        }
        if ($minutesSinceLastUse <= 60) {
            return 'healthy';      // < 1 hora
        }
        if ($minutesSinceLastUse <= 60 * 12) {
            return 'idle';         // < 12 horas (puede estar fuera de horario)
        }
        if ($minutesSinceLastUse <= 60 * 24 * 3) {
            return 'stale';        // < 3 días
        }
        return 'inactive';         // > 3 días sin actividad
    }

    /**
     * Listado de batches con filtros para la pantalla de listado.
     */
    public function getBatches(array $filters = [], int $perPage = 20)
    {
        $query = ResultBatch::with(['apiClient.labBranch']);

        if (!empty($filters['client_id'])) {
            $query->where('api_client_id', $filters['client_id']);
        }
        if (!empty($filters['date_from'])) {
            $query->whereDate('created_at', '>=', $filters['date_from']);
        }
        if (!empty($filters['date_to'])) {
            $query->whereDate('created_at', '<=', $filters['date_to']);
        }
        if (!empty($filters['has_rejections'])) {
            $query->where('items_rejected', '>', 0);
        }

        return $query->orderBy('created_at', 'desc')->paginate($perPage);
    }

    /**
     * Listado de ingestions con filtros para la pantalla de mensajes.
     */
    public function getIngestions(array $filters = [], int $perPage = 30)
    {
        $query = ResultIngestion::with(['batch.apiClient.labBranch']);

        if (!empty($filters['status'])) {
            $query->where('status', $filters['status']);
        }
        if (!empty($filters['rejection_reason'])) {
            $query->where('rejection_reason', $filters['rejection_reason']);
        }
        if (!empty($filters['protocol_number'])) {
            $query->where('protocol_number', 'like', '%' . $filters['protocol_number'] . '%');
        }
        if (!empty($filters['client_id'])) {
            $query->where('api_client_id', $filters['client_id']);
        }
        if (!empty($filters['date_from'])) {
            $query->whereDate('created_at', '>=', $filters['date_from']);
        }
        if (!empty($filters['equipment_name'])) {
            $query->where('equipment_name', $filters['equipment_name']);
        }

        return $query->orderBy('created_at', 'desc')->paginate($perPage);
    }

    /**
     * Detectar el tipo de protocolo (clinical/sample/vet) y armar la URL de detalle en labit.
     * Permite linkear desde el dashboard al protocolo real.
     */
    public function getProtocolUrl(string $protocolNumber, ?string $protocolType = null): ?string
    {
        $type = $protocolType ?? $this->detectProtocolType($protocolNumber);

        return match ($type) {
            'clinical' => route('admissions.show', ['number' => $protocolNumber]),
            'sample' => route('samples.show', ['number' => $protocolNumber]),
            'vet' => route('vet-admissions.show', ['number' => $protocolNumber]),
            default => null,
        };
    }

    private function detectProtocolType(string $protocolNumber): ?string
    {
        $prefix = strtoupper(substr($protocolNumber, 0, 2));
        return match ($prefix) {
            'C-' => 'clinical',
            'S-' => 'sample',
            'V-' => 'vet',
            default => null,
        };
    }
}
```

> **Nota sobre rutas:** los nombres de ruta (`admissions.show`, etc.) deben coincidir con los reales en `routes/web.php`. Verificar antes y ajustar el match.

### 3.2 — `App\Livewire\Api\BatchesList` (listado batches con filtros)

```php
<?php

namespace App\Livewire\Api;

use App\Models\ApiClient;
use App\Services\Api\ApiMonitorService;
use Livewire\Attributes\Url;
use Livewire\Component;
use Livewire\WithPagination;

class BatchesList extends Component
{
    use WithPagination;

    #[Url(as: 'cliente')]
    public ?int $clientId = null;

    #[Url(as: 'desde')]
    public ?string $dateFrom = null;

    #[Url(as: 'hasta')]
    public ?string $dateTo = null;

    #[Url(as: 'solo_rechazos')]
    public bool $hasRejections = false;

    public function clearFilters(): void
    {
        $this->reset(['clientId', 'dateFrom', 'dateTo', 'hasRejections']);
        $this->resetPage();
    }

    public function render()
    {
        $batches = app(ApiMonitorService::class)->getBatches([
            'client_id' => $this->clientId,
            'date_from' => $this->dateFrom,
            'date_to' => $this->dateTo,
            'has_rejections' => $this->hasRejections,
        ]);

        return view('livewire.api.batches-list', [
            'batches' => $batches,
            'clients' => ApiClient::orderBy('name')->get(['id', 'name']),
        ]);
    }
}
```

## PASO 6 — Comando `api:cleanup` para retención

Crear `app/Console/Commands/ApiCleanup.php`:

```php
<?php

namespace App\Console\Commands;

use App\Models\ResultBatch;
use App\Models\ResultIngestion;
use Illuminate\Console\Command;

class ApiCleanup extends Command
{
    protected $signature = 'api:cleanup
                            {--days= : sobrescribe API_LOG_RETENTION_DAYS}
                            {--dry-run : solo informa, no borra}';

    protected $description = 'Borra ResultBatch y ResultIngestion más viejos que el período de retención';

    public function handle(): int
    {
        $days = (int) ($this->option('days') ?? config('api.log_retention_days', 90));
        $cutoff = now()->subDays($days);

        $this->info("Borrando registros anteriores a {$cutoff->toDateTimeString()} (retención: {$days} días)");

        $ingestionsCount = ResultIngestion::where('created_at', '<', $cutoff)->count();
        $batchesCount = ResultBatch::where('created_at', '<', $cutoff)->count();

        $this->table(['Tabla', 'A borrar'], [
            ['result_ingestions', number_format($ingestionsCount)],
            ['result_batches', number_format($batchesCount)],
        ]);

        if ($this->option('dry-run')) {
            $this->warn('--dry-run: no se borró nada');
            return self::SUCCESS;
        }

        if ($ingestionsCount === 0 && $batchesCount === 0) {
            $this->info('Nada para borrar');
            return self::SUCCESS;
        }

        if (!$this->confirm('¿Confirmás el borrado?', true)) {
            return self::FAILURE;
        }

        // Borrar ingestions primero (FK a batches)
        ResultIngestion::where('created_at', '<', $cutoff)->delete();
        ResultBatch::where('created_at', '<', $cutoff)->delete();

        $this->info('Limpieza completada');
        return self::SUCCESS;
    }
}
```

## Riesgos y mitigaciones

| Riesgo | Mitigación |
|---|---|
| Queries pesadas con `JSON_*` sobre `items_summary` | Usar los contadores agregados de `ResultBatch` (`items_ingested`, `items_overwritten`, etc.) en lugar de re-agregar JSON. La query JSON queda solo en el detalle de 1 ingestion |
| Auto-refresh 10s con muchos users → carga DB | Counters están limitados a `WHERE created_at >= ahora-N`. Con índices del PASO 1 quedan en pocos ms. Si crece, migrar a query con `count(*)` cacheado en Redis cada 30s |
| Permission `view-protocols` no existe / no es transversal | Verificar en handoff. Alternativas: usar el permission real, crear uno nuevo, o restringir a `auth` simple si es overkill |
| Cleanup borra batches con info forense útil | El comando pide confirm interactivo (excepto en cron, donde corre con `--no-interaction`). Documentar bien la retención por defecto |
| Link a protocolo roto si el nombre de ruta no existe | El método `getProtocolUrl` retorna `null` si no detecta tipo. La vista debe mostrar el `protocol_number` plano si no hay URL |
| `raw_request` truncado de v1.51.0 puede mostrarse mal | Detectar marker `__truncated__` y avisar visualmente que el payload está truncado |
