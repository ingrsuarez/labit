# v1.14.0 — Precios en protocolos de aguas y alimentos

**VERSIÓN:** v1.14.0  
**SLUG:** precios-aguas-alimentos  
**DEPENDENCIAS:** ninguna  
**ESTIMACIÓN:** 2-3h  
**PRIORIDAD:** alta

---

## Descripción

Al crear un protocolo de muestras (aguas, alimentos, hielo), el recepcionista necesita
ver el precio de cada determinación para poder cobrar o facturar en el momento.

Actualmente:
- El buscador de determinaciones muestra TODAS las prácticas (clínicas + aguas) mezcladas
- No se muestran precios en el formulario
- `sample_determinations` no almacena precio
- No hay forma de diferenciar prácticas clínicas de aguas/alimentos en la tabla `tests`
- Los clientes (`customers`) no tienen campo de descuento

El pricing de aguas/alimentos es diferente al de análisis clínicos:
- Cada determinación tiene un **precio fijo** (en `tests.price`), no se usa NBU
- Los clientes pueden tener un **descuento porcentual** general que aplica a todas las prácticas
- Fórmula: `precio final = tests.price × (1 - customer.discount_percent / 100)`

## PASO 1 — Migraciones

### 1.1 — Campo `categories` en tabla `tests`

```bash
php artisan make:migration add_categories_to_tests_table
```

Agregar columna `categories` JSON nullable. Luego inicializar todos los tests
existentes con `["clinico"]` por defecto:

```php
public function up(): void
{
    Schema::table('tests', function (Blueprint $table) {
        $table->json('categories')->nullable()->after('box');
    });

    // Inicializar todos los tests existentes como clínicos
    DB::table('tests')->whereNull('categories')->update([
        'categories' => json_encode(['clinico']),
    ]);
}
```

Valores posibles dentro del array: `clinico`, `aguas_alimentos`, `veterinario` (futuro).

Un test puede pertenecer a múltiples categorías. Ejemplos:
- Solo clínico: `["clinico"]`
- Solo aguas/alimentos: `["aguas_alimentos"]`
- Compartido (ej. Coliformes Totales): `["clinico", "aguas_alimentos"]`
- Futuro veterinario: `["veterinario"]` o `["clinico", "veterinario"]`

### 1.2 — Campo `discount_percent` en tabla `customers`

```bash
php artisan make:migration add_discount_percent_to_customers_table
```

```php
$table->decimal('discount_percent', 5, 2)->default(0)->after('status');
```

### 1.3 — Campo `price` en tabla `sample_determinations`

```bash
php artisan make:migration add_price_to_sample_determinations_table
```

```php
$table->decimal('price', 10, 2)->nullable()->after('test_id');
```

Almacena el precio **al momento de crear** la determinación (precio con descuento ya aplicado),
para que el historial sea inmutable aunque cambien precios o descuentos después.

### 3.2 — Método `store()`

Al crear cada `SampleDetermination`, calcular y guardar el precio con descuento:

```php

$discountPercent = $customer->discount_percent ?? 0;
$discountMultiplier = 1 - ($discountPercent / 100);

// Dentro del foreach de determinations:

$finalPrice = round($basePrice * $discountMultiplier, 2);

SampleDetermination::create([
    'sample_id' => $sample->id,
    'test_id' => $testId,
    'price' => $finalPrice,
    'unit' => $test->unit,
    'method' => $test->method,
    'reference_value' => $parentRef['value'],
```

(su precio es 0 o null, ya que el precio es del test padre).

### 4.2 — Variables reactivas para descuento y totales

Agregar al objeto Alpine.js:

```javascript
customerId: '{{ old('customer_id', '') }}',
discountPercent: 0,

get subtotal() {
    return this.selectedTests.reduce((sum, t) => sum + parseFloat(t.price || 0), 0);
},

get discountAmount() {
    return this.subtotal * (this.discountPercent / 100);
},

get total() {
    return this.subtotal - this.discountAmount;
},

formatNumber(num) {
    return new Intl.NumberFormat('es-AR', { minimumFractionDigits: 2, maximumFractionDigits: 2 }).format(num);
},
```

## PASO 7 — Verificación

### 7.1 — Flujo completo de prueba

1. Ejecutar migraciones
2. Marcar al menos 2-3 tests existentes con `categories` incluyendo `aguas_alimentos` y asignarles `price`
   (ej: `UPDATE tests SET categories = '["clinico","aguas_alimentos"]', price = 15000 WHERE name LIKE '%coliformes%'`)
3. Asignar un descuento (ej. 10%) a un cliente
4. Crear un nuevo protocolo con ese cliente
5. Verificar que solo aparecen prácticas con `aguas_alimentos` en su array de categorías
6. Verificar que un test compartido (ej. Coliformes) aparece tanto en muestras como en admisiones clínicas
6. Verificar que se muestran precios en el dropdown y en la tabla
7. Verificar que el total aplica el descuento correctamente
8. Verificar que `sample_determinations.price` se guarda con el precio final (con descuento)
9. Verificar que la vista show muestra los precios

### 7.2 — Compatibilidad

- Protocolos existentes (sin precio en determinaciones) deben seguir funcionando
- El módulo de admisiones (lab clínico) no debe verse afectado
- Tests sin categoría (`categories = null` o solo `["clinico"]`) no aparecen en el buscador de muestras
- Tests compartidos (ej. `["clinico", "aguas_alimentos"]`) aparecen en ambos módulos
