# v1.112.1 — Portal del empleado: home con cards aprendidas + perfil

**VERSIÓN:** v1.112.1  
**SLUG:** portal-home-cards-perfil  
**DEPENDENCIAS:** v1.112.0  
**ESTIMACIÓN:** 2–3 h  
**PRIORIDAD:** media

---

## Descripción

En **v1.112.0** el motor de aprendizaje (`NavigationCatalog`, `user_navigation_stats`,
`TrackNavigationAccess`) ya registra visitas del portal, pero la pantalla de entrada
`portal.dashboard` sigue siendo solo **Mi Perfil** (datos personales, antigüedad, licencias).

Los empleados con rol **solo portal** nunca ven `/dashboard` — `CheckSystemAccess` los
redirige a `portal.dashboard`. Esta versión lleva el **mismo home con cards** al portal,
**sin duplicar** el motor de stats.

**Layout acordado:** cards arriba + bloque de perfil actual abajo (sin pestañas, sin
reemplazar el perfil).

**Handoff PM:** `agent-bootstrap/handoffs/v1.112.1-pm-to-dev.md`

## PASO 1 — Extender `NavigationCatalog` para el portal

En `app/Support/NavigationCatalog.php` agregar:

```php
/**
 * Shortcuts del portal (keys con prefijo portal-).
 *
 * @return array<int, array{name: string, description: string, route: string, icon: string}>
 */
public static function shortcutsForPortalUser(User $user, int $limit = 8): array
```

Lógica:

1. Filtrar `shortcuts()` donde `key` empiece con `portal-`.
2. Aplicar `userCanSeeShortcut()` (supervisor para `portal-team`, employee para el resto).
3. Reutilizar la **misma** lógica de ordenamiento por `user_navigation_stats` que
   `shortcutsForUser()` — extraer a método privado `sortShortcutsForUser()` si hace
   falta evitar duplicación.
4. Devolver máximo `$limit` cards (8 en portal — menos que admin porque el catálogo
   portal es más chico).

**Verificación:** test unitario — usuario con employee ve `portal-payslips`; sin employee
ve array vacío.

  el historial es el mismo — comportamiento deseado.
