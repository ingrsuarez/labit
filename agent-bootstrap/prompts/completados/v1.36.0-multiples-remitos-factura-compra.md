# v1.36.0 — Múltiples remitos asociados a una factura de compra

**VERSIÓN:** v1.36.0  
**SLUG:** multiples-remitos-factura-compra  
**DEPENDENCIAS:** v1.34.0 (remito + `updates_stock`), v1.32.4 (remitos editables solo sin FC)  
**ESTIMACIÓN:** medio día  
**PRIORIDAD:** alta

---

## Descripción

Hoy `purchase_invoices.delivery_note_id` permite **un solo** remito. En la operatoria real
una misma factura de compra puede agrupar **varios remitos** (3 o más).

Esta versión introduce una relación **muchos a muchos** entre facturas de compra y remitos,
migra los datos existentes, actualiza formularios y la lógica de stock / UI que hoy asume
un único `delivery_note_id`.

---

## Reglas de negocio

1. **Un remito solo puede estar vinculado a una factura de compra a la vez**  
   En la tabla pivote, `delivery_note_id` debe ser **único** (índice unique).

2. **Validación de coherencia** al asociar cada remito:
   - Mismo `supplier_id` que la factura.
   - Misma `company_id` que la factura (alineado con multi-empresa y listados).
   - **No reutilizar remitos:** ningún remito ya vinculado a **otra** factura de compra (pivote **o**
     `purchase_invoices.delivery_note_id` legacy mientras exista) puede seleccionarse. En **edición**,
     los remitos ya asignados a **esta** factura sí son válidos; el filtro excluye solo “ocupados por
     otras FC”.

3. **Importación de ítems** al agregar un remito (o botón explícito “Importar desde remitos”):
   - Para **cada** remito vinculado, traer ítems (insumo, cantidad, lote/vencimiento si aplica)
     **sin precio**, igual que v1.34.0.
   - **Comportamiento:** **apendear** líneas nuevas (no reemplazar toda la grilla).
   - **Líneas siempre separadas:** si el mismo insumo (o mismo insumo + lote) aparece en más de un
     remito, se generan **varias líneas** en la factura (una por línea de remito). **No** fusionar
     ni sumar cantidades en una sola fila.
   - Si el usuario agrega un segundo remito después de cargar precios, las líneas nuevas
     llegan con precio 0 (comportamiento actual).

4. **Stock / `updates_stock`:**  
   Si la factura tiene **al menos un** remito vinculado, el criterio es el mismo que hoy con
   un solo remito: el stock se entiende ya impactado por los remitos; los checkboxes
   “Actualiza stock” por línea quedan deshabilitados con el mismo mensaje informativo
   (adaptado al plural: “Los remitos asociados ya actualizaron el stock”).

5. **Quitar un remito** del formulario:
   - Solo desasocia el remito de la factura **no** borra automáticamente líneas de la FC
     (evita borrar precios cargados por error). Opcional: tooltip “Las líneas importadas
     de ese remito no se eliminan solas”.

6. **Modelo `DeliveryNote` → facturas:**  
   Reemplazar o complementar `hasPurchaseInvoice()` / comprobaciones que usen solo
   `PurchaseInvoice::where('delivery_note_id', …)` por consulta a la pivote.

## PASO 1 — Migración y pivote

1. Crear tabla pivote `delivery_note_purchase_invoice` (orden alfabético Laravel):
   - `id` bigIncrements (opcional; puede ser solo compuesta)
   - `purchase_invoice_id` FK → `purchase_invoices`, `cascadeOnDelete`
   - `delivery_note_id` FK → `delivery_notes`, `cascadeOnDelete`
   - `timestamps`
   - **Unique** `delivery_note_id` (un remito → una sola FC)
   - **Unique** compuesto `(purchase_invoice_id, delivery_note_id)` por si acaso

2. Migración de datos:
   ```sql
   INSERT INTO delivery_note_purchase_invoice (purchase_invoice_id, delivery_note_id, created_at, updated_at)
   SELECT id, delivery_note_id, NOW(), NOW()
   FROM purchase_invoices
   WHERE delivery_note_id IS NOT NULL;
   ```

3. **No eliminar** la columna `delivery_note_id` en esta versión si hay mucho código legacy:
   - Opción A (recomendada): mantenerla **sincronizada** con el “primer” remito de la pivote
     (orden por `id` de pivote o `created_at`) mediante `sync` en modelo/observers.
   - Opción B: deprecar y quitar en versión posterior tras refactor completo.

   Documentar en código que la fuente de verdad es la pivote.

## PASO 2 — Modelos Eloquent

`PurchaseInvoice`:

```php
public function deliveryNotes(): BelongsToMany
{
    return $this->belongsToMany(DeliveryNote::class, 'delivery_note_purchase_invoice')
        ->withTimestamps();
}

/** @deprecated Usar deliveryNotes; se mantiene por compatibilidad */
public function deliveryNote(): BelongsTo
{
    return $this->belongsTo(DeliveryNote::class);
}

public function hasDeliveryNotes(): bool
{
    return $this->deliveryNotes()->exists();
}
```

`DeliveryNote`:

```php
public function purchaseInvoices(): BelongsToMany
{
    return $this->belongsToMany(PurchaseInvoice::class, 'delivery_note_purchase_invoice')
        ->withTimestamps();
}

public function hasPurchaseInvoice(): bool
{
    return $this->purchaseInvoices()->exists();
}
```

Ajustar cualquier método que antes hacía `PurchaseInvoice::where('delivery_note_id', $this->id)`.

## PASO 4 — UI (create / edit)

- Reemplazar el selector único por:
  - Lista de **chips** (como la captura actual) con **varios** remitos.
  - Botón **“Agregar remito”** que abre modal o dropdown filtrado (proveedor + empresa de la FC).
  - **Listado de candidatos:** solo remitos **libres** (no aparecen si ya están en la pivote con
    otra `purchase_invoice_id`, ni si otra FC los tiene en `delivery_note_id` hasta migrar del todo).
    En edición, pasar `exclude_purchase_invoice_id` (o similar) al endpoint para ignorar la FC actual
    al comprobar “ocupado”.
  - Cada chip: número/remito + botón quitar.
  - Si el usuario intenta forzar un id inválido (POST tampering), responder **422** con mensaje claro.
- Texto de ayuda: stock actualizado por **los remitos asociados** (plural).
- Asegurar que la petición envía `delivery_note_ids[]`.

### PASO 4.1 — Endpoint “remitos disponibles” (recomendado)

- Ruta tipo `GET /delivery-notes/available-for-purchase-invoice` (nombre a convenir) con query:
  `supplier_id`, `company_id`, opcional `purchase_invoice_id` (excluir vínculos de esta FC al chequear duplicados).
- Respuesta: lista de remitos elegibles únicamente. Así la UI **no ofrece** remitos ya facturados en otra FC.

### PASO 4.2 — Orden de foco con Tab en la grilla de ítems

**Requisito de producto:** al cargar precios/cantidades con teclado, el Tab debe seguir:

`Cantidad ítem 0` → `Precio unitario ítem 0` → `Cantidad ítem 1` → `Precio unitario ítem 1` → …

- Archivos: `resources/views/purchase-invoices/create.blade.php` y `edit.blade.php`.
- Asegurar `:id="'item-unit-' + index"` en el input de **precio unitario** (además de `item-qty-${index}`).
- Implementación sugerida (elegir una y documentar):
  - **Opción A — `tabindex` dinámico:** asignar a cantidad y precio de cada fila valores `tabindex`
    consecutivos en el orden deseado (p. ej. base 100 + 2×index y +1). Ajustar para que el **último**
    precio unitario pase el foco al siguiente bloque lógico del formulario (percepciones, notas o botón
    guardar), sin dejar al usuario atrapado.
  - **Opción B — `@keydown.tab`:** en precio unitario, si no es Shift+Tab, `preventDefault` y enfocar
    `#item-qty-${index+1}`; en cantidad, Tab natural hacia precio de la misma fila; desde la última fila,
    Tab sale al siguiente control del formulario. Mantener **Shift+Tab** simétrico hacia atrás.

**Verificación manual:** con varias filas de ítems, Tab solo entre cantidad/precio recorre la secuencia
indicada sin saltos incorrectos.
