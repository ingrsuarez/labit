## Descripción

Space10 (SPACE4CLINIC, repo `c:\wamp64\www\space10`) ya expone `POST /api/upload/lab` con auth Sanctum. Recibe multipart: `laboratory` (PDF), `idPaciente` (DNI), `file_date` (YYYY-MM-DD). Busca al paciente por `Paciente.idPaciente` y registra el archivo en `upload_files`.

Labit debe subir automáticamente el PDF del informe clínico a Space10 en dos flujos:

1. **Al enviar email** (individual `sendEmail` y masivo `batchEmail`): tras email exitoso, subir a Space10 si aún no fue subido.
2. **Batch dedicado** desde el listado de admisiones: subir N PDFs sin enviar email.

**Idempotencia:** columna `space10_uploaded_at` en `admissions`. Si ya está seteada, **no** llamar a la API (skip con motivo "ya subido a Space10").

**Alcance:** solo lab clínico (`Admission`). Fuera: vet, muestras, cambios en Space10.

**Regla crítica:** fallo de Space10 **no bloquea** el envío de email. Email OK + Space10 falla → warning al usuario; reintento vía batch.

## PASO 3 — Servicio `Space10UploadService`

Crear `app/Services/Space10UploadService.php`:

**Responsabilidades:**

- `isEnabled(): bool` — `config('space10.enabled')` + URL + token no vacíos
- `uploadAdmission(Admission $admission): Space10UploadResult` (DTO o array con `success`, `skipped`, `message`)

**Lógica `uploadAdmission`:**

1. Si `!isEnabled()` → return skipped/disabled (sin HTTP)
2. Si `$admission->isUploadedToSpace10()` → return skipped `"ya subido a Space10"`
3. Cargar `patient`; si `patientId` vacío → return error `"sin DNI"`
4. Si `!admissionCanReceiveResultsEmail` equivalente (≥1 det. validada) → return error `"sin determinaciones validadas"`
5. Generar PDF binario reutilizando la misma vista/validador que `AdmissionResultMail::makePdfAttachment` — **extraer** método estático `AdmissionResultMail::generatePdfBinary(Admission $admission): string` para no duplicar lógica DomPDF
6. `file_date`: `$admission->date?->format('Y-m-d') ?? now()->format('Y-m-d')`
7. HTTP POST multipart con Guzzle/`Http::attach`:
   - Campo archivo: `laboratory`
   - Campos: `idPaciente` = DNI (`patient.patientId`), `file_date`
   - Header: `Authorization: Bearer {token}`
8. Respuestas:
   - 201 → `$admission->update(['space10_uploaded_at' => now()])`, audit `space10_uploaded`, success
   - 404 → error `"paciente no encontrado en Space10"` (mensaje API)
   - 400/500 → error con body resumido
9. Timeout desde config

**Verificación:** unit test con `Http::fake()` simulando 201 y 404.

## PASO 5 — Endpoint batch Space10

1. Ruta `POST lab/admissions/batch-space10` → `LabAdmissionController::batchSpace10`, nombre `lab.admissions.batch-space10`, **antes** de rutas `{admission}`

2. Validar: `admission_ids` required array min:1

3. Misma query de sede/company que `batchEmail`

4. Por cada admisión:
   - missing sede → skipped
   - ya `space10_uploaded_at` → skipped `"ya subido a Space10"`
   - sin DNI → skipped/error `"sin DNI"`
   - sin validadas → skipped
   - else → `uploadAdmission`

5. JSON: `{ uploaded: [protocol_numbers], skipped: ["C-… (motivo)"], errors: ["C-… (motivo)"] }`

6. Autorización: `lab-admissions.show` (igual que batch email)

**Verificación:** `php artisan route:list --name=lab.admissions.batch-space10`

## Notas / Consideraciones

- Space10 limita PDF a 8MB en validación (`max:8000` KB) — si informes muy grandes fallan, documentar; fuera de alcance comprimir.
- DNI Labit = `patients.patientId`; Space10 = `pacientes.idPaciente`.
- No modificar repo Space10 salvo necesidad de ampliar límite de tamaño (fuera de alcance).
- `sent_at` (email) y `space10_uploaded_at` (portal) son independientes: un protocolo puede estar en Space10 sin email, o email sin Space10 (reintento batch).
- Referencia script legacy: `space10/scripts/upload_lab.py` apunta a producción; Labit usa URL configurable para local/WAMP.
